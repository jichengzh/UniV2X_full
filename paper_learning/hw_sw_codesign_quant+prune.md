# 面向固定平台的剪枝+量化软硬件协同优化方案

> 项目: UniV2X
> 场景: 在固定GPU/Jetson平台上,通过结构化剪枝+PTQ量化压缩感知模型,同时优化精度、延迟和内存
> 核心挑战: 精度评估无法解析化,必须跑验证集;软件决策与TensorRT硬件性能存在非显然耦合

---

## 1. 搜索空间定义

### 1.1 软件搜索空间

**结构化剪枝:**

| 可调参数 | 范围 | 说明 |
|---------|------|------|
| 剪枝通道选择 | 各层的输出通道子集 | 按重要性指标(L1范数/Taylor展开等)选择保留通道 |
| 检测头中间表示维度 | 原始维度的30%-100% | 检测头的中间特征维度压缩 |
| 逐层剪枝率 | 0%-70% | 每层独立的通道剪枝比例 |

**PTQ量化:**

| 可调参数 | 范围 | 说明 |
|---------|------|------|
| 量化对象 | 权重/激活/两者 | 每层独立选择量化什么 |
| 量化粒度 | per-tensor / per-channel | 通道级精度更高但硬件开销更大 |
| 位宽 | INT8 / INT6 / INT4 | 每层独立的位宽分配 |
| 量化格式 | 均匀/非均匀 | 均匀量化硬件友好,非均匀精度更高 |
| 对称性 | 对称/非对称 | 对称量化计算更简单 |

### 1.2 硬件搜索空间

在TensorRT固定平台上,以下参数主要由TensorRT自动决策,但受软件配置间接影响:

| 参数 | 说明 | 与软件配置的耦合 |
|------|------|---------------|
| 算子融合策略 | TensorRT自动决定哪些算子融合 | 剪枝改变图结构 → 影响可融合的算子组合 |
| kernel选择 | TensorRT为每层选择最优kernel实现 | 通道数和位宽决定可用kernel集合 |
| 内存分配 | 激活tensor的内存复用策略 | 剪枝改变tensor形状 → 影响复用效率 |
| reformatting层 | 相邻层精度/格式不匹配时插入 | 混合精度配置直接决定reformatting数量 |

## 2. 核心问题: 软硬件耦合

### 2.1 为什么不能串行优化

串行方式(先优化剪枝量化 → 再部署TensorRT)会遗漏以下耦合关系:

| 软件决策 | 对硬件的隐含影响 |
|---------|---------------|
| 剪枝后通道数为奇数(如127) | TensorRT无法高效对齐(INT8要求通道数为8/16/32的倍数),延迟反增 |
| 某层量化到INT4 | TensorRT可能没有对应kernel → 回退FP32 → 比INT8更慢 |
| 剪枝率剧变的相邻层(80% vs 10%) | 形状不匹配 → 插入reshape/transpose → 打断融合链 |
| 检测头中间表示被大幅剪枝 | 改变计算图拓扑 → 可能破坏已有的算子融合模式 |
| 部分层量化+部分层不量化 | 精度切换处插入reformatting层 → 额外延迟开销 |

这些问题只有在**同时观察(软件配置→精度)和(软件配置→硬件延迟)**时才能发现。

### 2.2 与CASCO的对比

| 维度 | CASCO | 本方案 |
|------|-------|-------|
| 软件空间 | 层融合分组 + 算子映射 | 剪枝配置 + 量化配置 |
| 硬件空间 | PE数/存储/NoC(可设计) | TensorRT融合/调度(自动,不可直接控制) |
| 精度 | **不关心**(不改变模型权重) | **必须关心**(剪枝量化改变精度) |
| 评估瓶颈 | Timeloop(毫秒级,不是瓶颈) | 精度评估(分钟级,是瓶颈) |
| 协同本质 | HW配置 ↔ 融合/映射的EDP反馈 | 剪枝量化配置 ↔ TensorRT实际性能 |

**关键差异:** CASCO只优化EDP(能耗×延迟),有解析模型可以毫秒级算出。本方案必须同时优化精度+延迟+内存,其中精度没有解析公式,必须跑验证集——这决定了代理模型不是"加速手段"而是"实现协同的必选项"。

## 3. 协同评估方案: 三层联合漏斗

每一层**同时包含软件指标和硬件指标**的评估,而非先软件后硬件的串行流程。

```
搜索空间: (剪枝配置, 量化配置) 联合空间
         │
         ▼
┌──────────────────────────────────────────────────────┐
│  第1层: 联合快速筛选 (毫秒级, 过滤~85%候选)             │
│                                                        │
│  软件侧检查:                                            │
│    · 模型大小 = Σ(保留通道数 × 核大小 × 位宽 / 8)        │
│    · 参数量是否在可接受范围内                              │
│    · 剪枝率是否在合理区间(避免极端配置)                    │
│                                                        │
│  硬件侧检查:                                            │
│    · 通道对齐: 剪枝后通道数是否为8/16/32的倍数            │
│    · 量化可行性: 该层位宽是否有TensorRT对应kernel          │
│    · 融合预判: 剪枝后的图结构是否破坏已知融合模式           │
│    · reformatting预估: 混合精度边界数量                   │
│                                                        │
│  排除: 硬件不友好 OR 配置极端的候选                       │
└──────────────────────────────────────────────────────┘
         │  存活~15%候选
         ▼
┌──────────────────────────────────────────────────────┐
│  第2层: 联合代理模型 (毫秒级预测, 过滤~80%)              │
│                                                        │
│  一次性采样阶段 (离线, 一次性成本):                       │
│    从第1层存活候选中随机采样80-100个,每个做完整评估:       │
│    · 应用剪枝+量化 → 压缩模型                            │
│    · TensorRT build → engine                            │
│    · TensorRT profiling → 真实延迟+内存                  │
│    · 验证集子集(1/10数据)推理 → 近似精度                  │
│    → 得到80-100组 (config → mAP, latency, memory) 数据   │
│                                                        │
│  训练代理模型:                                           │
│    输入特征: [各层剪枝率, 各层通道数, 各层位宽,            │
│              量化格式, 对称性, ...]                       │
│    输出: (预测mAP, 预测latency, 预测memory)              │
│    模型选择: 高斯过程(候选<1000) 或 XGBoost(候选更多)     │
│                                                        │
│  代理模型能学到的耦合模式:                                │
│    · 通道数=127, INT8 → 延迟异常高 (对齐问题)            │
│    · 检测头剪枝80%+相邻层10% → 延迟反增 (reformatting)   │
│    · 某层INT4 → 延迟比INT8高 (kernel回退)                │
│                                                        │
│  搜索: 贝叶斯优化在代理模型上搜索Pareto前沿               │
│    每步同时看精度和延迟,采样方向融合了软硬件信息            │
└──────────────────────────────────────────────────────┘
         │  Pareto前沿上~20个候选
         ▼
┌──────────────────────────────────────────────────────┐
│  第3层: 完整验证 (分钟级)                                │
│                                                        │
│  · 完整验证集推理 → 真实mAP/NDS                          │
│  · TensorRT完整profiling → layer-by-layer延迟分析        │
│  · 确认无意外FP32回退层                                  │
│  · 确认无功能性问题                                      │
│                                                        │
│  → 最终输出: Pareto前沿上的最优配置                       │
└──────────────────────────────────────────────────────┘
```

## 4. 协同体现在哪

### 4.1 第1层的协同

不是只看"模型够不够小",而是**同时检查硬件友好性**:

```python
def joint_filter(pruning_config, quant_config):
    channels = apply_pruning(base_channels, pruning_config)
    
    # 软件侧
    model_size = compute_model_size(channels, quant_config)
    if model_size > max_model_size:
        return False  # 模型太大
    
    # 硬件侧: 通道对齐检查
    for ch in channels:
        if ch % 8 != 0:  # TensorRT INT8要求
            return False  # 通道不对齐,TensorRT效率低
    
    # 硬件侧: 量化kernel可用性检查
    for layer, bits in quant_config.layer_bits.items():
        if bits == 4 and layer.type == 'deformable_attention':
            return False  # 该算子无INT4 kernel
    
    # 硬件侧: reformatting边界数量
    reformat_count = count_precision_boundaries(quant_config)
    if reformat_count > threshold:
        return False  # 精度切换太多,reformatting开销大
    
    return True
```

一个剪枝方案即使压缩率很好,但通道不对齐就直接排除——这是硬件信息在最早阶段就参与筛选。

### 4.2 第2层的协同 (核心)

代理模型的输入是软件配置参数,输出**同时**包含精度和延迟。模型自动学到软硬件耦合:

```
代理模型训练数据示例:

config_1: [剪枝率=30%, 通道=[128,256,256,128], 位宽=[8,8,8,8], 对称=True]
  → mAP=0.62, latency=25ms, memory=450MB

config_2: [剪枝率=50%, 通道=[96,192,192,96], 位宽=[8,8,8,8], 对称=True]
  → mAP=0.58, latency=20ms, memory=320MB  ← 通道对齐,延迟正常下降

config_3: [剪枝率=50%, 通道=[97,191,193,95], 位宽=[8,8,8,8], 对称=True]
  → mAP=0.57, latency=28ms, memory=325MB  ← 通道不对齐,延迟反增!

config_4: [剪枝率=40%, 通道=[128,192,192,128], 位宽=[4,8,8,8], 对称=True]
  → mAP=0.54, latency=32ms, memory=280MB  ← INT4回退FP32,延迟更高!

→ 代理模型学到:
  · 通道数对8取模的余数 与 延迟 强相关
  · 第一层用INT4 → 延迟不降反增
  · 这些规律纯看精度或纯看延迟都发现不了
```

### 4.3 搜索过程的协同

贝叶斯优化在代理模型上搜索时,每一步的采样方向同时受精度和延迟影响:

```
搜索过程示例:

第1轮: 随机探索
  候选A: (剪枝30%, INT8全量化) → 预测(mAP=0.62, lat=25ms)
  候选B: (剪枝50%, INT8部分层) → 预测(mAP=0.58, lat=22ms)
  候选C: (剪枝50%, INT4部分层) → 预测(mAP=0.55, lat=30ms)

  发现: INT4在该平台上不划算(精度降了延迟还增了)
  → 这个判断融合了软件信息(精度)和硬件信息(延迟)

第2轮: 贝叶斯优化引导
  → 自动避开INT4区域
  → 在(剪枝30-50%, INT8, 通道对齐)区域密集采样
  → 搜索方向由软硬件联合指标驱动

第N轮: 收敛
  → 输出Pareto前沿: 精度-延迟的最优权衡曲线
  → 每个点都是软硬件联合最优,非串行拼接
```

## 5. 评估方法总结

| 层 | 评估内容 | 方法 | 成本 | 协同体现 |
|---|---------|------|------|---------|
| 第1层 | 可行性(模型大小+硬件友好性) | 解析公式 + 规则检查 | 毫秒 | 硬件对齐/kernel可用性在最早阶段参与筛选 |
| 第2层 | 精度+延迟+内存(联合预测) | 代理模型(GP/XGBoost) | 毫秒(预测) / 分钟(采样训练数据) | 一个模型同时预测软硬件指标,自动捕获耦合 |
| 第3层 | 全部指标(真实值) | 完整验证集 + TensorRT profiling | 分钟 | 最终确认,无新的协同机制 |

**协同的关键载体是第2层的联合代理模型**——它是唯一能在毫秒级同时给出精度和延迟预测的方法,也是搜索算法获取软硬件联合反馈的唯一途径。没有它,精度评估(分钟级)的成本使得搜索算法无法在合理时间内获得足够的反馈来学习软硬件耦合关系。

## 6. 与现有工作的定位

```
         纯软件优化                    协同优化                  纯硬件优化
         (本方案改进前)               (本方案目标)              (CASCO)
              │                          │                        │
  先剪枝量化→再部署TensorRT    剪枝量化配置↔TensorRT性能     HW配置↔融合+映射
  精度和延迟分别优化           联合代理模型统一评估           Timeloop解析模型
  可能精度好但延迟差           Pareto前沿上每点都协同最优    不涉及精度评估
```

本方案的定位: 在**固定硬件平台**上,通过**联合代理模型**实现剪枝/量化的软件决策与TensorRT硬件性能之间的协同优化。与CASCO的全栈DSE不同,本方案不设计新硬件,而是让软件压缩决策"感知"目标平台的硬件特性。
