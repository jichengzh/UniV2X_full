## 一、研究背景

### 1.1 UniV2X 系统

UniV2X 是一个**端到端车路协同 (V2X) 自动驾驶感知系统**：

- **单相机约束**：V2X-Seq-SPD 数据集每个 agent 仅 **1 个前视相机**。模型架构支持任意 `num_cam`，但 SPD 上 N=1。
- **双 agent 架构**：ego（车端，完整模型）+ infra（路侧，冻结子模型，产出 track queries）
- **V2X 融合**：query-level 融合（非 BEV-to-BEV），infra track queries 经坐标变换后注入 ego BEV grid
- **时序机制**：`prev_bev` 跨帧传递，经姿态旋转对齐后送入 TemporalSelfAttention

**完整推理流水线**：

```
img [B, 1, 3, H, W]
  → GridMask → ResNet-101+DCN backbone (34ms)
  → FPN neck
  → BEVFormerEncoder 6 layers: TSA(prev_bev) + SCA(image feats) (65ms)
  → bev_embed [200x200, B, 256]
  → [V2X: 注入 infra queries]
  → DetectionTransformerDecoder 6 layers (10ms)
  → 多任务头: tracking / detection / segmentation (89ms)
```

**关键计算瓶颈**（1.2 C.1 实测）：BEV encoder 的 MSDA 采样（memory-bound，128 万次 bilinear sampling），而非 FFN 矩阵乘（compute-bound）。因此 FFN 结构剪枝在 TRT 端零加速。

### 1.2 已建立的搜索空间

| 子空间 | 维度 | 已验证基数 | 最优点 | 来源 |
|:---:|---|:---:|---|---|
| B1 剪枝 | P1a (enc FFN), P1b (dec FFN), P9 (dec 层数), P11 (seg MLP) | 80 | D.1.4 enc=1.0 dec=0.7 AMOTA **0.367** | 1.2 |
| B2 量化 | Q1-Q6 (backbone/BEV/heads 精度/粒度/目标) | 32~96 | BEV INT8 AMOTA **0.364** | 1.1 |
| B1 x B2 联合 | 7 个 Pareto 采样点 | ~2,000 可行 | D.2.7 解耦+INT8 AMOTA **0.360** | 1.2 |

### 1.3 来自 1.1/1.2 的关键经验教训

| 教训 | 来源 | 对 D 空间的影响 |
|---|---|---|
| FFN 剪枝在 TRT 端零加速（瓶颈是 MSDA 采样） | 1.2 C.1 | 剪枝改变的是 compute-bound/memory-bound 特性，进而影响并行策略 |
| 真实加速来自量化 (1.9x) 和 decoder-only 剪枝 (ego_fwd -42%) | 1.2 C.1/D.1 | 各阶段延迟比变化影响流水线重叠策略 |
| 剪枝率与最优 queue_length 强耦合（30%用q=1，60%必须q=2） | 1.2 B.1 | 时序缓存帧数不是独立决策，受压缩率约束 |
| 解耦剪枝后 INT8 近乎无损(-0.007)，绑定剪枝后 INT8 严重下降(-0.048) | 1.2 D.2 | BEV 特征稳定性决定缓存精度可行性 |
| 量化误差层间相互抵消（交互项为负） | 1.1 A4 | 联合搜索中线性加性假设安全 |
| 模块级统一量化 >> 逐层配置 | 1.1 A4 | 搜索粒度应以模块为单位 |

## 二、评估指标与搜索空间判定标准

### 2.1 四个 trade-off 指标

| 指标 | 定义 | 测量方式 |
|:---:|---|---|
| **精度** | AMOTA / mAP / seg IoU | 验证集评估 |
| **时延** | 单帧端到端推理时间 (ms) | CUDA Event / wall-clock |
| **能耗** | 单帧推理 GPU 能量消耗 (mJ) | `nvidia-smi` 功率 x 时延 |
| **存储** | 模型体积 + 运行时峰值显存 (MB) | `.pth` 大小 + `torch.cuda.max_memory_allocated` |

### 2.2 判定规则

- **放入搜索空间**：该维度的不同取值在四个指标上存在**不可消解的 trade-off**——没有一个取值在所有指标上同时最优
- **锁定在搜索空间外**：有唯一最优解 / 是工程 bug / 是数据集或硬件的固有约束

## 三、D 空间搜索维度定义

> 硬件部署配置回答的问题：**给定一个固定的模型（结构、权重、精度都已确定），如何在目标硬件上最高效地执行它？**
> 聚焦 PyTorch 端实现，暂不涉及 TRT 集成和 V2X 通信调度。

### 3.1 D1: 多 Agent 并行度

**定义**：ego 和 infra agent 的推理使用多少个 CUDA stream 并行执行。

**搜索变量**：`num_parallel_streams`

**取值范围**：{1, 2, 3, 4}

| 取值 | 含义 |
|:---:|---|
| 1 | 全串行：ego backbone → ego BEV → infra backbone → infra BEV → fusion → decode |
| 2 | ego 和 infra 的 backbone+BEV 并行（2 streams） |
| 3 | ego backbone、ego BEV、infra backbone+BEV 三路并行 |
| 4 | ego backbone、ego BEV、infra backbone、infra BEV 四路全拆 |

**四指标 trade-off**：

| 指标 | stream=1 | stream=2 | stream=4 |
|:---:|---|---|---|
| 精度 | 基准 | 不变（数值等价） | 不变 |
| 时延 | 最高（全串行） | 降低（并行隐藏 infra 延迟） | 可能进一步降低或**反升**（SM 竞争） |
| 能耗 | 最低（GPU 利用率低） | 更高（并行提升利用率） | 最高（满载） |
| 存储 | 最低 | 更高（两个 context 同驻） | 最高 |

**为什么是搜索维度**：stream 数量增加到一定程度后，GPU SM 资源被多个 stream 分摊，单个 stream 的 kernel 吞吐下降，总时延可能不降反升。最优点取决于模型大小（受 B1 影响）和单 kernel 的资源需求。

#### D1 与 B1/B2 的双向耦合

**B1/B2 → D1（软件影响硬件选择）**：
- 剪枝后模型更小 → 单 agent 占用更少 SM/显存 → **更高并行度变得可行**
- INT8 量化后每层计算量减半 → 单 stream kernel 更短 → 并行的"可隐藏窗口"更短 → **并行收益减小**
- 激进剪枝 + INT8 可能让模型完全 memory-bound → 多 stream 分享带宽 → **并行反而有害**

**D1 → B1/B2（硬件反馈影响软件选择）**：
- 选择高并行度 → 每 stream 分得的 GPU 资源更少 → **需要更激进的剪枝**来适配有限资源
- 选择高并行度 → 峰值显存增加 → **量化变得更有价值**（INT8 减半显存，为并行腾出空间）
- 选择高并行度 → 总 GPU 功耗更高 → 在边缘设备功耗墙下 **可能被迫回退到更激进的压缩**

#### D1 实现方案

**修改文件**：推理主循环（`tools/test_with_pruning.py` 或等价入口）

```python
import torch.cuda as cuda

def infer_multi_stream(ego_model, infra_model, data, num_streams=2):
    if num_streams == 1:
        # 串行
        ego_bev = ego_model.get_bev(data['ego'])
        infra_bev = infra_model.get_bev(data['infra'])
    elif num_streams >= 2:
        stream_ego = cuda.Stream()
        stream_infra = cuda.Stream()
        with cuda.stream(stream_ego):
            ego_bev = ego_model.get_bev(data['ego'])
        with cuda.stream(stream_infra):
            infra_bev = infra_model.get_bev(data['infra'])
        cuda.current_stream().wait_stream(stream_ego)
        cuda.current_stream().wait_stream(stream_infra)
    # fusion + decode on default stream
    result = ego_model.decode(ego_bev, infra_bev)
    return result
```

### 3.2 D2: 流水线阶段重叠

**定义**：推理流水线的不同阶段之间是否在时间上重叠执行（帧间流水线）。

**搜索变量**：`pipeline_overlap`

**取值范围**：{无重叠, backbone-BEV 重叠, 多阶段全重叠}

**流水线结构**（ego 单 agent，各阶段延迟来自 1.2 实测）：

```
无重叠（当前）:

  Frame t:   [Backbone 34ms] → [BEV_enc 65ms] → [Decoder 10ms] → [Seg 89ms]
  Frame t+1:                                                                   [Backbone] → ...

  单帧延迟 = 34 + 65 + 10 + 89 = 198ms


backbone-BEV 重叠:

  Frame t:   [Backbone 34ms] → [BEV_enc 65ms] → [Decoder 10ms] → [Seg 89ms]
  Frame t+1:                   [Backbone 34ms] → [BEV_enc 65ms] → ...
                                ↑
                          BEV_enc(t) 和 Backbone(t+1) 在不同 stream 上并行
                          前提: BEV_enc 不依赖 Backbone(t+1) 的输出

  稳态帧间延迟 = max(backbone, bev_enc) + decoder + seg = 65 + 10 + 89 = 164ms
  (首帧仍需 198ms)


多阶段全重叠:

  Frame t:   [Back 34] → [BEV 65  ] → [Dec 10] → [Seg 89             ]
  Frame t+1:              [Back 34 ] → [BEV 65 ] → [Dec 10] → [Seg 89 ]
  Frame t+2:                           [Back 34] → [BEV 65 ] → ...

  稳态帧间延迟 = max(各阶段) = seg(89ms)
  需要同时容纳多帧的中间激活
```

**四指标 trade-off**：

| 指标 | 无重叠 | backbone-BEV 重叠 | 全重叠 |
|:---:|---|---|---|
| 精度 | 基准 | 不变（各帧独立计算） | 不变 |
| 时延 | 198ms | 164ms（稳态） | 89ms（理论下限） |
| 能耗 | 最低 | 更高（两阶段并行） | 最高（多阶段并行） |
| 存储 | 最低 | +backbone 中间激活 (~50MB) | +多帧中间激活 (~200MB) |

**为什么是搜索维度**：重叠程度越高时延越低，但显存和能耗同步增加。在边缘设备显存有限时，可能只能做 backbone-BEV 重叠而无法全重叠。最优重叠程度取决于各阶段的相对延迟（受 B1 剪枝影响）。

#### D2 与 B1/B2 的双向耦合

**B1/B2 → D2（软件影响硬件选择）**：
- **剪枝改变流水线平衡**：FFN 剪枝让 BEV encoder 65ms→55ms，backbone 34ms 不变 → 剪枝后 backbone-BEV 重叠收益从 34ms 降为 ~21ms → **剪枝越多，backbone-BEV 重叠收益越小**
- **decoder 剪枝改变瓶颈**：P9 decoder 5 层 10.9→8.7ms，seg_head 89ms 不变 → seg_head 始终是全重叠的瓶颈 → **decoder 剪枝不影响全重叠的稳态时延**
- **量化改变各阶段时延比**：INT8 让 BEV encoder 从 65ms 降到接近 backbone (34ms) → backbone-BEV 重叠几乎无收益 → **量化后应关注更深层的重叠**

**D2 → B1/B2（硬件反馈影响软件选择）**：
- 选择 backbone-BEV 重叠 → 稳态时延由 max(backbone, BEV) 决定 → **应剪枝使 BEV 时延 ≤ backbone 时延**，而非盲目追求最大剪枝率 → B1 获得**时延目标约束**
- 选择全重叠 → 稳态时延由最慢阶段 (seg_head 89ms) 决定 → 优化其他阶段的边际收益为零 → **B1 应聚焦 seg_head 剪枝而非 BEV encoder 剪枝**
- 选择重叠 → 需要更多显存存放多帧激活 → **B2 量化（减小激活尺寸）变得更有价值**

#### D2 实现方案

**修改文件**：推理主循环

```python
class PipelinedInference:
    """帧间流水线推理"""
    def __init__(self, model, overlap='none'):
        self.model = model
        self.overlap = overlap
        self.backbone_stream = cuda.Stream() if overlap != 'none' else None
        self.pending_backbone_feat = None  # 缓存下一帧的 backbone 结果
    
    def infer_frame(self, img, prev_bev):
        if self.overlap == 'none':
            feat = self.model.backbone(img)
            bev = self.model.bev_encoder(feat, prev_bev)
            det = self.model.decoder(bev)
            seg = self.model.seg_head(bev)
            return bev, det, seg
        
        elif self.overlap == 'backbone_bev':
            # 使用上一帧预取的 backbone 结果（如果有）
            if self.pending_backbone_feat is not None:
                feat = self.pending_backbone_feat
            else:
                feat = self.model.backbone(img)
            
            # BEV + decoder + seg 在默认 stream
            bev = self.model.bev_encoder(feat, prev_bev)
            
            # 同时在另一个 stream 预取下一帧 backbone
            # （下一帧图像需要提前准备好）
            if self.next_img is not None:
                with cuda.stream(self.backbone_stream):
                    self.pending_backbone_feat = self.model.backbone(self.next_img)
            
            det = self.model.decoder(bev)
            seg = self.model.seg_head(bev)
            
            # 确保 backbone 预取完成
            cuda.current_stream().wait_stream(self.backbone_stream)
            return bev, det, seg
```

### 3.3 D3: 时序缓存管理

**定义**：跨帧传递的 `prev_bev` 的存储精度和帧数。

**搜索变量**：`cache_precision` x `cache_frames`

**取值范围**：

| 子维度 | 取值 | 说明 |
|---|---|---|
| 缓存精度 | {FP16, INT8} | prev_bev 的存储方式 |
| 缓存帧数 | {0, 1, 2} | 0=无时序，1=前一帧，2=前两帧 |

**组合**：{(FP16,0), (FP16,1), (FP16,2), (INT8,1), (INT8,2)} = **5 种**

> **与 B2 量化的区别**：B2 控制**模型层的计算精度**（forward 中权重和激活的位宽），D3 控制**帧间缓存的存储精度**（BEV encoder 输出后、下一帧使用前的存储方式）。即使模型在 FP16 下计算，输出 BEV 也可以被 INT8 压缩存储再反量化使用。两者是独立决策但存在耦合。

**prev_bev 数据量**：

```
200 x 200 x 256 = 10,240,000 elements
FP16: 20.48 MB / frame      INT8: 10.24 MB / frame

2 agents x 2 frames:   FP16 = 81.92 MB,  INT8 = 40.96 MB  → 节省 40.96 MB
```

**四指标 trade-off**：

| 指标 | (FP16,1) | (INT8,1) | (FP16,2) | (INT8,2) |
|:---:|---|---|---|---|
| 精度 | 基准 | 量化误差累积 | 更好（更多时序） | 时序补偿 vs 量化损失 |
| 时延 | 基准 | 微降（带宽省） | 微升（多一帧 attention） | 折中 |
| 能耗 | 基准 | 略低 | 略高 | 折中 |
| 存储 | 20.5 MB | 10.2 MB | 41.0 MB | 20.5 MB |

#### D3 与 B1/B2 的双向耦合

**B1/B2 → D3（软件影响硬件选择）**：
- **1.2 Phase B 核心发现**：FFN 剪枝率与最优时序帧数强耦合
  - 30% 剪枝 → q=1 最优（单帧信息已够）
  - 60% 剪枝 → q=2 必须（单帧不足，需时序补偿）
  - D3.cache_frames 的最优值**直接取决于** B1 压缩率
- **1.2 Phase D.2 发现**：解耦剪枝 (encoder 不动) 后 INT8 近乎无损 (-0.007)，绑定剪枝后 INT8 严重下降 (-0.048)
  - B1 保守剪枝（encoder 不动）→ BEV 分布稳定 → D3.INT8 缓存大概率可行
  - B1 激进剪枝（encoder 也剪）→ BEV 分布被扰动 → D3.INT8 可能雪上加霜
- **B2 → D3**：B2=INT8 时模型输出 BEV 的有效精度本身就低于 FP16 → D3 用 INT8 缓存几乎不额外损失 → **B2=INT8 时 D3.INT8 的代价更低**

**D3 → B1/B2（硬件反馈影响软件选择）**：
- D3=(FP16,0) 无时序 → 模型完全依赖单帧 → **不能做激进剪枝**（1.2 已证明 60% 剪枝需要时序补偿）→ B1 剪枝上限被 D3 约束
- D3=(INT8,2) 时序丰富但有量化噪声 → 模型需要对时序量化噪声鲁棒 → **影响 B1 的微调策略**（数据增强应包含时序特征扰动）
- D3 帧数多 → 显存紧张 → **B1/B2 需要更激进的压缩**来腾出显存给时序缓存

#### D3 实现方案

**修改文件**：`projects/mmdet3d_plugin/univ2x/dense_heads/univ2x_head.py`

```python
class TemporalCacheManager:
    """时序 BEV 缓存管理"""
    def __init__(self, precision='fp16', max_frames=1):
        self.precision = precision
        self.max_frames = max_frames
        self.cache = []       # list of cached BEV tensors
        self.scale = None     # INT8 quantization scale (per-tensor)
    
    def store(self, bev_fp16):
        """BEV encoder 输出后调用"""
        if self.precision == 'int8':
            if self.scale is None:
                self.scale = bev_fp16.abs().amax() / 127.0
            bev_cached = torch.clamp(
                torch.round(bev_fp16 / self.scale), -127, 127
            ).to(torch.int8)
        else:
            bev_cached = bev_fp16
        
        self.cache.append(bev_cached)
        if len(self.cache) > self.max_frames:
            self.cache.pop(0)
    
    def retrieve(self):
        """下一帧 TemporalSelfAttention 使用前调用"""
        if not self.cache:
            return None
        if self.precision == 'int8':
            return [c.float() * self.scale for c in self.cache]
        return self.cache
```

### 3.4 D4: 显存分配策略

**定义**：运行时 GPU 显存的分配和管理方式。

**搜索变量**：`memory_strategy`

**取值范围**：{动态分配, 静态预分配, 碎片整理}

| 取值 | 含义 |
|---|---|
| 动态分配 | PyTorch 默认 CUDA allocator，按需 malloc/free |
| 静态预分配 | 启动时一次性分配所有需要的显存，运行时零分配 |
| 碎片整理 | `torch.cuda.memory.CUDAPluggableAllocator` 或 `PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True` |

**四指标 trade-off**：

| 指标 | 动态 | 静态 | 碎片整理 |
|:---:|---|---|---|
| 精度 | 不影响 | 不影响 | 不影响 |
| 时延 | 分配延迟抖动 (0.1-1ms) | 零分配延迟 | 低分配延迟 |
| 能耗 | 基准 | 略低（无分配开销） | 基准 |
| 存储 | 峰值波动大 | **固定预留，可能浪费 20-30%** | 介于两者之间 |

**为什么是搜索维度**：在边缘设备（Jetson Orin, 16GB）上，动态分配的碎片化可能导致 OOM，而静态预分配浪费显存。存储效率 vs 时延稳定性存在 trade-off。

#### D4 与 B1/B2 的双向耦合

**B1/B2 → D4（软件影响硬件选择）**：
- 剪枝/量化后模型更小 → 显存需求更可预测 → **静态预分配更可行**（浪费比例更小）
- 未压缩大模型 → 显存波动大 → 静态预分配需预留过多 → **动态分配更合适**

**D4 → B1/B2（硬件反馈影响软件选择）**：
- 选择静态预分配 → 显存预算**硬上限**确定 → 反过来限定 B1/B2 的**最低压缩要求**（模型 + 激活 + 缓存必须 fit 在预算内）
- 选择动态分配 → 显存约束是"软"的 → B1/B2 有更大的压缩灵活度

#### D4 实现方案

```python
def setup_memory_strategy(strategy, budget_mb=None):
    if strategy == 'dynamic':
        pass  # PyTorch 默认
    elif strategy == 'static':
        # 预分配固定大小的显存池
        torch.cuda.set_per_process_memory_fraction(
            budget_mb / torch.cuda.get_device_properties(0).total_memory * 1024**2
        )
    elif strategy == 'defrag':
        import os
        os.environ['PYTORCH_CUDA_ALLOC_CONF'] = 'expandable_segments:True'
```

**预计工作量**：~0.5 天

## 六、耦合关系全景图

```
B1 (剪枝)                    D (硬件部署)                    B2 (量化)
─────────                    ───────────                    ─────────

剪枝率 ─────────────────────► D1 并行度 ◄──────────────────── INT8 减半显存
  模型更小→更高并行度可行        │                              更多 stream 可行
                                │
  ◄─────────────────────────────┘
  高并行度→每 stream 资源少→需要更激进剪枝

剪枝改变阶段延迟比 ──────────► D2 流水线重叠 ◄─────────────── 量化改变阶段时延
  FFN剪枝→BEV变快→重叠收益变小    │                           INT8→BEV接近backbone
                                  │                           →重叠收益变小
  ◄───────────────────────────────┘
  重叠选择→决定"应该剪哪个阶段"(平衡流水线 vs 最小化瓶颈)

剪枝率→最优缓存帧数 ─────────► D3 时序缓存 ◄──────────────── B2精度→缓存精度可行性
  60%剪枝必须q=2                  │                          INT8计算→INT8缓存自然
  encoder不动→INT8缓存可行        │
                                  │
  ◄───────────────────────────────┘
  D3=(FP16,0)无时序→限制最大剪枝率
  D3帧数多→显存紧→需要更激进压缩

剪枝/量化→显存可预测 ────────► D4 显存策略 ◄──────────────── 量化→显存需求降低
  小模型→静态预分配可行            │
                                  │
  ◄───────────────────────────────┘
  静态预分配→硬显存上限→限定B1/B2最低压缩要求
```

**核心洞察**：剪枝/量化改变了模型的计算特性（compute-bound vs memory-bound、单帧充足 vs 需时序补偿、各阶段延迟比），从而改变了硬件策略的最优选择；反过来，硬件约束（显存上限、并行资源、功耗墙）限定了软件压缩的必要程度和方向。**这就是联合搜索的价值——单独优化任一子空间无法找到全局最优。**

### 7.2 联合约束

| 约束 | 条件 | 限制 |
|---|---|---|
| C1 | B1.ffn_ratio < 0.5 | D3.cache_frames 必须 >= 2 |

### 7.3 搜索方法

采用 **BoTorch** (Meta) 两级评估漏斗 + 贝叶斯优化：

```
外循环: 遍历 B1(剪枝 Pareto 点) + D(拓扑配置: D1, D2, D4)
  │
  ├── 约束预检查 (C1-C5)
  ├── 复用已有剪枝 ckpt（1.2 已有多个）
  │
  └── 内循环: BO 搜索 B2(量化) + D(非拓扑: D3)
       │
       ├── Level 1 廉价评估 (<1s/候选):
       │     精度预估 = sensitivity_map 线性加性
       │     延迟预估 = latency_LUT 查表
       │     存储预估 = 解析计算
       │     过滤 70-80%
       │
       └── Level 2 真实评估 (10-30min/候选):
             量化校准 → 推理 → AMOTA (1/10 验证集)
             延迟/显存/功率实测
             → 反馈至 GP 代理模型
```

## 十一、风险与回滚

| 风险 | 触发条件 | 回滚方案 |
|---|---|---|
| 多流并行无加速 | GPU SM 竞争抵消并行收益 | 锁定 D1=1 |
| 全重叠显存不足 | 边缘设备 OOM | 退化到 backbone-BEV 重叠 |
| INT8 缓存精度崩溃 | AMOTA 下降 > 0.02 | 锁定 D3.precision=FP16 |
| BoTorch 搜索不收敛 | GP 拟合差 | 退化为网格搜索 Top-K |
| 静态预分配浪费过多 | > 30% 显存浪费 | 改用碎片整理方案 |
