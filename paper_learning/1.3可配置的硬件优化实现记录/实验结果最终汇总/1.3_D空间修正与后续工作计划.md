## 一、核心认知升级

### 1.1 硬件部署优化的两类本质

通过 EMOS 论文分析和 1.3 实验，我们认识到硬件部署优化分为两类本质不同的工作：

| 类型 | 特征 | 方法 | 占比 |
|---|---|---|:---:|
| **工程实现** | 无 trade-off，做了一定更好 | 一次性实现，成为搜索的"地基" | **~80%** |
| **搜索维度** | 有 trade-off，最优解取决于 B1/B2 | 与 B1/B2 联合搜索 | **~20%** |

之前的错误是把两类混在一起，导致搜索空间虚大（216 种）。修正后搜索空间仅 **6 种**，其余作为工程优化一次性实现。

### 1.2 与 B1/B2 的对比

| 子空间 | 本质 | 搜索维度数 | 工程实现量 |
|---|---|:---:|:---:|
| B1 剪枝 | 精度换效率（有大量 trade-off） | 4 维, 80 种 | 少（剪枝框架已有） |
| B2 量化 | 精度换效率（有大量 trade-off） | 6 维, 32 种 | 中（Q/DQ 注入等） |
| **D 部署** | **纯效率提升（大部分无 trade-off）** | **2 维, 6 种** | **大（EMOS 四大优化）** |

## 二、修正后的 D 空间

### 2.1 搜索维度（有 trade-off，需联合搜索）

| 维度 | 变量 | 取值 | Trade-off | 与 B1/B2 的耦合 |
|:---:|---|---|---|---|
| **D2** | 流水线重叠深度 | {无重叠, backbone-BEV 重叠, 全重叠} | 延迟↓ vs 存储↑ | B1 改变阶段延迟比 → 改变最优重叠深度；B2 压缩后给重叠腾显存 |
| **D3b** | 时序缓存帧数 | {1, 2} | 精度↑ vs 存储↑ | B1 高压缩需 2 帧时序补偿；B1 低压缩 1 帧即够 |

**搜索基数**：3 × 2 = **6 种**

**联合搜索空间**：B1(80) × B2(32) × D_search(6) = **15,360 种**

### 2.2 锁定维度（已通过实验确定最优，无 trade-off）

| 维度 | 锁定值 | 实验依据 |
|---|---|---|
| D3a 时序缓存精度 | **INT8** | AMOTA 0.339 ≥ FP16 的 0.333，显存 -50%，免费午餐 |
| D4 显存分配策略 | **defrag** | 降低 44% 延迟抖动，无额外成本 |
| D1 多 agent 并行 | **移除** | 真实 V2X 部署中 ego/infra 在不同设备上，天然并行 |

### 2.3 搜索空间外围的工程优化（参考 EMOS，一次性实现）

| 编号 | 优化手段 | 对应 EMOS | 效果 | 无 trade-off 的原因 |
|:---:|---|---|---|---|
| **E1** | 共享内存优化 | SI-1 | 减少全局显存访问延迟 | 配置推理引擎参数即可，不影响精度 |
| **E2** | I/O-计算重叠 | SI-2 | 隐藏数据加载延迟 | double-buffering 额外显存极小 |
| **E3** | 分阶段内存释放 | SI-3 | 峰值显存从"所有阶段之和"降为"最大单阶段" | 阶段边界由模型结构决定，overhead 微乎其微 |
| **E4** | 计算图优化 | SI-4 | 减少冗余算子和中间显存 | TRT/ONNX Runtime 自动或半自动完成 |

## 三、后续工作计划

### Phase A：搜索空间外围的工程优化（E1-E4，参考 EMOS）

> 这些优化构成搜索的"地基"——先全部实现，再在此基础上搜索 D2 × D3b 的最优配置。

#### A.1 E3 分阶段内存释放（优先级最高，预计 2 天）

**目标**：在 UniV2X 推理流水线的阶段边界主动释放已完成阶段的 GPU 资源。

**UniV2X 的阶段边界定义**（参考 EMOS Fig. 6 的四边界设计）：

```
Boundary 1: Backbone + FPN 完成
  → 释放: img 原始输入 tensor, GridMask 中间结果
  → 保留: 多尺度特征图 (传给 BEV encoder)

Boundary 2: BEV Encoder 完成
  → 释放: 多尺度特征图, SCA/TSA 中间激活
  → 保留: bev_embed (传给 decoder 和 seg_head)

Boundary 3: Detection Decoder 完成
  → 释放: decoder 中间激活, attention 中间结果
  → 保留: track queries + bbox predictions

Boundary 4: Seg Head 完成
  → 释放: seg_head 中间激活
  → 保留: 分割输出
```

**实现方式**：

```python
class StagedMemoryManager:
    """分阶段 GPU 内存管理"""

    def release_stage(self, stage_name):
        """在阶段边界调用，释放该阶段的中间张量"""
        # 1. 将该阶段的中间张量引用置 None
        # 2. 显式调用 torch.cuda.empty_cache() 回收碎片
        # 3. 记录释放的显存量用于 profiling
        torch.cuda.empty_cache()

    def wrap_forward(self, model):
        """在 ego agent 的 forward 各阶段间插入释放点"""
        original_forward = model.forward
        def staged_forward(*args, **kwargs):
            # Stage 1: Backbone
            img_feats = model.extract_img_feat(...)
            self.release_stage('backbone_input')  # 释放原始图像

            # Stage 2: BEV Encoder
            bev_embed = model.pts_bbox_head.get_bev_features(img_feats, ...)
            self.release_stage('img_feats')  # 释放多尺度特征

            # Stage 3: Decoder
            det_results = model.pts_bbox_head.get_detections(bev_embed, ...)
            self.release_stage('decoder_intermediate')

            # Stage 4: Seg Head
            seg_results = model.seg_head(bev_embed, ...)
            self.release_stage('seg_intermediate')

            return det_results, seg_results
        model.forward = staged_forward
```

**预期效果**：
- 当前峰值显存 ~2476MB（所有阶段中间激活同时存在）
- 预期降至 ~1800-2000MB（仅最大单阶段 + 保留的跨阶段数据）
- 在 Jetson Orin 8GB 上可能是"能跑 vs 不能跑"的关键差异

**验证**：
- 对比实施前后的 `torch.cuda.max_memory_allocated()`
- 确认 AMOTA 不变（阶段间传递的数据不被释放）

#### A.2 E2 I/O-计算重叠（预计 1.5 天）

**目标**：当前帧的模型计算与下一帧的数据加载/预处理并行。

**当前流程**（串行）：

```
Frame t:   [数据加载 CPU] → [预处理 CPU] → [模型推理 GPU] → [后处理 CPU]
Frame t+1:                                                                [数据加载] → ...
```

**优化后（double-buffering）**：

```
Frame t:   [数据加载] → [预处理] → [模型推理 GPU        ]
Frame t+1:                         [数据加载] → [预处理] → [模型推理 GPU]
                                    ↑ 在 GPU 推理时 CPU 做下一帧的加载和预处理
```

**实现方式**：

```python
class AsyncDataPrefetcher:
    """异步数据预取器"""

    def __init__(self, data_loader):
        self.loader_iter = iter(data_loader)
        self.stream = torch.cuda.Stream()
        self.next_data = None
        self._prefetch()  # 预取第一帧

    def _prefetch(self):
        try:
            self.next_data = next(self.loader_iter)
        except StopIteration:
            self.next_data = None
            return
        # 在独立 stream 上异步搬运到 GPU
        with torch.cuda.stream(self.stream):
            for k, v in self.next_data.items():
                if isinstance(v, torch.Tensor):
                    self.next_data[k] = v.cuda(non_blocking=True)

    def get_next(self):
        torch.cuda.current_stream().wait_stream(self.stream)
        data = self.next_data
        self._prefetch()  # 立即开始预取下一帧
        return data
```

**预期效果**：
- 当前数据加载+预处理占端到端时间的 ~50ms（from D2 分析：460ms 非模型开销的一部分）
- 重叠后可隐藏大部分 CPU 数据处理时间
- 显存额外开销：~50MB（一帧的预处理结果 buffer）

#### A.3 E4 计算图优化（预计 1 天）

**目标**：对 UniV2X 的 ONNX 图进行 EMOS 风格的优化。

**具体内容**（参考 EMOS SI-4）：

**1) 图重写**：
- 常量折叠：将 `prev_bev` 的姿态旋转矩阵等常量运算在导出时计算
- 冗余节点消除：移除 identity op、无效 reshape 等
- 语义保持融合：Conv + BatchNorm + ReLU → 单算子

**2) 复合节点融合**：
- BEV encoder 中的 LayerNorm 融合：ReduceMean + Sub + Pow + Sqrt + Div → 单个 LayerNorm kernel
- FFN 中的 GELU 融合：Mul + Erf + Add → 单个 GELU kernel
- Attention 中的 GEMM + 激活融合

**实现方式**：
- TRT 部署：大部分由 TRT builder 自动完成（`BuilderConfig` 开启 `FP16` + `INT8` flag）
- ONNX Runtime 部署：使用 `onnxruntime.transformers.optimizer` 或 `onnx-simplifier`
- 手动优化：对 TRT/ORT 无法自动融合的 custom plugin（MSDAPlugin）做手工算子融合

```bash
# ONNX 图简化
python -m onnxsim input.onnx output_sim.onnx

# ONNX Runtime 图优化
python -m onnxruntime.transformers.optimizer \
    --input input.onnx --output output_opt.onnx \
    --model_type bert --opt_level 2
```

#### A.4 E1 共享内存优化（预计 0.5 天）

**目标**：配置推理引擎最大化 GPU 共享内存的利用率。

**实现方式**：

```python
# TRT 端：设置最大共享内存
config = builder.create_builder_config()
config.set_memory_pool_limit(trt.MemoryPoolType.WORKSPACE, 2 << 30)  # 2GB workspace

# PyTorch 端：配置 CUDA allocator
# 增大 L2 cache 的 persistent 窗口（Ampere+）
torch.cuda.set_stream(torch.cuda.Stream())  # 独立 stream 减少同步

# ONNX Runtime 端：session options
import onnxruntime as ort
opts = ort.SessionOptions()
opts.enable_mem_pattern = True          # 内存模式优化
opts.enable_mem_reuse = True            # 内存复用
opts.graph_optimization_level = ort.GraphOptimizationLevel.ORT_ENABLE_ALL
```

**预期效果**：减少全局显存往返次数，延迟降低 5-15%（参考 EMOS 的 L1/L2 cache hit rate 提升）。

**每个配置需评估**：AMOTA / 延迟 / 能耗 / 存储（四指标）

**每个配置需与 B1 Pareto 点交叉**：
- baseline (无剪枝)
- D.1.4 (enc=1.0 dec=0.7)

## 五、本次修正的意义

| 版本 | D 空间搜索维度 | 搜索基数 | 联合空间 |
|---|:---:|:---:|:---:|
| v1 初版 | 9 维 | 128 | 327,680 |
| v2 第一次修正 | 4 维 (D1-D4) | 216 | 552,960 |
| v3 移除 D1 | 3 维 (D2-D4) | 18 | 46,080 |
| **v4 当前** | **2 维 (D2, D3b)** | **6** | **15,360** |

每次修正都在缩小搜索空间、扩大工程底座。这符合一个规律：**对硬件部署优化理解越深，搜索维度越少，工程实现越多。** 因为真正有 trade-off 的配置很少——大部分硬件优化是"做了就更好"的确定性工程。
