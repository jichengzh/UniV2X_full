# 1.3 可配置硬件优化 — 实验反思

> 日期：2026-04-21
> 性质：对 D 空间设计和实验过程中暴露的认知问题的深度反思

---

## 一、核心反思：搜索空间设计脱离了真实部署场景

### 1.1 D1 多流并行——一个不应该存在的搜索维度

**错误**：将"ego agent 和 infra agent 在同一 GPU 上多 CUDA stream 并行"作为搜索维度。

**根本问题**：UniV2X 是一个**车路协同 (V2X)** 系统。在真实部署中：

- **Ego agent** 跑在**车端**边缘设备（如 Jetson Orin）
- **Infra agent** 跑在**路侧**边缘设备（如路侧 RSU）
- 两者**物理上分布在不同设备上**，天然就是并行的
- 通信内容是 infra 的 track queries，经 V2X 通信链路（DSRC/C-V2X）传输
- Ego 收到 queries 后做 fusion + decoder，这发生在 ego 设备上

所以"ego 和 infra 在同一 GPU 上多 stream 并行"这个场景**在真实部署中根本不存在**。我花了时间实现和测试一个不会在实际中使用的配置。

### 1.2 错误的根源

回顾问题的来源：

1. **对 multi_agent.py 代码的表面理解**：看到 `forward_test` 里 infra 和 ego 是串行调用的，就想当然地认为"并行化"是一个优化方向。但没有追问**为什么它们是串行的**——答案是这只是单机仿真的实现方式，不是部署架构。

2. **混淆了仿真环境和部署环境**：当前所有实验都在**单机多 GPU 仿真环境**中运行（一块 RTX 4090 跑完整的 MultiAgent 模型）。我把仿真环境中的"串行瓶颈"当成了部署环境中的真实瓶颈。

3. **搜索空间设计时缺乏"部署场景审查"步骤**：设计搜索维度时只问了"这个参数能不能调？调了有没有 trade-off？"，没有先问**"这个参数在真实部署中是否存在可调的场景？"**

### 1.3 应有的设计流程

正确的搜索维度设计流程应该是：

```
Step 1: 画出真实部署架构图
  - 哪些组件在哪个设备上？
  - 设备之间怎么通信？
  - 每个设备上的推理流水线是什么？

Step 2: 识别每个设备上的可调参数
  - 只关注**单个设备**内部的配置
  - 跨设备的通信和调度是另一个维度（通信优化，不是计算优化）

Step 3: 验证每个参数是否有 trade-off
  - 在目标部署设备上（而非开发机上）评估

Step 4: 分析参数间的耦合
  - 只分析同一设备内、或有数据依赖的参数间的耦合
```

如果按这个流程，D1 在 Step 1 就会被排除——因为 ego 和 infra 分属不同设备，它们之间的"并行"由物理架构决定，不是一个可搜索的软件配置。

## 二、延伸反思：仿真环境 vs 部署环境的认知偏差

### 2.1 问题不仅在 D1

这个"仿真 ≠ 部署"的认知偏差可能也影响了其他维度的理解：

**D2 流水线重叠**：在仿真环境中测出的 586ms 端到端延迟包含大量 Python/mmdet3d 框架开销（460ms）。但真实部署中会用 TRT C++ 推理引擎，Python 开销不存在。所以：
- D2 的"理论 125ms 稳态"可能更接近真实部署的实际延迟
- "Python 开销是真实瓶颈"这个结论只在仿真环境中成立，不应外推到部署

**D3 时序缓存**：测量的 AMOTA 是在仿真环境中的（含 mmdet3d 评估管线）。缓存精度的影响应该在部署环境中是一致的（FP16/INT8 的数值行为不依赖运行环境），但延迟和显存数据需要在目标设备上重新测量。

**D4 显存策略**：RTX 4090（24GB）上的显存压力与 Jetson Orin（8/16GB）上完全不同。在 4090 上 waste 12.7% 不算什么，但在 Orin 8GB 上可能是致命的。D4 的结论不应从 4090 直接外推到 Orin。

### 2.2 教训

**教训 1：搜索空间设计必须以目标部署场景为起点，而非以当前仿真代码为起点。**

看代码发现的"可优化点"可能只是仿真实现的产物。应该先画部署架构图，再从部署架构出发推导有哪些可配置参数。

**教训 2：仿真环境的 benchmark 数据需要明确标注适用范围。**

测出的延迟/显存/能耗数据，应该区分哪些是"模型固有的"（如 INT8 缓存的精度影响），哪些是"环境特定的"（如 Python 开销主导的端到端延迟）。前者可以外推，后者不能。

**教训 3：V2X 系统的优化维度应该分为"单设备内"和"跨设备"两类。**

- **单设备内**（车端 ego agent 内部）：流水线重叠、缓存管理、显存策略——这些是真正的 D 空间
- **跨设备**（车端 - 路侧通信）：通信压缩（已在 B2 量化中覆盖）、通信延迟隐藏、通信 - 计算重叠——这是另一个维度，更接近分布式系统优化而非单设备部署优化

D1 的错误就在于把"跨设备"的并行问题当作了"单设备内"的优化来做。

## 三、D 空间修正后的搜索维度

基于以上反思，D 空间应从 4 维缩减为 3 维：

| 维度 | 变量 | 取值 | 适用范围 | 状态 |
|:---:|---|---|---|:---:|
| ~~D1~~ | ~~多 agent 并行~~ | ~~{1, 2}~~ | ~~仿真环境特有~~ | **移除** |
| D2 | 流水线阶段重叠 | {无重叠, backbone-BEV 重叠} | 单设备内（ego agent） | 保留 |
| D3 | 时序缓存管理 | 精度{FP16, INT8} × 帧数{1, 2} | 单设备内 | 保留 |
| D4 | 显存分配策略 | {动态, 碎片整理} | 单设备内 | 保留 |

**D 空间搜索基数**：3 × 3 × 2 = **18 种**（从原来的 216 大幅缩减）

## 四、实验结论的有效性评估

尽管 D1 的设计有误，其他实验结论的有效性如下：

| 发现 | 有效性 | 原因 |
|---|:---:|---|
| D1 多 stream 无收益 | ❌ **无效** | 实验场景不符合真实部署；且实现未真正并行 |
| D2 backbone 可被流水线隐藏 | ✅ 有效 | 模型各阶段的相对延迟是固有属性，不依赖运行环境 |
| D2 Python 开销是当前瓶颈 | ⚠️ **仅限仿真** | 部署环境（TRT C++）无此开销 |
| D3 INT8 缓存免费午餐 | ✅ 有效 | FP16/INT8 的数值行为不依赖运行环境 |
| D3 时序不可禁用 | ✅ 有效 | 模型对时序信息的依赖是固有属性 |
| D3-B1 耦合（INT8-1帧≈FP16-2帧） | ✅ 有效 | 精度 trade-off 是模型固有属性 |
| D4 defrag 降低抖动 44% | ⚠️ **设备相关** | 4090 上的结论，Orin 可能不同 |

## 五、对后续工作的指导

1. **任何新搜索维度的提出**，必须先通过"部署场景审查"：画出目标部署架构，确认该维度在目标架构中存在且可调

2. **benchmark 数据标注**：所有延迟/显存数据需标注"仿真环境 (RTX 4090, PyTorch)" vs "部署环境 (Jetson Orin, TRT)"，避免错误外推

3. **UniV2X 的部署架构图**应该成为 D 空间设计的第一张参考图：

```
真实 V2X 部署架构:

  路侧 RSU (边缘设备)              车端 Jetson Orin
  ┌─────────────────┐              ┌──────────────────────────────┐
  │ Infra Camera    │              │ Ego Camera                   │
  │   ↓             │              │   ↓                          │
  │ Backbone + BEV  │              │ Backbone + BEV encoder       │
  │   ↓             │              │   ↓                          │
  │ Decoder         │   V2X通信     │ Fusion (接收 infra queries)  │
  │   ↓             │ ──────────►  │   ↓                          │
  │ Track queries   │  track queries│ Decoder + Heads              │
  └─────────────────┘              │   ↓                          │
                                   │ Track / Det / Seg / Planning │
                                   └──────────────────────────────┘

  D 空间的搜索维度应全部在车端 Orin 的框内:
    D2: Backbone → BEV 流水线重叠
    D3: prev_bev 缓存管理
    D4: 显存分配策略
```

4. **D1 的实验代码和数据不删除**，但在报告中标注为"仿真环境测试，不适用于真实 V2X 部署"，作为反面教材保留
