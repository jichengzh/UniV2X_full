## 一、D 空间最终状态

经过四次修正（v1→v4）和完整实验验证，D 空间所有维度的最优配置已确定：

| 维度 | 最终状态 | 最优值 | 实验依据 |
|---|---|---|---|
| D1 多agent并行 | **移除** | — | 真实V2X中ego/infra在不同设备，天然并行 |
| D2 流水线重叠 | **PyTorch端收益仅1.06x** | backbone-BEV重叠 | backbone仅占e2e的5.5%(31.7ms/571.9ms) |
| D3a 缓存精度 | **锁定 INT8** | INT8 | AMOTA 0.339≥FP16 0.333，显存-50% |
| D3b 缓存帧数 | **锁定 1帧** | 1帧 | 2帧AMOTA完全相同(0.333=0.333)，只增显存 |
| D4 显存策略 | **锁定 defrag** | defrag | 降44%延迟抖动，无额外成本 |
| E1 共享内存 | **已配置** | cudnn.benchmark=True | 标准PyTorch优化 |
| E2 数据预取 | **已实现** | AsyncDataPrefetcher | 收益0.2%(workers=0) |
| E3 分阶段释放 | **PyTorch无效** | — | CUDA allocator已自动管理 |
| E4 图优化 | **不可用** | — | onnxsim不支持MSDAPlugin自定义算子 |

**结论**：D 空间在 PyTorch 端已完全确定，**无需搜索**。唯一有实测价值的优化是 D3a=INT8 时序缓存。

## 四、核心发现总结

1. **INT8 时序缓存是唯一有实际价值的 D 空间优化**（AMOTA无损+显存-50%）
2. **2帧缓存无精度收益**（与1帧完全相同），D3b 锁定为1帧
3. **时序信息不可禁用**（0帧 AMOTA 崩溃93.7%）
4. **PyTorch 端 85% 的延迟是非模型开销**（486.5ms/571.9ms），backbone 仅占5.5%
5. **EMOS 的四大优化在 PyTorch 端均无效或不可用**：E2 收益0.2%、E3 释放0MB、E4 onnxsim失败、E1 是标准配置
6. **真正的部署加速需要 TRT C++ 推理引擎**消除Python开销后，backbone-BEV重叠才有意义

## 五、推荐部署配置

```
B1 剪枝:  D.1.4 (enc=1.0, dec=0.7)   → AMOTA 0.367, ego_fwd -42%
B2 量化:  BEV INT8                    → TRT 1.9x 加速
D3a:     INT8 时序缓存               → 显存 -50%, 无精度损失
D3b:     1 帧                        → 2帧无收益
D4:      defrag                      → 延迟抖动 -44%
D2:      backbone-BEV 重叠           → TRT 部署后理论 4.77x (PyTorch 端仅 1.06x)
```
