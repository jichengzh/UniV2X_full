## 〇、研究背景确认

### UniV2X 系统概述

UniV2X 是一个**端到端车路协同 (V2X) 自动驾驶感知系统**，核心特点：

- **单相机约束**：V2X-Seq-SPD 数据集中每个 agent 仅有 **1 个前视相机**（`VEHICLE_CAM_FRONT`）。模型架构虽然支持任意 `num_cam`（继承自 nuScenes 的 6 相机设定），但在 SPD 上运行时 `N=1`。因此 camera_batching **不构成搜索维度**——它是数据集约束。

- **双 agent 架构**：
  - **Ego agent**（车端）：完整模型，backbone → BEV encoder → decoder → 多任务头
  - **Infra agent**（路侧）：冻结的子模型（`freeze_bev_encoder=True`），产出 track queries + box predictions

- **V2X 融合方式**：**不是** BEV-to-BEV 融合，而是 **query-level 融合**。Infra 的 track queries 经坐标变换后注入 ego 的 BEV grid（`cross_agent_query_interaction` + `_get_coop_bev_embed`）。

- **完整推理流水线**：

```
img [B, 1, 3, H, W]
  → GridMask → ResNet-101+DCN backbone → FPN neck
  → 4-scale 特征 [B, 1, 256, h, w]
  → BEVFormerEncoder (6 layers):
      TemporalSelfAttention(prev_bev) + SpatialCrossAttention(image feats)
  → bev_embed [200x200, B, 256]
  → [V2X: 注入 infra queries 到 bev_embed]
  → DetectionTransformerDecoder (6 layers):
      900 object queries attend to bev_embed
  → 多任务头: tracking / detection / segmentation / motion / planning
```

- **时序机制**：`prev_bev` 从前一帧传递，经 ego 姿态旋转对齐后送入 TemporalSelfAttention。训练时 `queue_length` 控制时序上下文长度（1.2 已证明这与剪枝率强耦合）。

- **关键计算瓶颈**（1.2 C.1 已实测）：
  - BEV encoder 的 MSDA 采样（memory-bound，128 万次 bilinear sampling）
  - 而非 FFN 矩阵乘（compute-bound）
  - 因此 FFN 结构剪枝在 TRT 端零加速

## 一、搜索空间的判定标准

### 1.1 四个 trade-off 指标

所有搜索维度的评判标准统一为以下四个指标之间的 trade-off：

| 指标 | 定义 | 测量方式 |
|:---:|---|---|
| **精度** | AMOTA / mAP / seg IoU | 验证集评估 |
| **时延** | 单帧端到端推理时间 (ms) | CUDA Event / wall-clock |
| **能耗** | 单帧推理的 GPU 能量消耗 (mJ) | `nvidia-smi` 功率 x 时延；或 `torch.cuda.power_draw` |
| **存储** | 模型体积 + 运行时峰值显存 (MB) | `.pth`/`.engine` 大小 + `torch.cuda.max_memory_allocated` |

### 1.2 判定规则

**放入搜索空间**的条件：该维度的不同取值在上述四个指标上存在**不可消解的 trade-off**——即没有一个取值在所有四个指标上同时最优。

**锁定在搜索空间外**的条件：
- 有唯一最优解（一个取值在所有指标上都优于或等于其他取值）
- 是工程 bug（修复后有唯一最优解）
- 是数据集/硬件的固有约束（没有选择空间）

## 二、D 空间搜索维度（修正版）

> 聚焦 PyTorch 端实现，暂不涉及 TRT 集成和 V2X 通信调度。

### D1: 多 Agent 并行度

**定义**：ego 和 infra agent 的推理使用多少个 CUDA stream 并行执行。

**取值范围**：{1, 2, 3, 4}

| 取值 | 含义 |
|:---:|---|
| 1 | 全串行：ego → infra → fusion → decode |
| 2 | ego 和 infra 的 backbone+BEV 并行（2 streams） |
| 3 | ego backbone、ego BEV、infra backbone+BEV 三路并行 |
| 4 | ego backbone、ego BEV、infra backbone、infra BEV 四路全拆 |

**四指标 trade-off**：

| 指标 | stream=1 | stream=2 | stream=4 |
|:---:|---|---|---|
| 精度 | 基准 | 不变（数值等价） | 不变 |
| 时延 | 最高（全串行） | 降低（并行隐藏 infra 延迟） | 可能进一步降低或**反升**（SM 竞争） |
| 能耗 | 最低（GPU 利用率低） | 更高（并行提升利用率） | 最高（满载） |
| 存储 | 最低 | 更高（两个 engine/context 同驻） | 最高 |

**为什么是搜索维度**：stream 数量增加到一定程度后，GPU SM（Streaming Multiprocessor）资源被多个 stream 分摊，单个 stream 的 kernel 吞吐下降，总时延可能**不降反升**。最优点取决于模型大小（受 B1 剪枝影响）和单 kernel 的资源需求。

#### 与 B1/B2 的双向耦合

**B1/B2 → D1（软件影响硬件选择）**：

- 剪枝后模型更小 → 单 agent 占用更少 SM/显存 → **更高并行度变得可行**
- INT8 量化后每层计算量减半 → 单 stream 的 kernel 更短 → 并行的"可隐藏窗口"更短 → **并行收益减小**
- 激进剪枝 + INT8 可能让模型变成完全 memory-bound → 多 stream 分享带宽 → **并行反而有害**

**D1 → B1/B2（硬件影响软件选择）**：

- 选择高并行度 → 每个 stream 分得的 GPU 资源更少 → **需要更激进的剪枝**来让每个 agent 在有限资源内跑完
- 选择高并行度 → 峰值显存增加 → **量化变得更有价值**（INT8 减半显存占用，为并行腾出空间）
- 选择高并行度 → 总 GPU 功耗更高 → 在边缘设备（Jetson Orin）的功耗墙下，**可能被迫回退到更激进的压缩**

### D2: 流水线阶段重叠

**定义**：推理流水线的不同阶段之间是否在时间上重叠执行。

**取值范围**：{无重叠, backbone-BEV 重叠, 多阶段全重叠}

**流水线结构**（以 ego 单 agent 为例）：

```
无重叠（当前）:
  Frame t:   [Backbone 34ms] → [BEV_enc 65ms] → [Decoder 10ms] → [Seg 89ms]
  Frame t+1:                                                                   [Backbone] → ...
  单帧延迟 = 34 + 65 + 10 + 89 = 198ms

backbone-BEV 重叠:
  Frame t:   [Backbone 34ms] → [BEV_enc 65ms] → [Decoder 10ms] → [Seg 89ms]
  Frame t+1:                   [Backbone 34ms] → [BEV_enc 65ms] → ...
                                ↑ 在 BEV_enc 执行时，下一帧的 Backbone 已经开始
  有效帧间延迟 = max(backbone, bev_enc) + decoder + seg = 65 + 10 + 89 = 164ms
  (首帧仍需全量 198ms，稳态 164ms)

多阶段全重叠:
  Frame t:   [Back] → [BEV ] → [Dec] → [Seg         ]
  Frame t+1:          [Back] → [BEV ] → [Dec] → [Seg ]
  Frame t+2:                   [Back] → [BEV ] → ...
  有效帧间延迟 = max(各阶段) = seg(89ms)
  (需要足够显存同时容纳多帧的中间激活)
```

**四指标 trade-off**：

| 指标 | 无重叠 | backbone-BEV 重叠 | 全重叠 |
|:---:|---|---|---|
| 精度 | 基准 | 不变（各帧独立计算） | 不变 |
| 时延 | 198ms | 164ms（稳态） | 89ms（稳态，理论下限） |
| 能耗 | 最低 | 更高（两阶段并行） | 最高（多阶段并行） |
| 存储 | 最低 | +backbone 中间激活 (~50MB) | +多帧中间激活 (~200MB) |

**为什么是搜索维度**：重叠程度越高，时延越低，但显存和能耗同步增加。在边缘设备上显存有限，可能只能做 backbone-BEV 重叠而无法全重叠。最优重叠程度取决于各阶段的相对延迟（受 B1 剪枝影响——剪枝改变阶段间的延迟比例）。

#### 与 B1/B2 的双向耦合

**B1/B2 → D2（软件影响硬件选择）**：

- **剪枝改变流水线平衡**：1.2 实测 FFN 剪枝让 BEV encoder 从 65ms 降到 55ms，但 backbone（34ms）不变 → 剪枝后 backbone-BEV 的重叠收益从 34ms 变为 min(34, 55-34)=21ms → **剪枝越多，backbone-BEV 重叠收益越小**
- **decoder 剪枝改变瓶颈**：P9 decoder 5 层（10.9→8.7ms），seg_head 基本不变（89ms）→ seg_head 始终是全重叠的瓶颈，decoder 剪枝不影响全重叠的稳态时延
- **量化改变各阶段时延比**：INT8 让 BEV encoder 从 65ms 降到接近 backbone（34ms）→ backbone-BEV 重叠几乎无收益（两者已接近等长）→ **量化后应关注更深层的重叠**

**D2 → B1/B2（硬件影响软件选择）**：

- 选择 backbone-BEV 重叠 → 稳态时延由 max(backbone, BEV) 决定 → **应该剪枝使 BEV 时延 ≤ backbone 时延**（而不是盲目追求最大剪枝率）→ 这给 B1 提供了一个**时延目标约束**
- 选择全重叠 → 稳态时延由最慢阶段 (seg_head 89ms) 决定 → 优化其他阶段的边际收益为零 → **B1 应聚焦 seg_head 剪枝而非 BEV encoder 剪枝**
- 选择重叠 → 需要更多显存 → **B2 量化（减小模型和激活尺寸）变得更有价值**

### D3: 时序缓存管理

**定义**：跨帧传递的 `prev_bev` 的存储精度和帧数。

**子维度**：
- **缓存精度**：{FP16, INT8}
- **缓存帧数**：{0（无时序）, 1, 2}

**组合取值**：{(FP16,0), (FP16,1), (FP16,2), (INT8,1), (INT8,2)} = 5 种

> 注：缓存精度与 B2（量化）的区别——B2 控制**模型层的计算精度**（权重和激活在 forward 中的位宽），D3 控制 **帧间缓存的存储精度**（BEV encoder 输出后、下一帧使用前的存储方式）。即使模型在 FP16 下计算，输出 BEV 也可以被 INT8 压缩存储再反量化使用。两者是独立的决策，但存在耦合（见下文）。

**prev_bev 的数据量**：

```
200 x 200 x 256 = 10,240,000 elements
FP16: 10.24M x 2 bytes = 20.48 MB / frame
INT8: 10.24M x 1 byte  = 10.24 MB / frame

2 agents x 2 frames:
  FP16: 81.92 MB
  INT8: 40.96 MB  → 节省 40.96 MB
```

**四指标 trade-off**：

| 指标 | (FP16, 1) | (INT8, 1) | (FP16, 2) | (INT8, 2) |
|:---:|---|---|---|---|
| 精度 | 基准 | 量化误差累积 | 更好（更多时序） | 时序补偿 vs 量化损失 |
| 时延 | 基准 | 微降（带宽省） | 微升（多一帧 attention） | 折中 |
| 能耗 | 基准 | 略低（少传输） | 略高（多计算） | 折中 |
| 存储 | 20.5 MB | 10.2 MB | 41.0 MB | 20.5 MB |

**为什么是搜索维度**：更多帧、更高精度 → 更好的精度，但更大的存储和时延开销。在边缘设备上，时序缓存的显存占用可能占总显存的 10-20%，精度/存储 trade-off 明显。

#### 与 B1/B2 的双向耦合

**B1/B2 → D3（软件影响硬件选择）**：

- **1.2 Phase B 核心发现**：FFN 剪枝率与最优时序帧数强耦合
  - 30% 剪枝 → q=1 最优（单帧信息已够）
  - 60% 剪枝 → q=2 必须（单帧不足，需时序补偿）
  - 这意味着 D3.缓存帧数 的最优值**直接取决于** B1 的压缩率

- **1.2 Phase D.2 发现**：解耦剪枝 (encoder 不动) 后 INT8 量化近乎无损 (-0.007)，但绑定剪枝后 INT8 严重下降 (-0.048)
  - 如果 B1 选择保守剪枝（encoder 不动）→ BEV 特征分布稳定 → D3.INT8 缓存大概率可行
  - 如果 B1 选择激进剪枝（encoder 也剪）→ BEV 特征分布已被扰动 → D3.INT8 缓存可能雪上加霜

- **B2 量化 → D3 缓存精度**：如果 B2 已经让 BEV encoder 在 INT8 下计算，那么输出 BEV 的有效精度本身就低于 FP16 → D3 用 INT8 缓存几乎不额外损失精度 → **B2=INT8 时 D3.INT8 的代价更低**

**D3 → B1/B2（硬件影响软件选择）**：

- 选择 D3=(INT8, 2) → 时序信息丰富但有量化噪声 → **模型需要对时序量化噪声鲁棒** → 微调时的数据增强应包含时序特征扰动 → 影响 B1 的微调策略
- 选择 D3=(FP16, 0)（无时序）→ 模型完全依赖单帧 → **不能做激进剪枝**（1.2 已证明 60% 剪枝需要时序补偿）→ B1 的剪枝上限被 D3 约束
- 选择 D3 帧数多 → 显存紧张 → **B1/B2 需要更激进的压缩**来腾出显存给时序缓存

### D4: 显存分配策略

**定义**：运行时 GPU 显存的分配和管理方式。

**取值范围**：{PyTorch 默认动态分配, 静态预分配池, 显存碎片整理}

| 取值 | 含义 |
|---|---|
| 动态分配 | PyTorch 的默认 CUDA allocator，按需 malloc/free |
| 静态预分配 | 启动时一次性分配所有需要的显存，运行时零分配 |
| 碎片整理 | `torch.cuda.memory.CUDAPluggableAllocator` 自定义分配器 |

**四指标 trade-off**：

| 指标 | 动态 | 静态 | 碎片整理 |
|:---:|---|---|---|
| 精度 | 不影响 | 不影响 | 不影响 |
| 时延 | 分配延迟抖动 (~0.1-1ms) | 零分配延迟 | 低分配延迟 |
| 能耗 | 基准 | 略低（无分配开销） | 基准 |
| 存储 | 峰值波动大 | **固定预留，可能浪费** | 介于两者之间 |

**为什么是搜索维度**：在边缘设备（Jetson Orin, 16GB 显存）上，动态分配的碎片化可能导致 OOM，而静态预分配会浪费 20-30% 的显存。两者之间存在存储效率 vs 延迟稳定性的 trade-off。

#### 与 B1/B2 的双向耦合

**B1/B2 → D4（软件影响硬件选择）**：

- 剪枝/量化后模型更小 → 显存需求更可预测 → **静态预分配更可行**（浪费比例更小）
- 未压缩的大模型 → 显存波动大 → 静态预分配需预留过多 → **动态分配或碎片整理更合适**

**D4 → B1/B2（硬件影响软件选择）**：

- 选择静态预分配 → 显存预算**硬上限**确定 → 反过来限定 B1/B2 的最低压缩要求（模型 + 激活 + 缓存必须 fit 在预算内）
- 选择动态分配 → 显存约束是"软"的 → B1/B2 有更大的压缩灵活度

## 五、耦合关系全景图

```
B1 (剪枝)                    D (硬件部署)                    B2 (量化)
─────────                    ───────────                    ─────────

剪枝率 ─────────────────────► D1 并行度 ◄──────────────────── INT8 减半显存
  模型更小→更高并行度可行        │                              更多stream可行
                                │
  ◄─────────────────────────────┘
  高并行度→每stream资源少
  →需要更激进剪枝

剪枝改变阶段延迟比 ──────────► D2 流水线重叠 ◄─────────────── 量化改变阶段时延
  FFN剪枝→BEV变快→重叠收益变小    │                           INT8→BEV接近backbone
                                  │                           →重叠收益变小
  ◄───────────────────────────────┘
  重叠选择→决定"应该剪哪个阶段"
  (平衡流水线 vs 最小化瓶颈)

剪枝率→最优缓存帧数 ─────────► D3 时序缓存 ◄──────────────── B2精度→缓存精度可行性
  60%剪枝必须q=2                  │                          INT8计算→INT8缓存自然
  encoder不动→INT8缓存可行        │
                                  │
  ◄───────────────────────────────┘
  D3=(FP16,0)无时序→限制最大剪枝率
  D3帧数多→显存紧→需要更激进压缩

剪枝/量化→显存可预测 ────────► D4 显存策略 ◄──────────────── 量化→显存需求降低
  小模型→静态预分配可行            │
                                  │
  ◄───────────────────────────────┘
  静态预分配→硬显存上限
  →限定B1/B2最低压缩要求
```

**核心洞察**：所有耦合都围绕一个中心问题——**剪枝/量化改变了模型的计算特性（compute-bound vs memory-bound、单帧信息充足 vs 需要时序补偿、各阶段延迟比），从而改变了硬件部署策略的最优选择；反过来，硬件约束（显存上限、并行资源、功耗墙）又限定了软件压缩的必要程度和方向。**

这就是为什么联合搜索有价值——单独优化 B1、B2、D 中任何一个子空间都无法找到全局最优。
