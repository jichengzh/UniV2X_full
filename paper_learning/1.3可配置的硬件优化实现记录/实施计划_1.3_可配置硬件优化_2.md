## 一、核心认知

### 1.1 硬件部署优化的两类本质

| 类型 | 特征 | 方法 | 占比 |
|---|---|---|:---:|
| **工程实现** | 无 trade-off，做了一定更好 | 一次性实现，成为搜索的"地基" | ~80% |
| **搜索维度** | 有 trade-off，最优解取决于 B1/B2 | 与 B1/B2 联合搜索 | ~20% |

### 1.2 修正后的 D 空间（v4）

**搜索维度**（有 trade-off）：

| 维度 | 变量 | 取值 | Trade-off |
|:---:|---|---|---|
| **D2** | 流水线重叠深度 | {无重叠, backbone-BEV 重叠, 全重叠} | 延迟↓ vs 存储↑ |
| **D3b** | 时序缓存帧数 | {1, 2} | 精度↑ vs 存储↑ |

**锁定维度**（已确定最优）：

| 维度 | 锁定值 | 依据 |
|---|---|---|
| D3a 时序缓存精度 | INT8 | 免费午餐（AMOTA ≥ FP16，显存 -50%） |
| D4 显存分配策略 | defrag | 降低 44% 抖动，无额外成本 |

**工程优化底座**（无 trade-off，一次性实现）：

| 编号 | 优化手段 | 对应 EMOS |
|:---:|---|---|
| E1 | 共享内存优化 | SI-1 |
| E2 | 数据异步预取（I/O-计算重叠） | SI-2 |
| E3 | 分阶段内存释放 | SI-3 |
| E4 | 计算图优化 | SI-4 |

### 1.3 D2 与 E2 的关系（重要澄清）

D2 和 E2 **不是同一件事**，而是**两层不同的重叠**，且 E2 是 D2 的前置依赖：

```
E2 数据预取:  CPU 数据加载 与 GPU 推理并行       → CPU-GPU 重叠
D2 流水线:    GPU backbone(t+1) 与 GPU BEV(t) 并行 → GPU-GPU 重叠

依赖关系: D2 需要 frame t+1 的数据提前在 GPU 上 → 这正是 E2 的功能
```

**完整流水线的三个层次**：

```
层次 0 (当前，无优化):
  CPU: [load t] ─────────────────── [load t+1] ─────────────────────
  GPU:           [back→BEV→dec→seg]              [back→BEV→dec→seg]
  帧间延迟 = CPU_load + GPU_full = ~50ms + ~198ms = ~248ms (纯模型视角)

层次 1 (仅 E2，CPU-GPU 重叠):
  CPU: [load t][load t+1][load t+2] ...
  GPU:         [back→BEV→dec→seg][back→BEV→dec→seg] ...
  帧间延迟 = max(CPU_load, GPU_full) = max(50, 198) = 198ms
  收益: 隐藏 CPU 数据加载时间

层次 2 (E2 + D2，CPU-GPU + GPU-GPU 双层重叠):
  CPU: [load t][load t+1][load t+2] ...
  GPU: [back t] → [BEV t ] → [dec t] → [seg t     ]
                  [back t+1] → [BEV t+1] → [dec→seg]
  帧间延迟 = max(各 GPU 阶段) = seg(89ms)
  收益: backbone(32ms) 完全隐藏在上一帧的后续阶段中
```

因此：**E2 是实现 D2 的前置基础设施**，在计划中 E2 应放在 D2 之前实现。

#### 0.2 E2 数据异步预取实现（预计 1 天）

**为什么在 Phase 0**：E2 是 D2 流水线重叠的前置依赖。没有数据预取，D2 无法启动下一帧的 backbone。

**实现内容**：

```python
class AsyncDataPrefetcher:
    """异步数据预取：当前帧 GPU 推理时，CPU 加载并预处理下一帧"""

    def __init__(self, data_loader):
        self.loader_iter = iter(data_loader)
        self.stream = torch.cuda.Stream()
        self.next_data = None
        self._prefetch()

    def _prefetch(self):
        try:
            self.next_data = next(self.loader_iter)
        except StopIteration:
            self.next_data = None
            return
        with torch.cuda.stream(self.stream):
            # 异步 CPU→GPU 搬运
            for k, v in self.next_data.items():
                if isinstance(v, torch.Tensor):
                    self.next_data[k] = v.cuda(non_blocking=True)

    def get_next(self):
        torch.cuda.current_stream().wait_stream(self.stream)
        data = self.next_data
        self._prefetch()
        return data
```

**验证**：
- [ ] 对比有/无预取的端到端延迟
- [ ] 确认预取不影响 AMOTA（数据内容不变）

#### 0.3 D2 流水线重叠完整实现（预计 2 天）

**现状**：只有模块级 hook 计时的理论分析，未实现真正的帧间 GPU 阶段重叠。

**实现核心**：将 ego agent 的 forward 拆分为可独立调度的阶段。

```python
class PipelinedEgoForward:
    """帧间流水线：backbone(t+1) 与 BEV+dec+seg(t) 在不同 stream 并行"""

    def __init__(self, ego_model, overlap='none'):
        self.model = ego_model
        self.overlap = overlap
        self.backbone_stream = torch.cuda.Stream()
        self.pending_feat = None  # 上一次预取的 backbone 输出

    def run_backbone(self, img, img_metas):
        """Stage 1: Backbone + FPN（可在独立 stream 上执行）"""
        img_feats = self.model.extract_img_feat(img, img_metas)
        return img_feats

    def run_bev_decoder_seg(self, img_feats, prev_bev, img_metas, ...):
        """Stage 2-4: BEV encoder + Decoder + Seg（依赖 backbone 输出）"""
        bev = self.model.pts_bbox_head.get_bev_features(img_feats, prev_bev, ...)
        det = self.model.pts_bbox_head.get_detections(bev, ...)
        seg = self.model.seg_head(bev, ...) if self.model.with_seg_head else None
        return bev, det, seg

    def forward_frame(self, data, next_data=None):
        """单帧推理，同时预取下一帧 backbone"""
        # 使用当前帧的 backbone 结果（可能是上一帧预取的）
        if self.pending_feat is not None:
            img_feats = self.pending_feat
            self.pending_feat = None
        else:
            img_feats = self.run_backbone(data['img'], data['img_metas'])

        # 如果有下一帧数据且开启重叠，异步启动下一帧 backbone
        if self.overlap != 'none' and next_data is not None:
            with torch.cuda.stream(self.backbone_stream):
                self.pending_feat = self.run_backbone(
                    next_data['img'], next_data['img_metas']
                )

        # 当前帧的 BEV + Decoder + Seg（在默认 stream 上）
        bev, det, seg = self.run_bev_decoder_seg(img_feats, ...)

        # 等待下一帧 backbone 完成（如果有）
        if self.pending_feat is not None:
            torch.cuda.current_stream().wait_stream(self.backbone_stream)

        return bev, det, seg
```

**关键挑战**：
- 需要侵入 `univ2x_track.py` 和 `univ2x_e2e.py` 的 forward 流程
- `extract_img_feat` 和 `get_bev_features` 的接口需要确认可独立调用
- infra agent 的结果注入时机需要与 pipeline 对齐

**验证**：
- [ ] backbone-BEV 重叠模式下 AMOTA 与无重叠一致
- [ ] 实测稳态帧间延迟（目标：接近理论值 125ms）
- [ ] 全重叠模式的可行性验证（显存是否足够）

### Phase A：工程优化底座（E1, E3；E4 最后）

> 在 D2/D3b 完整实现后，补充搜索空间外围的工程优化。

#### A.1 E3 分阶段内存释放（优先级最高，预计 2 天）

**目标**：在推理流水线的阶段边界主动释放已完成阶段的 GPU 资源。

**UniV2X 阶段边界定义**（参考 EMOS Fig. 6）：

```
Boundary 1: Backbone + FPN 完成
  → 释放: 原始图像 tensor, GridMask 中间结果
  → 保留: 多尺度特征图

Boundary 2: BEV Encoder 完成
  → 释放: 多尺度特征图, SCA/TSA 中间激活
  → 保留: bev_embed

Boundary 3: Detection Decoder 完成
  → 释放: decoder 中间激活, attention 中间结果
  → 保留: track queries + bbox predictions

Boundary 4: Seg Head 完成
  → 释放: seg_head 中间激活
  → 保留: 分割输出
```

**实现方式**：

```python
class StagedMemoryManager:
    """分阶段 GPU 内存管理（参考 EMOS SI-3）"""

    def __init__(self):
        self.stage_tensors = {}  # stage_name -> list of tensor refs

    def register(self, stage_name, *tensors):
        """注册某阶段的中间张量"""
        self.stage_tensors[stage_name] = list(tensors)

    def release_stage(self, stage_name):
        """在阶段边界调用，释放该阶段的中间张量"""
        tensors = self.stage_tensors.pop(stage_name, [])
        for t in tensors:
            del t
        torch.cuda.empty_cache()
```

**预期效果**：
- 峰值显存从 ~2476MB 降至 ~1800-2000MB
- 在 Jetson Orin 8GB 上可能是"能跑 vs 不能跑"的关键

**验证**：
- [ ] 对比实施前后的 `torch.cuda.max_memory_allocated()`
- [ ] 确认 AMOTA 不变

#### A.3 E4 计算图优化（最低优先级，预计 0.5 天）

**放在最后的原因**：TRT builder 在构建 engine 时会自动完成大部分图优化（常量折叠、算子融合、LayerNorm 合并等）。手动优化的价值仅在使用 ONNX Runtime 或 PyTorch 直接推理时。

**实现**（当且仅当使用 ORT 或需要 ONNX 图简化时）：
```bash
# ONNX 图简化
python -m onnxsim input.onnx output_sim.onnx

# ORT 图优化
python -m onnxruntime.transformers.optimizer \
    --input input.onnx --output output_opt.onnx --opt_level 2
```
