# CASCO: Cascaded Co-Optimization for Holistic Neural Network Acceleration

> 论文来源: DATE 2024, 华为技术
> 核心主题: DNN加速器的硬件-软件全栈协同优化

---

## 1. 问题背景与动机

传统的DNN加速器设计存在两个关键缺陷:

1. **HW-SW割裂**: 现有协同设计方法(如HASCO, FAST, DIGAMMA)虽然同时搜索硬件配置和算子映射,但假设每一层独立映射到硬件上执行,忽略了**层融合(Layer Fusion)**带来的巨大收益。
2. **层融合与协同设计脱节**: 层融合工具(如Optimus, DeFiNES)通常在固定硬件上运行,无法反馈影响硬件设计决策。简单地将层融合工具堆叠在协同设计框架之上,效果远不如联合优化。

CASCO的核心思想: 将**硬件架构搜索**、**片上算子映射**、**片外层融合**三个维度纳入一个统一的级联优化框架,并设计高效的搜索算法来应对由此产生的超大联合搜索空间。

## 2. 搜索空间的构建

### 2.1 硬件搜索空间

硬件设计空间由两类原语(primitives)定义:

| 原语类别 | 具体参数 | 含义 |
|---------|---------|------|
| **架构原语** | Memory Level | 片上/片外缓冲层级数(L1, L2, DRAM) |
| | Memory Capacity | 每一级的存储容量 |
| | PE Number | 处理单元数量 |
| | PE Shape | PE阵列形状与规模 |
| | NoC Topology | 片上网络拓扑,决定PE间通信方式 |
| **数据流原语** | Partitioning Size | 数据分区大小 |
| | Loop Unrolls | 循环展开方式 |
| | Replication Loop | 复制循环,三者共同定义循环组织的子集 |

这些原语的组合构成了硬件设计空间。在实验中,PE数量以步长1搜索,片上存储容量以8B步长搜索,片外存储以16KB步长搜索。同时施加面积约束(5e3 um^2)和功耗约束(≤2W),约束后PE数≤384,L2≤4KB。

### 2.2 软件搜索空间

软件搜索空间包含两个子空间,但需要注意:**论文对软件搜索空间具体参数的描述是不充分的**,其重点贡献在于三层联合搜索框架和搜索预算分配算法,而非软件空间本身的定义。

**a) 层融合空间 (Layer Fusion)**

这里的"层"指DNN中的网络层,即Conv、BN、ReLU、Pooling等算子节点。层融合的含义是将多个原本逐层执行的算子合并为一组,在片上存储中连续计算,避免每层都将中间结果写回片外DRAM。

```
不融合（逐层执行）:
  Conv1 → 写回DRAM → 读取 → Conv2 → 写回DRAM → 读取 → Conv3
  每层之间都有片外存储搬运开销

融合后:
  [Conv1 → Conv2 → Conv3]  作为一个融合组
  中间结果全部留在片上缓冲，只在组的输入/输出时访问DRAM
```

基于DNN的DAG(有向无环图)表示来构建层融合搜索空间:

- 网络表示为 `D = (N, E)`,N为算子节点,E为连接边
- **搜索变量是一个离散的分组方案(partition)**,将N个节点划分为若干融合组
- 融合必须满足**数据依赖约束**: 不能将存在未满足依赖关系的节点放入同一组
- 存在节点重复计算的权衡: 例如 `{n1,n2}`, `{n1,n3}`, `{n1,n4}` 都包含n1导致重复计算,而合并为 `{n1,n2,n3,n4}` 只计算一次n1但需要更大的片上缓冲

```
具体搜索变量示例（8个节点的网络）:
  方案A: {n1,n2,n3}, {n4,n5}, {n6,n7,n8}
  方案B: {n1,n2}, {n3,n4,n5,n6}, {n7,n8}
  方案C: {n1}, {n2}, ..., {n8}   ← 完全不融合（退化为逐层执行）
```

本质上是一个**组合优化问题**,没有连续参数可调,搜索空间大小受DAG依赖关系约束。

**b) 算子映射空间 (Operator Mapping)**

给定一个硬件配置和一个算子(或融合算子组),算子映射定义了数据在硬件上的排布和计算调度方式。**论文直接复用Timeloop[13]的启发式搜索方法,未展开任何细节。**

根据Timeloop论文本身的定义,算子映射的可配置变量应包括:

| 参数 | 含义 |
|------|------|
| Loop Order | 计算嵌套循环的排列顺序 |
| Tiling Factors | 每层循环在各存储层级上的分块粒度 |
| Parallelism Mapping | 哪些循环维度映射到PE阵列的空间维度 |

但CASCO论文将这部分完全作为**黑盒子程序**调用,如需深入理解具体参数,需参考Timeloop原文[13]。

**小结**: 软件搜索空间中,层融合的搜索变量是明确的(DAG节点分组方案),算子映射的搜索变量则被论文抽象为黑盒,未给出具体可配置参数。

## 3. 硬件空间的探索方法

### 3.1 多目标贝叶斯优化 (MoBo)

CASCO使用MoBo来引导硬件搜索空间探索,核心机制是**高保真代理模型更新策略**:

**搜索流程:**
1. 每轮迭代,通过最小化获取函数(如Hyper-volume Improvement)采样一批N个硬件候选
2. 对每个硬件候选并行执行软件栈搜索
3. 用高保真数据点更新代理模型

**高保真更新策略:**

定义保真标量 `vParEGO`:

```
vParEGO = max_{j∈{1,2,3,4}} (w_j * y_j) + ρ * Y^T * W
```

计算每个硬件配置与当前最优配置的L2距离:

```
d = ||vParEGO - vBest_ParEGO||_2
```

引入**上限更新参数(UUL)**作为筛选阈值:
- 只有 `d ≤ UUL` 的硬件配置才被选入来更新代理模型
- UUL在每轮迭代后更新为数据集D的α-分位数(95%置信度)
- 随着迭代推进,UUL趋于减小,筛选标准越来越严格

**这样做的优势:**
- 代理模型只用高质量数据训练,避免被低质量设计点干扰
- 提高找到更优硬件配置的概率
- 加速全局最优点的收敛

### 3.2 改进的连续减半 (Modified Successive Halving, MSH)

在MoBo的每轮迭代内部,CASCO通过MSH来**提前终止不良硬件候选**:

**标准连续减半 (SH):**
- 每轮只保留当前最好的一半候选,基于终值(Terminal Value, TV)筛选

**CASCO的改进 (MSH):**
- 同时考虑**终值(TV)**和**曲线下面积(AUC)**来决定淘汰
- 动机: 收敛速率陡峭的候选也可能是有潜力的,即使当前终值不是最好,给予更多预算后可能反超
- 这避免了过早淘汰"慢热"但潜力大的硬件配置

**资源分配效果:**
- 有前途的硬件候选获得更多的软件搜索预算
- 不利候选被尽早淘汰,节省计算资源
- 实现了搜索预算的自适应分配

## 4. 软件空间的探索方法

### 4.1 层融合搜索: 遗传算法 (EA)

**为什么不用动态规划(DP)?**

之前的工作(如Optimus)使用DP搜索融合组,时间复杂度为 `O(2^{max_i |succ(n_i)|} * |N|)`,随网络规模指数增长。在协同搜索场景下,层融合搜索作为内层被反复调用,DP的开销不可接受。

**EA的搜索流程:**

1. **初始化**: 随机生成一组融合方案作为初始种群
2. **评估**: 计算每个方案的EDP目标 `J = (Off-Chip Energy + On-Chip Energy) * cycles`
3. **选择**: 基于目标值选择当前最优种群
4. **交叉 (Crossover)**: 从top种群中随机选两个父代P1, P2,沿DAG节点顺序,依次为每个节点随机选择P1或P2中对应的融合组,标记组内所有节点为已访问,直到覆盖所有节点
5. **变异 (Mutation)**: 随机应用以下三种操作之一:
   - **Merge**: 合并两个相邻融合组为一个更大的组
   - **Split**: 将一个组拆分为两个更小的组
   - **Steal**: 选择一个边界节点,从当前组移到相邻组
6. 重复直到预算用尽

**EA的优势:**
- 搜索预算可灵活控制(通过代数),不像DP必须等待完整流程
- 交叉操作继承并混合父代中的优秀融合组
- Merge/Split提供大幅修改能力,Steal提供微调能力,平衡了利用与探索

### 4.2 算子映射搜索

对每个融合组,使用Timeloop的启发式搜索方法寻找最优的片上数据映射方式,计算对应的片上能耗和延迟。

片外能耗由Optimus的内存代价模型计算,捕获融合层组的最小可达片外存储开销。

## 5. 搜索加速方法总结

CASCO采用了多种方法来应对超大联合搜索空间:

| 加速方法 | 作用层次 | 核心机制 |
|---------|---------|---------|
| **MoBo高保真更新** | 硬件搜索 | 只用高质量数据训练代理模型,避免噪声干扰,加速收敛 |
| **改进连续减半(MSH)** | 硬件-软件衔接 | 提前淘汰不良硬件候选,自适应分配软件搜索预算 |
| **遗传算法替代DP** | 层融合搜索 | 将指数级复杂度降为可控的预算分配,支持灵活的搜索预算控制 |
| **批量采样+并行评估** | 整体流程 | 每轮MoBo采样一批硬件候选,并行执行各自的软件搜索 |
| **级联结构本身** | 整体架构 | 将三层优化组织为嵌套结构(HW外层→融合+映射内层),利用层次化分解降低联合搜索难度 |

### 6.3 性能评估方法

论文中的性能评估**不是在真实硬件上运行**,而是使用**分析模型(Analytical Model)**估算:

| 评估指标 | 计算来源 | 说明 |
|---------|---------|------|
| **片上能耗** (On-Chip Energy) | Timeloop | 给定HW配置+算子映射,Timeloop模拟计算数据搬运和运算的能耗 |
| **延迟** (Cycles) | Timeloop | 同上,Timeloop模拟计算总周期数 |
| **片外能耗** (Off-Chip Energy) | Optimus代价模型 | 根据融合分组方案,计算各组之间的DRAM访问量,乘以每次访问的能耗 |
| **面积** (Area) | 硬件参数直接计算 | 由PE数量、存储容量等硬件参数确定 |
| **功耗** (Power) | 硬件参数直接计算 | 由PE数量、工作频率等确定 |

最终优化目标为**EDP (Energy-Delay Product)**:

```
J = (Off-Chip Energy + On-Chip Energy) × Cycles
```

这是一个多目标优化问题中的标量化目标,同时优化能耗和延迟。面积和功耗作为约束条件而非优化目标。

### 6.4 最优配置的产生

搜索结束后,CASCO并不输出单一最优解,而是维护一个**Pareto前沿集合**:

```
搜索过程中:
  · 每轮MoBo迭代产生若干(HW, 融合方案, 映射)三元组及其PPA指标
  · 非支配解被加入Pareto前沿集合
  · 被新解支配的旧解从集合中移除

搜索结束后:
  · Pareto前沿上有多个设计点,代表不同的能耗-延迟-面积权衡
  · 论文选取 min-Euclidean distance 的设计点作为最终推荐方案
  · 即: 选择在多个目标维度上最"均衡"的设计点
```

### 6.5 一个具体的执行示例

假设搜索ResNet18,MoBo迭代T=10轮,每轮采样N=8个硬件候选:

```
第1轮MoBo:
  采样8个HW候选: HW_1(PE=64,L2=2KB,...), HW_2(PE=128,L2=1KB,...), ...
  │
  MSH第1轮(预算b1=10代EA):
    对8个HW各自运行10代EA层融合搜索
    → HW_1: EDP=5000, HW_2: EDP=3200, ..., HW_8: EDP=8100
    淘汰后4个(TV+AUC最差的): 淘汰HW_1, HW_5, HW_6, HW_8
  │
  MSH第2轮(预算b2=20代EA):
    对存活的HW_2, HW_3, HW_4, HW_7继续搜索
    → HW_2: EDP=2800, HW_3: EDP=3100, HW_4: EDP=2500, HW_7: EDP=4000
    淘汰后2个: 淘汰HW_3, HW_7
  │
  MSH最终轮(预算b3=40代EA):
    HW_2最终EDP=2400, HW_4最终EDP=2100
  │
  高保真筛选:
    HW_4的vParEGO距离vBest较近(d=0.3 ≤ UUL=0.5) → 选为高保真点
    HW_2的vParEGO距离较远(d=0.8 > UUL=0.5) → 不用于更新代理模型
  │
  更新代理模型,更新Pareto前沿

第2轮MoBo:
  代理模型已学到"PE=100-200, L2=2-3KB区域更有前途"
  采样偏向该区域的8个新HW候选
  重复上述流程...

  ...

第10轮MoBo:
  收敛,Pareto前沿稳定
  选取min-Euclidean距离的设计点:
    最终输出: HW=(PE=160, L2=2.5KB, NoC=Mesh, ...)
              融合方案: {Conv1,BN1,ReLU1,Conv2,BN2}, {ReLU2,Conv3,...}, ...
              算子映射: 每个融合组的Timeloop最优映射
```

## 7. 软硬件协同实现流程（非串行）

### 7.1 与串行方法的本质区别

**串行方法（如HASCO → Optimus堆叠）:**

```
第1步: 搜索硬件 → 确定HW配置（此时不知道层融合的需求）
第2步: 固定HW → 搜索算子映射（此时不知道哪些层会被融合）
第3步: 固定HW+映射 → 搜索层融合（硬件已无法调整）
→ 各步独立,后续信息无法反馈到前面的决策
```

**CASCO的协同方法:**

```
硬件搜索、层融合搜索、算子映射搜索在每一轮迭代中同时进行:
  · 硬件配置决定了片上存储大小 → 约束了层融合组能多大
  · 层融合分组方式决定了片外通信量 → 反馈影响硬件评分和淘汰
  · 算子映射决定了片上能耗和延迟 → 反馈影响融合方案的选择
→ 三者的评估结果在每轮迭代中交叉影响,形成闭环
```

### 7.2 协同流程逐步拆解

以下按照模板格式,描述CASCO各层是如何协同而非串行工作的:

---

**a) 硬件架构层: 以软件栈的联合EDP为指导信号搜索加速器配置**

- 硬件配置不是独立搜索出来的,每个硬件候选的**评分来自其上运行的完整软件栈**(层融合+算子映射)的EDP结果
- MoBo代理模型学到的是"什么样的硬件配置能让软件栈整体运行最优",而非"什么硬件参数本身看起来最好"
- 这意味着硬件搜索天然感知了层融合和算子映射的需求

```
协同体现:
  HW候选(PE=64, L2=1KB) → 软件栈搜索 → EDP=5000（L2太小,融合组只能很小,片外通信大）
  HW候选(PE=64, L2=4KB) → 软件栈搜索 → EDP=2000（L2足够,融合组可以大,片外通信小）
  → 代理模型学到: "增大L2比增大PE数量更能降低EDP" → 下轮采样偏向大L2方向
  → 这个信号来自层融合的结果,串行方法中硬件搜索无法获得
```

---

**b) 层融合层: 在给定硬件约束下,通过DAG分组优化片外通信调度**

- 层融合搜索不是在"理想硬件"上进行的,而是在**每个具体硬件候选**的约束下进行
- 不同的HW配置（不同的L2容量）会导致完全不同的最优融合方案
- 融合组的大小直接受限于片上存储容量: L2越大,融合组可以越大,片外通信越少

```
协同体现:
  同一个网络,在不同HW下的最优融合方案不同:
    HW_A(L2=1KB): 最优融合 = {Conv1,BN1}, {ReLU1,Conv2}, ...  （小组,受限于L2）
    HW_B(L2=4KB): 最优融合 = {Conv1,BN1,ReLU1,Conv2,BN2}, ... （大组,L2够用）
  → 融合方案和硬件配置是耦合的,不能先确定一个再确定另一个
```

---

**c) 算子映射层: 针对融合组的计算特征,结合加速器结构选择数据流与调度策略**

- 算子映射不是对"单个算子"做的,而是对**融合组整体**做的
- 融合组内多个算子共享片上缓冲,映射需要考虑组内数据复用
- 映射结果（片上能耗+延迟）直接作为融合方案的评估信号,影响EA的选择和进化方向

```
协同体现:
  融合方案A: {Conv1(3x3), Conv2(1x1)}
    → Timeloop映射搜索: 发现两层共享输入特征图,最优映射利用输入复用
    → 片上能耗=10mJ, 延迟=100cycles

  融合方案B: {Conv1(3x3)}, {Conv2(1x1)}  （不融合）
    → Timeloop分别映射: 无法利用输入复用,中间结果需写回DRAM
    → 片上能耗=8mJ, 但片外能耗=20mJ, 延迟=200cycles

  → EA根据EDP选择方案A更优 → 进化方向偏向更大的融合组
  → 但如果HW的L2太小,方案A不可行 → 倒逼HW搜索方向调整
```

---

**d) MSH连续减半: 实现三层之间的实时资源调配**

- MSH不是单纯的"淘汰机制",而是三层搜索之间的**信息通道**
- 硬件候选的淘汰依据来自软件搜索的中间结果（而非硬件参数本身）
- 存活的硬件候选获得更多软件搜索预算,搜索精度更高,反过来又提供更准确的硬件评估信号

```
协同体现:
  MSH第1轮: 8个HW候选各获10代EA预算 → 粗略EDP排序 → 淘汰4个
  MSH第2轮: 存活4个各获20代EA预算 → 更精确的EDP → 淘汰2个
  MSH第3轮: 存活2个各获40代EA预算 → 高精度EDP → 最终评估

  → 好的HW候选: 获得70代EA预算,层融合搜索充分,评估可靠
  → 差的HW候选: 只用了10代EA预算就被淘汰,节省计算资源
  → 搜索预算在HW和SW之间动态流动,而非预先固定分配
```

### 7.3 协同闭环总结

```
                    ┌──────────────────────────────────────┐
                    │         MoBo代理模型更新               │
                    │   学到"什么HW配置让整体EDP最优"         │
                    └───────────┬──────────────────────────┘
                                │ 指导下轮HW采样方向
                                ▼
┌──────────┐    约束融合组大小    ┌──────────┐    约束映射空间    ┌──────────┐
│  硬件配置  │ ──────────────→ │  层融合    │ ──────────────→ │  算子映射  │
│ (PE,L2,  │                  │ (DAG分组)  │                  │ (数据流,  │
│  NoC...) │                  │            │                  │  调度)    │
└──────────┘                  └──────────┘                  └──────────┘
      ▲                            │                            │
      │                            │                            │
      │         EDP评估信号          │      片上能耗+延迟          │
      │ ◄──────────────────────────┤ ◄──────────────────────────┘
      │                            │
      │    片外能耗反馈              │
      │ ◄──────────────────────────┘
      │
      │  MSH: 根据软件搜索中间结果淘汰HW候选
      │  高保真更新: 筛选优质HW数据点更新代理模型
      └──── 闭环: 软件搜索结果反向驱动硬件搜索方向 ────┘
```

**核心要点: 三层搜索不是"先HW→再融合→再映射"的串行流水线,而是在每轮MoBo迭代中同时进行、相互约束、相互反馈的闭环。** 硬件配置约束了软件搜索的可行域,软件搜索的EDP结果反过来驱动硬件搜索的方向——这就是"协同"而非"串行"的本质。

## 8. 实验验证方法

### 7.1 对比固定硬件架构

将CASCO生成的加速器与三个知名固定架构对比:
- **Eyeriss** [Chen et al., 2016]
- **ShiDianNao** [Du et al., 2015]
- **EdgeTPU** [Yazdanbakhsh et al., 2021]

在6个网络(MobileNet, AlexNet, SqueezeNet, VggNet, ResNet18, GoogleNet)上,使用各基线硬件的资源约束作为CASCO的搜索空间上界:

| 对比基线 | 平均加速比 | 平均EDP节省 |
|---------|----------|-----------|
| Eyeriss | 1.2x | 1.3x |
| ShiDianNao | 1.7x | 1.7x |
| EdgeTPU | 2.3x | 2.3x |

验证目的: 证明在相同资源约束下,协同优化能找到比人工设计更好的加速器。

### 7.2 对比协同设计方法

设置两个基线:
- **Baseline 1 (HASCO)**: 无层融合的HW-SW协同设计
- **Baseline 2 (HASCO + Optimus)**: HASCO搜索硬件,再用Optimus做层融合(堆叠方式)

**逐网络单独优化 (Table I):**

| 对比 | 平均能耗降低 | 平均加速 |
|------|-----------|---------|
| CASCO vs HASCO | 5.8x | 2.2x |
| CASCO vs HASCO+Optimus | 3.8x (EDP综合) | - |

在9个网络上验证(新增ResNet50, UNet, SRGAN, VIT)。

**验证目的:**
- vs HASCO: 证明加入层融合的价值
- vs HASCO+Optimus: 证明联合优化优于简单堆叠,CASCO的搜索算法是另一个必要组件

### 7.3 泛化性验证 (Table II)

在2个训练网络(ResNet18, SRGAN)上协同优化生成硬件设计,然后在4个未见网络(MobileNet, ResNet50, UNet, VIT)上验证:

| 验证网络 | EDP降低(vs HASCO+Optimus) |
|---------|-------------------------|
| MobileNet | 3.24x |
| ResNet18 (训练集) | 2.74x |
| ResNet50 | 2.67x |
| VIT | 1.75x |
| UNet | 1.43x |

验证目的: 证明CASCO生成的硬件设计具有跨网络泛化能力。论文解释,层融合优化作为"高层正则化"驱动硬件设计同时对片上/片外通信友好,而非只针对特定网络过拟合。

### 7.4 收敛性分析 (Fig. 4)

通过Hyper-volume收敛曲线和EDP收敛曲线,展示CASCO相比基线方法具有更快的收敛速度和更优的最终结果,验证了自适应搜索算法的有效性。

## 9. 核心启示

1. **协同优化的层次性**: 真正的软硬件协同优化不是各层独立优化后拼装,而是让各层优化的目标和反馈信号能够跨层传递。
2. **搜索效率是关键瓶颈**: 联合搜索空间的爆炸性增长使得暴力搜索不可行,自适应资源分配(如MSH)和高效代理模型(如高保真MoBo)是解决这一问题的关键。
3. **层融合的双重作用**: 层融合不仅直接降低片外通信开销,还作为正则化信号引导硬件设计,使其对数据搬运更友好。
4. **遗传算法在结构化离散空间的适用性**: 层融合空间是高度结构化的离散空间,定制化的交叉和变异操作能有效利用DAG结构信息进行探索。
