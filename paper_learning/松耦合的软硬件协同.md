# 松耦合 vs 紧耦合的软硬件协同

> 以 EMOS (TPAMI 2026) 为例，对比我们设计的联合搜索方案，分析两种不同层次的软硬件协同

---

## EMOS 算不算软硬件协同？

**算，但属于一种比较"松耦合"的软硬件协同**，和我们设计的联合搜索方案有本质区别。

---

## 核心差异对比

### 1. 协同的"层次"不同

| 维度 | EMOS（松耦合） | 我们的方案（紧耦合） |
|------|----------------|---------------------|
| **软件侧做了什么** | 设计MoE算法架构（专家选择、路由机制、多模态融合） | 剪枝率、量化位宽、通信压缩等**可调参数** |
| **硬件侧做了什么** | ONNX/TRT部署优化、零拷贝缓冲、线程管理、共享内存优化 | 部署配置（图分割、内存池、多流策略、精度覆盖等**可调参数**） |
| **两者如何"协同"** | **串行**：先设计好算法 → 再手动做系统工程优化 | **联合搜索**：软件参数和硬件参数在同一优化循环里自动搜索 |

简单说：
- **EMOS**：算法设计者人工考虑了"这个算法将来要部署在边缘"，然后手动做了系统优化——这是**设计意识上的协同**
- **我们的方案**：把软件决策和硬件决策放进同一个搜索空间，用贝叶斯优化自动找联合最优——这是**优化机制上的协同**

### 2. "协同"发生在什么阶段

```
EMOS的流程（松耦合）:
  算法设计(MoE架构) → 训练 → 手动系统优化(ONNX/TRT/线程/内存) → 部署
       ↑                              ↑
    人的设计经验               人的工程经验
    (完全是手动的)             (完全是手动的)
    
    两个阶段之间没有自动化的反馈回路

我们的流程（紧耦合）:
  固定网络结构 → 自动化搜索 [剪枝率 × 量化位宽 × 部署配置] → 蒸馏/QAT → 部署
                     ↑                    ↑
               软件子空间            硬件子空间
                     └──── 联合优化 ────┘
                           ↑
                     自动化反馈回路
                (精度/延迟/显存共同评估)
```

### 3. 对"硬件特性"的感知深度不同

**EMOS 对硬件的感知**：
- 知道 Jetson 有 GPU SM、共享内存
- 知道 TRT 可以做算子融合、graph rewriting
- 手动写了 CUDA 优化（稀疏3D卷积的并行化、线程分4阶段管理）
- 但这些都是**工程实现**，不是**自动化搜索**

**我们的方案对硬件的感知**：
- Phase 0 构建了 `latency_LUT[layer_type, ch, bits]` — 对硬件实际行为做了profiling
- 搜索空间里包含 `memory_pool_strategy`、`multi_stream_strategy`、`camera_batching` 等硬件执行策略
- 贝叶斯优化的目标函数里**同时包含精度、延迟和显存**三个来自硬件的反馈
- 这意味着：**如果某个剪枝率配合某个内存池策略在Jetson上反而更慢，搜索算法会自动发现并避开**

### 4. 可迁移性不同

**EMOS** 的系统优化高度特化于特定硬件：
- 换一个平台（比如从 Jetson 换到 FPGA），大量 CUDA 优化代码需要重写
- 算法设计本身不会自动调整

**我们的方案** 迁移时：
- 换平台 → 重新跑 Phase 0 的硬件 profiling → 得到新的 `latency_LUT`
- 搜索空间里的硬件子空间可能需要调整（比如 FPGA 没有 DLA offload）
- 但**搜索框架本身不变**，自动化搜索会找到新平台上的最优组合

---

## 一句话总结

| | EMOS（松耦合） | 我们的方案（紧耦合） |
|--|----------------|---------------------|
| **协同方式** | 人工经验驱动的"算法+系统工程"二步走 | 自动化搜索驱动的软硬件参数联合优化 |
| **协同强度** | 松耦合（设计理念上考虑硬件，但没有自动化反馈） | 紧耦合（软硬件参数在同一优化循环里交互影响） |
| **核心贡献** | MoE算法创新 + 工程优化能力 | 联合搜索框架 + 自动化最优配置发现 |

EMOS 更像是"**一个好的算法工程师同时也是好的系统工程师**"做出来的工作——他在设计算法时心里有硬件，部署时认真做了优化。但**软件决策和硬件决策之间没有形成闭环优化**。

我们的方案则是试图回答一个更深的问题：**当剪枝率、量化位宽、部署拓扑这些决策相互耦合时，如何自动找到全局最优？** 这才是"协同机制不清"这个难点的核心——EMOS 绕开了这个问题（靠人工分步处理），我们试图正面解决它。
