> 基于 survey_raw 中 49 篇论文 + CASCO + EMOS + 我们的方案，提炼搜索空间的层次结构
> 目标：回答"软硬件协同优化中，到底在搜什么？"

五大子空间，其中 B 分为 B1/B2 两个层次。不再设独立的"交界空间"——B2（数值表示）本身就是天然的软硬件交界点，当使用标准格式时它是纯软件决策，当发明新格式时它自然驱动硬件侧变化。

## 软件端搜索空间

### A. 网络架构设计 — "网络长成什么样"

决定神经网络的结构本身，包括层的类型、宽度、深度、连接方式等。搜索结果是一个全新的网络拓扑。

| 子空间 | 搜索变量 | 取值示例 | 涉及论文 |
|--------|---------|---------|---------|
| **层类型选择** | 每层的算子类型 | {Conv3x3, Conv1x1, DepthwiseConv, MBConv, Attention, ...} | HW-NAS 类工作 |
| **通道宽度** | 每层的输出通道数 | {16, 32, 64, 128, 256, ...} | HW-NAS 类工作 |
| **网络深度** | 每个 stage 的层数 | {1, 2, 3, 4, ...} | HW-NAS 类工作 |
| **连接拓扑** | 跳跃连接、密集连接等 | {ResBlock, DenseBlock, ...} | HW-NAS 类工作 |
| **专家架构 (MoE)** | 每个专家的内部结构和能力定位 | EPE(轻量)/LAPE(中等)/FAPE(完整) | EMOS |
| **专家路由策略** | 场景到专家的分配规则 | 置信度阈值、距离判据、Top-K 选择 | EMOS, MoME, AW-MoE |
| **任务特化设计** | 面向硬件约束重新设计网络结构 | 2D cell 替代 3D 体素（LiFT） | LiFT |

**特点**：改变网络的"基因"，搜索代价最高（通常需要训练），但潜在收益也最大。

---

### B. 模型压缩与数值表示 — "如何压缩/表示已有模型"

在已有网络架构不变的前提下，通过裁剪模型结构或改变数据表示格式来压缩模型。B1 和 B2 的区分逻辑是：**B1 改变"模型有多少参数/连接"，B2 改变"每个参数用什么格式存储和计算"**。

#### B1. 压缩策略 — "裁剪模型结构"

减少模型的参数量或计算量，不涉及数据表示格式的改变。

| 子空间 | 搜索变量 | 取值示例 | 涉及论文 |
|--------|---------|---------|---------|
| **结构化剪枝** | 逐层/逐块通道剪枝率 | {0%, 10%, 20%, 30%, 50%} | 我们的方案, UPAQ |
| **剪枝准则** | 重要性评估指标 | {L1范数, Taylor展开, FPGM, ...} | 我们的方案 |
| **稀疏模式** | 结构化/非结构化稀疏的模式 | {N:M稀疏, block稀疏, 非结构化} | Phi, SOFA, TASDER |
| **蒸馏策略** | 教师/学生对、蒸馏损失设计 | 特征蒸馏 / logit蒸馏 / 视角引导蒸馏 | PointDistiller, QD-BEV, PlanKD, DiMA |
| **通信空间压缩 (V2X)** | 通信特征的空间降采样 | {none, 2x, 4x, roi} | QuantV2X, 我们的方案 |

**特点**：纯软件侧决策，不要求硬件配合改变计算方式。搜索结果是"保留哪些参数/连接"。

#### B2. 数值表示 — "改变数据格式"

改变模型参数和激活值的数值表示方式。**这是软硬件的天然交界点**：使用标准格式（INT8/FP16）时是纯软件决策；发明新格式（M-ANT、OliVe pair encoding）时则需要配套硬件支持，自然形成紧耦合。

| 子空间 | 搜索变量 | 取值示例 | 涉及论文 |
|--------|---------|---------|---------|
| **量化位宽** | 逐层权重/激活位宽 | {INT4, INT6, INT8, FP16} | AWQ, QD-BEV, Q-PETR, 我们的方案 |
| **量化粒度** | 量化参数共享范围 | {per-tensor, per-channel, per-group, per-block} | FGMP, 我们的方案 |
| **量化方案** | 对称性、校准方法 | {对称/非对称} x {MinMax/Entropy/Percentile} | 我们的方案 |
| **混合精度分配** | 不同层使用不同位宽的组合 | 敏感层FP16 + 非敏感层INT4 | FGMP, PTQAT |
| **通信特征精度 (V2X)** | 通信特征的数值精度 | {fp16, int8, int4} | QuantV2X, 我们的方案 |
| **自适应数值类型** | 数据类型的数学定义（新格式） | MANT 自适应浮点格式 | M-ANT |
| **离群值编码** | 离群值的特殊编码方式（新格式） | outlier-victim pair, microscaling | OliVe, MicroScopiQ |
| **bit-serial 表示** | 逐位串行处理的编码格式（新格式） | bit-slice GEMM | Panacea, BitMoD |
| **稀疏编码格式** | 稀疏数据的压缩存储表示 | {CSR, bitmap, block-sparse, N:M} | Megablocks, TASDER |

**B2 内部的分界线**：

```
B2 数值表示
|
+-- 使用标准格式（INT8/FP16/FP4 等已有硬件支持的格式）
|   -> 纯软件侧决策，搜索"每层用什么标准格式"
|   -> 不需要硬件配合，属于松耦合
|   -> 例：AWQ, QD-BEV, 我们的方案
|
+-- 发明新格式（自适应数值类型、特殊编码方案）
    -> 需要同时设计算法端的编码逻辑和硬件端的解码电路
    -> 天然形成紧耦合，B2 驱动 C 变化
    -> 例：M-ANT, OliVe, BitMoD, MicroScopiQ
```

## 硬件端搜索空间

### C. 硬件架构设计 — "芯片长成什么样"

设计专用加速器的物理架构参数。搜索结果是一个芯片的规格说明。**只有定制硬件（ASIC/FPGA/CIM）时才有此子空间。**

| 子空间 | 搜索变量 | 取值示例 | 涉及论文 |
|--------|---------|---------|---------|
| **计算阵列** | PE 数量、阵列形状、计算精度 | PE={64,128,256,384}, 形状={1D,2D} | CASCO, DOSA, Gemini |
| **存储层次** | 缓冲层级数、各级容量 | L1={256B-2KB}, L2={1KB-4KB}, DRAM | CASCO, DOSA |
| **片上网络 (NoC)** | PE 间互连拓扑与带宽 | {Bus, Mesh, Ring, Crossbar} | CASCO, INDM |
| **Chiplet 封装** | Die 数量、Die 间互连、封装方式 | {2.5D, 5.5D}, Die间带宽 | Gemini, Chiplet-Gym |
| **存内计算 (CIM)** | 宏单元结构、位精度、阵列配置 | SRAM-PIM 宏, 位级稀疏支持 | DB-PIM, CIMFlow |
| **FPGA 资源分配** | LUT/BRAM/DSP 分配比例 | DSP={100-500}, BRAM={50-200} | LiFT, hls4ml |
| **专用功能单元** | 特殊算子的硬件加速支持 | 稀疏跳零引擎、动态向量剪枝单元 | SPADE, Phi |

**特点**：搜索代价依赖评估方式——用分析模型（Timeloop）可以秒级评估，用 RTL 仿真则需要小时级。紧耦合工作集中于此。

### D. 部署配置与执行策略 — "如何使用已有硬件"

硬件架构已固定（如 GPU/Jetson），搜索的是如何配置和调度计算。搜索结果是一组部署参数。

| 子空间 | 搜索变量 | 取值示例 | 涉及论文 |
|--------|---------|---------|---------|
| **精度执行策略** | 逐层精度覆盖 | {follow_quant / force_fp16 / force_fp32} | 我们的方案 |
| **算子实现选择** | 同一算子的不同 kernel 实现 | {trt_native / flash_attn / custom_plugin} | 我们的方案 |
| **内存管理策略** | 显存池、缓存策略 | {isolated / shared_static / shared_streaming} | 我们的方案, EMOS(手动) |
| **执行调度** | 多流并行、流水线策略 | {single / branch_parallel / pipeline} | 我们的方案, EMOS(手动) |
| **图分割策略** | ONNX 图的切分方式 | {monolithic / split_head / split_agent} | 我们的方案 |
| **批处理策略** | 输入数据的打包方式 | {pad_to_max / dynamic / bucketed} | 我们的方案, Orca |
| **时序缓存** | 时序特征缓存深度和精度 | frames={0,1,2,4} x precision={same/fp16/int8} | 我们的方案 |
| **异构卸载** | 不同计算单元的任务分配 | DLA offload 层集合, CPU/GPU 分工 | 我们的方案(Orin), Splitwise |
| **存储放置** | 数据在 GPU/CPU/Disk 间的分配 | KV cache 的分页与放置策略 | FlexGen, vLLM |
| **服务调度** | 请求级别的调度策略 | 连续批处理, prefill/decode 分离 | Orca, DistServe, Splitwise |

**特点**：不改变硬件物理架构，但对实际性能影响巨大。大部分松耦合工作在此空间做人工优化，我们的方案的创新在于将此空间纳入自动化联合搜索。

### E. 编译与代码生成 — "如何将计算映射到硬件"

编译器决定如何将高层计算图转化为硬件可执行的低层指令。

| 子空间 | 搜索变量 | 取值示例 | 涉及论文 |
|--------|---------|---------|---------|
| **算子融合** | 哪些算子合并为一个 kernel | Conv+BN+ReLU 融合, 注意力融合 | Souffle, MCFuser, ClusterFusion |
| **层融合分组** | DNN DAG 的节点分组方案 | 组合优化：N 个节点的分区方式 | CASCO |
| **算子映射/调度** | 循环顺序、分块大小、并行映射 | Timeloop 搜索空间 | CASCO, DOSA |
| **Kernel Tactic** | 每层选用哪个具体 kernel 实现 | TRT 自动 profiling 择优 | TRT 自动(我们方案的外层) |
| **内存复用** | 激活 tensor 的生命周期与复用 | TRT 自动优化 | TRT 自动 |
| **稀疏编译** | 稀疏算子的融合与代码生成 | 跨表达式稀疏融合 | FuseFlow |
| **通信重叠** | 计算与数据搬运的重叠调度 | chunk 流水线, 预取策略 | DeepCompile, EMOS(手动) |
| **拓扑感知调度** | 对齐物理网络拓扑的通信编排 | 通信组与交换机层级对齐 | Arnold |

**特点**：编译器处于软硬件接口，天然具有双向感知能力。但大部分编译器工作是"给定硬件做最优编译"（单向感知），只有少数（如 CASCO 将层融合纳入联合搜索）做到了双向驱动。

## 关键洞察

### 1. B2 是紧耦合的"开关"

B2（数值表示）内部存在一条清晰的分界线：
- **使用标准格式**（INT8/FP16等） -> B2 是纯软件决策，不驱动硬件变化 -> 松耦合
- **发明新格式**（M-ANT/OliVe/BitMoD等） -> B2 同时要求硬件设计配套的编解码电路 -> 紧耦合

这解释了为什么 Agent-2（量化/剪枝硬件协同）方向的紧耦合工作最多——它们几乎都在 B2 里做了新格式设计。

### 2. 紧耦合的两条路径

从调研中归纳出两条通向紧耦合的路径：

```
路径一：B2(新格式) <-> C(新硬件)
  发明新的数值格式，同时设计配套硬件解码电路
  代表：OliVe, M-ANT, BitMoD, MicroScopiQ, Panacea

路径二：E(编译映射) <-> C(新硬件)
  将编译决策（层融合/算子映射）纳入硬件架构搜索的评估循环
  代表：CASCO, DOSA, Gemini
```

两条路径的共同点：**C（硬件架构）都参与了搜索**。

### 3. 我们方案填补的空白

```
现有格局:
  固定硬件(C不参与) + 标准格式(B2不驱动硬件) -> 几乎都是松耦合
  定制硬件(C参与)   / 新格式(B2驱动硬件)     -> 可以做紧耦合

我们的定位:
  固定硬件(C不参与) + 标准格式(B2不驱动硬件)
  -> 但通过将 D(部署配置) 纳入搜索空间
  -> 在 B1(剪枝)+B2(量化) 和 D(部署配置) 之间建立自动化反馈
  -> 实现固定硬件上的紧耦合

  这是第三条路径：B <-> D（不经过 C）
```

这就是为什么我们的方案在整个调研版图中是独特的：不需要设计新芯片（C 不参与），不需要发明新数值格式（B2 用标准格式），但仍然能做到紧耦合——因为我们发现了 D 子空间（部署配置）这个被大部分工作忽略的、具有丰富可搜索维度的硬件端空间。
