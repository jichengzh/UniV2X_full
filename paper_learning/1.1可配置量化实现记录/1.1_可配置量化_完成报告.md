## 一、工作目标与成果概述

打通 PyTorch 侧快速验证到 TRT 侧精确部署的完整量化链路，使量化配置可控、可搜索、可部署。

**核心成果**:
- 新增/修改 9 个文件，~2,490 行代码
- 打通 PyTorch 校准 -> quant_config.json -> ONNX Q/DQ 注入 -> TRT explicit INT8 engine 全链路
- 显式 INT8 engine: **38.2 MB**（vs 隐式 43 MB, FP16 75 MB）
- 确定最终量化搜索空间: **32~96 种**模块级组合（适合联合搜索）

## 二、PyTorch 端 vs TRT 端可配置维度对比

| 维度 | PyTorch 端 | TRT 隐式模式 | TRT 显式模式 (Q/DQ) | 是否纳入搜索 |
|------|:---------:|:-----------:|:------------------:|:----------:|
| **位宽** (D1/D2) | 2-8 bit 任意 | 仅 INT8/FP16 | INT8 容器模拟 4/6 bit (无速度增益) | **是** (INT8/FP16) |
| **粒度** (D3/D4) | per-tensor / per-channel / per-group | 不可控 | per-tensor / per-channel | **是** (tensor/channel) |
| **对称性** (D5) | 对称 + 非对称 | 仅对称 | 仅对称 (zp=0) | **否** (锁定对称) |
| **校准方法** (D6) | mse/minmax/entropy/percentile | 仅 entropy | Q/DQ 传递 PyTorch scale | **否** (锁定 minmax/mse) |
| **混合精度** (D7) | 逐层任意 | 全局统一 | 逐层 INT8/FP16 | **是** (模块级) |
| **量化目标** (D8) | W-only/A-only/W+A/none | 全局统一 | W+A 或 none | **是** (W+A/W-only/none) |
| **V2X 通信** (D9) | fp16/int8/int4 | 不在 TRT 图中 | 不在 TRT 图中 | **否** (锁定 INT8) |
| **模块选择** (D10) | 全模块可选 | 逐引擎独立 | 逐引擎独立 | **是** (核心维度) |

**TRT 的刚性约束**: 对称量化 (GPU 硬件限制), 无原生 per-group, 无原生 INT4 kernel。
**Q/DQ 模式可绕过的限制**: 校准方法、逐层混合精度。
**不可绕过的限制**: 非对称量化、per-group 的 ONNX 表达。

## 三、模块级量化搜索空间

### 3.1 UniV2X 模块与量化路径

| 模块 | 当前部署方式 | 延迟占比 | 量化路径 | 搜索价值 |
|------|:-----------:|:-------:|---------|:-------:|
| ResNet-FPN backbone | PyTorch FP32 | **33%** (~30ms) | PyTorch INT8 / TRT (需去 DCNv2) | **高** |
| BEV encoder (ego=infra 共享) | TRT FP16/INT8 | 30% (~20-40ms) | TRT 隐式/显式 INT8 | **高** |
| 检测头 | TRT FP16/INT8 | ~15% | TRT 隐式 INT8 | **中** |
| 下游头 (ego=infra 共享) | TRT FP16/INT8 | ~10% | TRT 隐式 INT8 | **低** (已验证不影响 AMOTA) |
| V2X 通信 | PyTorch | ~5% | CommQuantizer | **否** (锁定 INT8) |

### 3.2 搜索空间定义

```
锁定在搜索空间之外:
  D5  对称性:    symmetric=True     (TRT GPU 硬约束)
  D6  校准方法:  minmax(搜索)/mse(部署)  (与其他维度无耦合)
  D9  V2X通信:   INT8               (已验证可行, 无耦合)
  D10 下游头:    INT8               (已验证不影响 AMOTA)
  逐层配置:      模块级统一          (逐层收益微乎其微, 见反思 5.2)

保留在搜索空间:

  Backbone (需先做可行性验证):
    部署方式: {PyTorch_FP32, TRT_FP16, TRT_INT8}  = 3 种
    (TRT 路径需替换 DCNv2 为标准 Conv)

  BEV encoder (ego=infra 共享):
    精度: {FP16, INT8}                = 2
    粒度: {per-tensor, per-channel}   = 2
    目标: {W+A, W-only}              = 2
    小计: 2 x 2 x 2                  = 8 种

  检测头:
    精度: {FP16, INT8}                = 2
    粒度: {per-tensor, per-channel}   = 2
    小计: 2 x 2                       = 4 种
```

### 3.3 搜索空间总规模

**TRT 端 (实际可部署)**:

| 场景 | Backbone | BEV | 检测头 | 总量化组合 |
|------|:--------:|:---:|:-----:|:---------:|
| 保守 (backbone 不可替换) | 1 (锁定 FP32) | 8 | 4 | **32 种** |
| 激进 (backbone 可替换) | 3 | 8 | 4 | **96 种** |

**PyTorch 端 (快速验证, 配置自由度更高)**:

各模块统一按 {位宽} x {粒度} x {量化目标} 三个维度展开。选择 FP16/FP32 意味着"不量化", 此时粒度和目标维度不存在, 所以是 `1(不量化) + 量化选项的乘积`, 不是全部相乘:

| 模块 | 不量化 | 量化位宽 | 量化粒度 | 量化目标 | 组合数 | 说明 |
|------|:------:|---------|---------|---------|:------:|------|
| Backbone | 1 (FP32) | {INT8, INT6, INT4} | {tensor, channel, group} | {W+A, W-only, A-only} | 1 + 3x3x3 = **28** | 含 DCNv2 也可 fake-quant |
| BEV encoder | 1 (FP16) | {INT8, INT6, INT4} | {tensor, channel, group} | {W+A, W-only, A-only} | 1 + 3x3x3 = **28** | 同上 |

> `QuantModule.set_quant_state(weight_quant=False, act_quant=True)` 直接支持。
> PyTorch 端保留完整配置自由度, TRT 端根据可行性裁剪。

| 场景 | Backbone | BEV | 检测头 | PyTorch 总组合 |
|------|:--------:|:---:|:-----:|:-------------:|

### 3.4 剪枝搜索空间预估 (B1)

基于 UniV2X 模型结构分析 (详见 下一步工作全局规划.md):

| 优先级 | 剪枝对象 | 参数量占比 | 搜索取值 | 组合数 |
|:------:|---------|:---------:|---------|:------:|
| **P0** | FFN 中间维度 (1024->?) | ~40% Transformer | {512, 640, 768, 896, 1024} | 5 |
| **P1** | 检测头中间维度 (256->?) | ~5% | {128, 192, 256} | 3 |
| P2 | Attention 维度 (256->?) | ~15% Encoder | {128, 192, 256} | 3 |
| P3 | Backbone 通道 | ~50% total | 取决于 backbone 是否纳入搜索 | - |

**推荐剪枝方法**: DepGraph (Fang et al., CVPR 2023) — 自动分析 Transformer+CNN 混合架构的层间依赖。

## 四、有意义的敏感度分析结果

### 4.1 BEV encoder 逐层敏感度 (GPU 实测, 54 层)

| 分类 | 数量 | 含义 |
|:----:|:----:|------|
| safe_int8 | 45 (83%) | cosine >= 0.9999, 全部可安全量化 |
| search | 9 (17%) | 0.999 < cosine < 0.9999, 量化有轻微影响 |
| skip_fp16 | 0 (0%) | 无层需要保持 FP16 |

**核心结论**: BEV encoder 整体对 INT8 非常鲁棒，可以模块级统一 INT8。

### 4.2 量化误差交互项 (GPU 实测)

| 层对 | 联合误差 | 各层误差之和 | 交互项 |
|------|:--------:|:----------:|:------:|
| (layer5.ffn, layer5.attn) | 0.000 | 0.001121 | -0.001121 |
| (layer5.ffn, layer5.attn.proj) | 0.000 | 0.001016 | -0.001016 |
| (layer5.attn, layer5.attn.proj) | 0.000 | 0.000717 | -0.000717 |

**核心结论**: 交互项全部为负，量化误差层间**相互抵消**，不存在非线性恶化。联合搜索中的线性加性假设是安全的。

## 五、反思与关键经验

### 5.1 架构决策: PyTorch 搜索 + TRT 部署

PyTorch 负责"搜索量化决策"，TRT 负责"执行量化决策"。通过 Q/DQ 显式模式，PyTorch 的 scale 可以精确传递到 TRT。这是整个链路打通的关键认知。

### 5.2 逐层配置的投入产出比极低

Phase A4 数据表明：54 层全 INT8 → AMOTA 0.364（vs FP16 0.381）。9 个 search 层即使全部 FP16 也最多改善 ~0.006 AMOTA，但每种配置需要 ~20min 评估。**正确的决策粒度是模块级，而非逐层。**

### 5.3 已排除方案清单

| 方案 | 排除原因 | AMOTA 影响 |
|------|---------|:----------:|
| 非对称量化 | TRT GPU 硬件不支持 zp!=0 | 构建失败 |
| 权重-only Q/DQ (无激活) | TRT 进入 explicit 模式, Calibrator 被忽略 | 0.137 (崩溃) |
| AdaRound 内嵌 ONNX (无 Q/DQ) | TRT 重新推导 scale, 双重量化 | 0.190 (崩溃) |
| PLUGIN_V2 显式精度设置 | 插入额外 dequant/quant 节点, 级联误差 | 0.278 |
| sampling_offsets/attention_weights 量化 | 坐标层量化后 BEV 特征失效 | BEV 全错 |
| BEV encoder 逐层配置搜索 | 投入产出比极低, 最多改善 0.006 AMOTA | 微乎其微 |

### 5.4 为什么失败方案效果差

1. **非对称量化**: GPU Tensor Core 不支持偏移加法，是硬件计算单元的设计决定，非软件限制。
2. **权重-only Q/DQ**: TRT 的 explicit/implicit 模式是全局二元开关——看到任何 Q/DQ 就接管全部量化控制。
3. **AdaRound 内嵌**: `round(round(W/s1)*s1/s2)*s2` 的两次 round 复合误差远大于单次——"优化了但传不过去"比"不优化"更差。
4. **坐标层量化**: 坐标和注意力权重是信息流的"路由层"，决定信息从哪里来；路由层误差被下游放大，不像内容层误差可被残差连接吸收。

### 5.5 Backbone 应纳入搜索空间

ResNet-FPN backbone 占优化后管线延迟的 33%（~30ms / ~90ms），是唯一未被 TRT 加速的模块。路径: 替换 DCNv2 为标准 Conv 后导出 TRT，或用 PyTorch native INT8。需先做可行性验证（~0.5 天）。

## 附录 A: 评估指标说明

**err (Mean Absolute Error, 均值绝对误差)**:
```
err = mean(|x_original - x_quantized|)
```
衡量量化重建精度，用于 Phase A1/A2 单元测试（随机张量）。

**cos (Cosine Similarity, 余弦相似度)**:
```
cos = cosine_similarity(BEV_fp32.flatten(), BEV_quantized.flatten())
```
衡量全模型量化对 BEV 特征方向的影响，用于 Phase A4 敏感度分析（真实模型+校准数据）。
分类: >= 0.9999 safe_int8, 0.999~0.9999 search, < 0.999 skip_fp16。

## 附录 B: 9 个可配置维度的实现细节

### D1/D2 位宽

PyTorch: `UniformAffineQuantizer(n_bits=)` + `bitwidth_refactor()`，2-8 bit。
TRT: Q/DQ scale 范围控制 (INT8=max/127, INT6=max/31, INT4=max/7)，物理走 INT8 kernel。
测试: 4bit qmin=-7/qmax=7, 6bit qmin=-31/qmax=31, 8bit qmin=-127/qmax=127 全部正确。

### D3 权重粒度

PyTorch: per-tensor (`channel_wise=False`), per-channel (`channel_wise=True`), per-group (`group_size=N`)。
TRT: Q/DQ scale 维度控制 per-tensor/per-channel，per-group 不原生支持。

精度排序 (INT8, 随机权重 1024x256):
```
per-group(32): err=0.004472 < per-group(64): 0.005024 < per-channel: 0.005956 < per-tensor: 0.008172
```

### D4 激活粒度

PyTorch: per-tensor / per-channel (entropy+channel-wise 已修复，但性能慢 ~6min/32通道)。
TRT: per-tensor / per-channel 通过 Q/DQ 支持。

### D5 对称性

锁定 symmetric=True (TRT GPU 硬约束)。对称比非对称误差大 ~25%。

### D6 校准方法

锁定在搜索空间外 (与其他维度无耦合)。
精度: mse(0.005384) < entropy(0.005790) < minmax(0.007356)。
策略: 搜索阶段用 minmax (最快), 部署用 mse (最精确)。

### D8 量化目标

W+A: 权重+激活都量化 (标准, INT8 kernel)。
W-only: 仅权重量化, 激活 FP16 (TRT 需透明激活 Q/DQ)。
none: 不量化, FP16。

### D9 V2X 通信

锁定 INT8 (CommQuantizer, err=0.008789, 可接受)。不在 TRT 图内, Python 侧处理。

## 附录 D: Bug 修复记录

1. **CommQuantizer n_bits>=16**: `UniformAffineQuantizer` 要求 2<=n_bits<=8, FP16 passthrough 时不应创建 quantizer。
2. **相对导入兼容**: 添加 try/except fallback 支持单元测试环境。
3. **sensitivity_report 覆盖**: 保存时合并已有报告而非覆盖。
4. **a_scale list 类型**: JSON 读取为 list, 需 isinstance 检查后转 np.array。
5. **weight scale 维度不匹配**: PyTorch per-channel scale 维度与 ONNX 权重形状不一致, 从 ONNX 权重重新计算。
