# 1.1 可配置量化 — 实验反思总结

> 更新日期：2026-04-19
> 来源：Phase A1 ~ Phase B2 全部实验中积累的反思、认知和教训

---

## 一、架构决策："PyTorch 搜索 + TRT 部署"的双端分工

### 1.1 核心认知

PyTorch 负责"搜索量化决策"，TRT 负责"执行量化决策"。通过 Q/DQ 显式模式，PyTorch 的 scale 可以精确传递到 TRT。

这是整个链路打通的**关键认知**。之前的失败方案（AdaRound 内嵌、PLUGIN_V2、权重-only Q/DQ）都是因为没有正确理解这个分工导致的。

### 1.2 两端的配置自由度差异

| 维度 | PyTorch 端 | TRT 端 | 差异来源 |
|---|---|---|---|
| 位宽 | 2-8 bit | 仅 INT8 | GPU Tensor Core 硬件 |
| 粒度 | per-tensor/channel/group | per-tensor/channel | ONNX Q/DQ 规范 |
| 对称性 | 对称+非对称 | 仅对称 | GPU 不支持 zp 偏移 |
| 校准 | mse/minmax/entropy/percentile | 不可控 | Q/DQ 传递 scale |

**教训**：PyTorch 端保留更大自由度用于快速筛选，TRT 端根据硬件约束裁剪。先在 343 种 PyTorch 配置中快筛，再决定 TRT 端搜索空间是 32 还是 96。

## 二、逐层配置的投入产出比极低

### 2.1 数据支撑

Phase A4 数据：54 层全 INT8 → AMOTA 0.364。9 个 search 层即使全部 FP16 也最多改善 ~0.006 AMOTA，但每种配置需要 ~20 min 评估。

### 2.2 结论

**正确的决策粒度是模块级（backbone / BEV encoder / 检测头 / 下游头），而非逐层。**

逐层配置搜索空间 2^54 ≈ 10^16，即使用贝叶斯优化也不可行。模块级搜索空间 32-96 种，完全可穷举。

## 三、失败方案的根因分析

### 3.1 非对称量化 (AMOTA: 构建失败)

**表面原因**：TRT 报错 "non-zero zero_point"。

**根因**：GPU Tensor Core 不支持偏移加法，这是硬件计算单元的设计决定，非软件限制。

**教训**：**PyTorch 侧敏感度分析必须用 symmetric=True**，否则结果可能低估 TRT 部署时的实际损失（对称比非对称误差大 25%）。

### 3.2 权重-only Q/DQ (AMOTA: 0.137, 崩溃)

**表面原因**：TRT 推理时大量层退化到 FP16，精度反而更差。

**根因**：TRT 的 explicit/implicit 模式是**全局二元开关**——看到任何 Q/DQ 就接管全部量化控制，Calibrator 被忽略。没有激活 Q/DQ 的层不会自动用 Calibrator 校准。

**教训**：TRT 显式模式下，**每个需要量化的层必须同时有权重和激活的 Q/DQ**。W-only 场景需要插入"透明"激活 Q/DQ（large scale passthrough）。

### 3.3 AdaRound 内嵌 ONNX (AMOTA: 0.190, 崩溃)

**表面原因**：精度严重退化。

**根因**：`round(round(W/s1)*s1/s2)*s2` — 两次 round 的复合误差远大于单次。AdaRound 优化了权重 rounding，但其 scale 传不到 TRT，TRT 用自己的 scale 重新量化 → "优化了但传不过去"比"不优化"更差。

**教训**：端到端链路中，优化的结果**必须能原封不动传递到部署端**。中间有任何"翻译"步骤都可能引入误差。

### 3.4 坐标层量化 (BEV 全错)

**表面原因**：BEV 特征完全失效。

**根因**：`sampling_offsets` 和 `attention_weights` 是信息流的"路由层"——决定从 BEV 的哪个位置采样。路由层的量化误差被下游**放大**，不像内容层误差可被残差连接**吸收**。

**教训**：**坐标敏感层（sampling_offsets, attention_weights）必须保持 FP16**。这是 MSDA 架构的硬约束，不可通过更精细的校准方法绕过。

## 四、精度代理指标的局限

### 4.1 BEV cosine similarity 的适用范围

- **适合**：逐层敏感度分类（safe/search/skip），因为不同层的 cosine 差异足够大
- **不适合**：全模型配置对比（calibration/granularity/symmetry/bitwidth），因为 45 个不敏感层将 cosine 稀释到 1.000000

### 4.2 改进方向

1. 对比分析应只测量 search 层的输出 cosine
2. 或用完整 AMOTA（代价是慢 60 倍）
3. 或用 BEV feature 的 MSE 而非 cosine（MSE 对微小差异更敏感）

**教训**：代理指标的分辨力取决于测量对象的粒度。全模型聚合 → 分辨力不足；逐层单独 → 分辨力充足。

## 五、量化误差的线性加性

### 5.1 发现

所有测试的层对交互项均为负（-0.0007 ~ -0.0011），说明联合量化误差**小于**各层误差之和。

### 5.2 机制推测

Transformer 的残差连接和 LayerNorm 对误差有"吸收"和"归一化"效果。多层量化误差通过这些结构被部分抵消。

### 5.3 实际意义

- Level 1 的廉价评估（逐层 cosine 之和）是**安全的保守估计**
- 联合搜索中不需要担心"非线性耦合恶化"
- 可以用贪心策略（逐层决策）替代全局搜索，大幅缩减搜索成本

## 六、entropy + channel-wise 的性能瓶颈

### 6.1 问题

32 通道的 entropy 校准耗时约 6-8 分钟。BEV encoder 有 ~30 层量化目标，如果全部用 entropy+channel-wise，单次校准可能需要 **3+ 小时**。

### 6.2 解决策略

- 搜索阶段用 **minmax**（毫秒级）做快速筛选
- 仅对 **Top-K 候选**用 entropy/mse 做精细校准
- 部署阶段用 **mse**（最精确）做最终校准

**教训**：校准方法的选择不仅看精度，还要看是否能嵌入搜索循环。最优 ≠ 最实用。

---

## 七、对称量化的 25% 精度代价

### 7.1 数据

对称 err=0.007356 vs 非对称 err=0.005885，差距约 25%。

### 7.2 影响

这意味着在 PyTorch 端用**非对称**校准做敏感度分析的结果，可能**低估**某些层在 TRT 对称量化下的实际损失。

### 7.3 对策

Phase A4 敏感度分析**必须用 symmetric=True**。这是本次实验的一个关键方法论发现。

## 八、Backbone 应纳入搜索空间

### 8.1 现状

ResNet-FPN backbone 占优化后管线延迟的 **33%**（~30ms / ~90ms），是唯一未被 TRT 加速的模块。

### 8.2 路径

替换 DCNv2 为标准 Conv → 导出 TRT → INT8 量化。需先做可行性验证（AMOTA 损失 < 0.02）。

### 8.3 与 1.2 剪枝实验的关联

1.2 的 DCN 零重训替换实验已证明：零重训 AMOTA -68%（失败），但李星峰已证明加 3-6 epoch 微调可恢复。因此 backbone 的 TRT 路径是可行的。

## 九、Q/DQ 注入的工程经验

### 9.1 激活 Q/DQ 去重

多个 MatMul 共享同一激活输入时（如 attention 的 Q/K/V），只插入一对 Q/DQ。通过 `_act_qdq_inserted` dict 跟踪，避免重复插入。

### 9.2 拓扑排序维护

Q/DQ 节点必须正确插入 ONNX 图的拓扑序中：
- 权重 Q/DQ 紧随源节点
- 激活 Q/DQ 在消费 MatMul 之前

违反拓扑序会导致 TRT 构建时的"节点未定义"错误。

### 9.3 无 PyTorch 依赖

`inject_qdq_from_config.py` 是纯 ONNX + numpy 操作，不需要加载模型或校准数据。所有信息来自 quant_config.json。这使得 Part B 可以独立于 PyTorch 环境运行。

## 十、方法论层面的总结

### 10.1 量化配置的正确抽象层次

9 个维度 (D1-D9) 中：
- 4 个锁定（D5 对称、D6 校准、D9 通信、D10 下游头） → 不需要搜索
- 5 个保留（D1/D2 位宽、D3/D4 粒度、D7 混合精度、D8 量化目标） → 但应以**模块级**而非逐层级搜索

### 10.2 搜索策略

```
第一阶: PyTorch 快筛 (1-2天)
  343 种量化配置快速评估, 确认 TRT 搜索空间是 32 还是 96

第二阶: TRT 端联合搜索 (2-3天)
  3,840 种 → Level 1 过滤 70% → ~1,100 种
  Level 2 (~20min/次): 贝叶斯优化采样 ~80-120 次

backbone 可行性验证 (优先, ~0.5天):
  去 DCNv2 后 AMOTA 损失 < 0.02 → 纳入搜索 (96种), 否则锁定 FP32 (32种)
```

### 10.3 三句话总结

1. **PyTorch 搜索 + TRT 部署的双端架构是正确的**：Q/DQ 显式模式是打通两端的关键
2. **模块级统一量化 >> 逐层配置**：投入产出比差 100 倍
3. **坐标敏感层（sampling_offsets, attention_weights）是量化的绝对禁区**：路由层误差被放大，不可通过校准绕过
