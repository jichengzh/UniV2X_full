## 一、B2 空间（数值表示/量化）：已有基础与待完善配置

### 1.1 已实现的配置

| 模块 | 方案 | 位宽 | 校准方法 | AMOTA | 状态 |
|------|------|:----:|---------|:-----:|:----:|
| BEV encoder | Vanilla PTQ | W8A8 | Entropy, 50帧temporal | 0.364 | 可用 |
| 检测头 | Vanilla PTQ | W8A8 | Entropy | 有引擎 | 可用 |
| 下游头 | Vanilla PTQ | W8A8 | Entropy, 50帧 | 0.255 | 可用 |
| BEV encoder | AdaRound | W8 | AdaRound 10帧 | 0.190 | 废弃 |
| BEV encoder | AdaRound+Q/DQ | W8(Q/DQ) | - | 0.137 | 废弃 |

### 1.2 需要进一步完善的配置

#### (a) 校准方法对比（低成本，优先做）

已有 Entropy 校准，但未对比其他方法。实施成本低（只需改 Calibrator 配置）：

| 校准方法 | TRT 支持 | 实现方式 | 预期 |
|---------|:--------:|---------|------|
| **MinMax** | IInt8MinMaxCalibrator | 修改 `build_trt_int8_univ2x.py` 的 Calibrator 类 | 基线对比 |
| **Entropy**（已有） | IInt8EntropyCalibrator2 | 当前实现 | AMOTA=0.364 |
| **Percentile** | 需自定义 | 在 calibrator 中实现 p99.99 截断 | 可能优于 entropy |

**工作量**：~0.5 天。只需替换 Calibrator 类并重新构建引擎。

#### (b) 混合精度逐层搜索（中等成本，关键）

当前所有层统一 W8A8，但已知 `sampling_offsets` 和 `attention_weights` 必须 FP16。需要系统化地确定每层的最优精度：

**实施方案**：逐层敏感度分析（对应 pipeline Phase 0）

```
对 BEV encoder 的每一层 i:
  1. 仅将该层设为 INT8，其余 FP16
  2. 构建 TRT engine，测量 AMOTA 降幅 delta_i
  3. 按 delta_i 排序，划分为三档：
     - delta_i < 0.005: 可安全量化（INT8）
     - 0.005 < delta_i < 0.02: 中等敏感（搜索空间）
     - delta_i > 0.02: 必须 FP16（锁定）
```

**工作量**：BEV encoder 约 20 层 x 20min/层 = ~7 小时（可自动化过夜跑）。
**参考**：Q-PETR 的逐层敏感度分析方法，QD-BEV 的选择性量化策略。

#### (c) INT4 可行性探测（探索性）

TRT 10.x 对 INT4 的支持有限（主要通过 FP8/INT4 混合），需要先验证：

```
测试步骤：
  1. 检查 TRT 10.13 是否支持 INT4 kernel（trtexec --int4 标志）
  2. 对低敏感层尝试 INT4 量化，观察是否回退 FP32
  3. 如果支持，测量延迟增益和精度损失
```

**工作量**：~0.5 天探测。如果 TRT 不原生支持 INT4，则该维度从搜索空间中移除。

#### (d) 完整的权重+激活 Q/DQ 方案（高成本，研究性）

AdaRound 失败的根因是"权重-only Q/DQ"。正确方案需要同时插入权重和激活的 Q/DQ：

```
正确的 AdaRound for TRT 路径：
  1. 修改 UniformAffineQuantizer: sym=True, INT8
  2. AdaRound 在对称 INT8 网格上优化
  3. ONNX 导出时同时插入：
     - 权重 Q/DQ: scale = scale_adaround_sym
     - 激活 Q/DQ: scale = calibration entropy scale
  4. TRT explicit 模式正确 fuse 为 INT8 kernel
```

**工作量**：~3-5 天。需要重写量化器 + 激活 Q/DQ 插入 + 全量验证。
**判断**：仅在 Vanilla PTQ 精度不够时才做，当前 0.364 vs FP16 0.370 差距仅 0.006，优先级低。

## 二、B1 空间（剪枝）：从零开始的实施规划

### 2.1 UniV2X 模型结构与剪枝对象分析

```
UniV2X 完整结构:

  ResNet-101 + FPN (backbone, ~45M params)
       |
       v
  BEVFormerEncoder (BEV编码器, ~12M params)
    |- SpatialCrossAttention x 6层 (SCA)
    |    |- sampling_offsets (Linear 256->48)    <-- 不可剪枝（坐标敏感）
    |    |- attention_weights (Linear 256->48)   <-- 不可剪枝（坐标敏感）
    |    |- value_proj (Linear 256->256)         <-- 可剪枝目标
    |    |- output_proj (Linear 256->256)        <-- 可剪枝目标
    |
    |- TemporalSelfAttention x 6层 (TSA)
    |    |- 同上结构
    |
    |- FFN x 6层
         |- fc1 (Linear 256->1024)              <-- 主要剪枝目标（中间表示）
         |- fc2 (Linear 1024->256)              <-- 随 fc1 联动
       |
       v
  DetectionTransformerDecoder (检测头, ~8M params)
    |- 6层 Decoder Layer
    |    |- Self-Attention
    |    |- Cross-Attention (MSDeformableAttention)
    |    |- FFN (256->1024->256)                <-- 可剪枝目标
    |
    |- cls_branches (Linear 256->10, x7层)
    |- reg_branches (Linear 256->256->10, x7层)  <-- 中间表示可剪枝
       |
       v
  下游头 (Motion + Occ + Planning, ~30M params)
    |- MotionTransformerDecoder (6层)
    |- OccHead (CVT_Decoder)
    |- PlanningHead
```

### 2.2 剪枝对象优先级

| 优先级 | 剪枝对象 | 类型 | 参数量占比 | 剪枝方式 | 预期收益 |
|:------:|---------|------|:---------:|---------|---------|
| **P0** | **FFN 中间表示维度** (1024->?) | 宽度剪枝 | ~40% of Transformer | `fc1` 输出通道 + `fc2` 输入通道联动裁减 | 延迟和参数量同时大幅下降 |
| **P1** | **检测头 reg_branches 中间维度** (256->?) | 宽度剪枝 | ~5% | 中间 Linear 输出通道裁减 | 检测头加速 |
| **P2** | **value_proj / output_proj** (256->?) | 通道剪枝 | ~15% of Encoder | 注意力维度裁减 | BEV 编码器加速 |
| P3 | backbone ResNet 通道 | 通道剪枝 | ~50% of total | 逐层通道剪枝 | 参数量大幅下降，但 backbone 不在 TRT 中 |
| P4 | Decoder 层数 (6->?) | 深度剪枝 | ~8M × (removed/6) | 直接移除层 | 延迟线性下降，但精度影响大 |

**为什么 P0 最优先**：FFN 的中间维度 1024 是 BEV 编码器和检测头中最"肥"的部分，占 Transformer 参数量的 ~2/3（每层 FFN: 256x1024 + 1024x256 = 524K，而 Attention 部分约 256K）。将 1024 剪到 512 或 768 可以直接减少 25%-50% 的 Transformer 参数量和计算量。

**为什么 backbone 排在 P3**：ResNet-FPN backbone 占总参数 ~50%，但当前它不在 TRT 引擎中（DCNv2 不可导出），剪枝它不能直接改善 TRT 延迟。除非未来解决 DCNv2 导出问题。

### 2.3 剪枝方法选择

针对 CNN+Transformer 混合架构，参考以下方法：

#### (a) 结构化通道剪枝（适用于 P0/P1/P2）

**参考方法**：

| 方法 | 论文/来源 | 核心思想 | 适用场景 |
|------|---------|---------|---------|
| **L1-Norm** | Li et al., ICLR 2017 | 按 L1 范数排序，移除最小通道 | 简单基线 |
| **Taylor Expansion** | Molchanov et al., CVPR 2019 | 按梯度 x 激活的 Taylor 展开评估重要性 | 精度更优，需要前向+反向 |
| **FPGM** | He et al., CVPR 2019 | 移除几何中心附近的冗余滤波器 | 无需微调即可保持精度 |
| **DepGraph** | Fang et al., CVPR 2023 | 自动分析层间依赖，处理残差/跳跃连接 | **Transformer + CNN 混合架构推荐** |

**推荐首选 DepGraph**：UniV2X 的 BEVFormer 有大量残差连接和注意力跳跃连接，手动分析层间依赖容易出错。DepGraph 自动构建依赖图，确保剪枝后的模型结构一致性。

**实施方案**：

```python
# Step 1: 安装 torch-pruning（DepGraph 的实现库）
pip install torch-pruning

# Step 2: 构建剪枝器
import torch_pruning as tp

model = load_univ2x_model()
example_inputs = get_example_inputs()

# 指定不可剪枝的层
ignored_layers = [
    model.pts_bbox_head.transformer.encoder.layers[i].attentions[0].sampling_offsets
    for i in range(6)
] + [
    model.pts_bbox_head.transformer.encoder.layers[i].attentions[0].attention_weights
    for i in range(6)
]

# 构建剪枝器（L1 或 Taylor）
pruner = tp.pruner.MetaPruner(
    model, example_inputs,
    importance=tp.importance.MagnitudeImportance(p=1),  # L1
    pruning_ratio=0.3,  # 全局剪枝率
    ignored_layers=ignored_layers,
)

# Step 3: 执行剪枝
pruner.step()

# Step 4: 验证剪枝后的模型
pruned_model = pruner.model
# 检查通道对齐（INT8 要求 ch%8==0）
for name, module in pruned_model.named_modules():
    if isinstance(module, nn.Linear):
        assert module.out_features % 8 == 0, f"{name}: {module.out_features} not aligned"
```

**工作量**：~2-3 天完成基础剪枝框架 + 验证。

#### (b) FFN 中间维度专项剪枝（P0 快速实现）

FFN 中间维度不需要复杂的剪枝算法，可以直接"瘦身"：

```python
# 将 BEVFormerEncoder 和 DetectionTransformerDecoder 中所有 FFN 的
# 中间维度从 1024 统一降为 target_dim

target_dims = [512, 640, 768, 896]  # 搜索空间

for target_dim in target_dims:
    model = load_univ2x_model()
    for layer in model.encoder.layers:
        # FFN: fc1(256->1024) -> fc2(1024->256)
        # 剪枝: fc1(256->target_dim) -> fc2(target_dim->256)
        prune_ffn_dim(layer.ffns[0], target_dim)
    
    # 知识蒸馏微调（几个 epoch）
    distill(teacher=original_model, student=model, epochs=5)
    
    # 评估
    amota = evaluate(model)
```

**工作量**：~1 天。代码简单，主要时间在微调。

#### (c) 注意力头剪枝（P2，探索性）

| 方法 | 思路 | 参考 |
|------|------|------|
| Head Importance | 按注意力头的梯度贡献排序，移除最不重要的头 | Michel et al., NeurIPS 2019 |
| 统一维度缩减 | 将 attention dim 从 256 统一降到 192/128 | 简单但有效 |

**判断**：注意力头剪枝的收益不如 FFN 剪枝大（注意力占 Transformer 参数的 ~1/3），且 MSDeformableAttention 的特殊结构使头剪枝实现更复杂。建议 FFN 剪枝验证有效后再考虑。

### 2.4 剪枝后的关键约束

```
剪枝后通道数必须满足:
  1. ch % 8 == 0      (INT8 TensorRT 对齐要求)
  2. ch % num_heads == 0  (多头注意力整除要求)
  3. 残差连接两端通道数一致 (DepGraph 自动处理)

剪枝后必须执行:
  1. 知识蒸馏微调 (5-10 epochs，恢复精度)
  2. 重新 ONNX 导出 (通道数变了，图结构变了)
  3. 重新 TRT 构建 (新的 shape -> 新的 kernel tactic)
  4. 重新量化校准 (权重分布变了)
```

### 3.2 需要实现的 D 空间配置

#### (a) 多流并行执行（参考 EMOS 的系统优化）

```
当前: ego BEV -> infra BEV -> 融合 -> 检测头 -> 下游头 (串行)

优化: 
  Stream 1: ego backbone + ego BEV encoder
  Stream 2: infra backbone + infra BEV encoder  (与 Stream 1 并行)
  Sync barrier
  Stream 1: 融合 + 检测头 + 下游头
```

**参考论文**：EMOS（TPAMI 2026）的"parallel communication and computation"策略。

**实现方式**：
```python
# TRT execution context 支持多 stream
stream_ego = cuda.Stream()
stream_infra = cuda.Stream()

# 并行执行 BEV encoder
with stream_ego:
    ego_bev = ego_bev_engine.execute(ego_inputs)
with stream_infra:
    infra_bev = infra_bev_engine.execute(infra_inputs)

# 同步
stream_ego.synchronize()
stream_infra.synchronize()

# 后续串行
result = detection_engine.execute(ego_bev, infra_bev)
```

**工作量**：~1-2 天。
**预期收益**：BEV encoder 阶段从串行 ~80ms 降至并行 ~45ms（受限于较慢的那个）。

#### (b) 时序缓存精度降级

```
当前: prev_bev 以 FP16 全精度缓存
优化: prev_bev 缓存为 INT8，使用时反量化到 FP16

实现:
  1. 在 BEV encoder 输出后添加量化步骤: bev_int8 = quantize(bev_fp16)
  2. 下一帧使用时反量化: prev_bev = dequantize(bev_int8)
  3. 显存节省: 200x200x256 x 2bytes -> 200x200x256 x 1byte = 50% 缓存内存节省
```

**工作量**：~0.5 天。
**预期收益**：对多 agent 场景（缓存 2-4 帧 x 多 agent）的显存节省显著。

#### (c) 动态 shape optimization profile

当前 TRT engine 固定 shape。对于 V2X 场景中 agent 数量变化（1-4），可构建多 profile engine：

```python
profile = builder.create_optimization_profile()
# BEV encoder 的 query 数量随 agent 数变化
profile.set_shape("queries", min=(1, 200, 256), opt=(2, 400, 256), max=(4, 800, 256))
config.add_optimization_profile(profile)
```

**工作量**：~1 天。
**参考**：TensorRT Developer Guide 中的 Dynamic Shapes 章节。

## 四、联合搜索实现：全流程规划

### 4.1 搜索空间构建

基于 B1/B2/D 三个子空间，按 pipeline 文档的内外循环结构构建：

```
外循环搜索空间（结构性决策，~15-30 种配置）:
  B1:
    ffn_mid_dim:     {512, 640, 768, 896, 1024}    # FFN 中间维度
    head_mid_dim:    {128, 192, 256}                # 检测头中间维度
  D (拓扑性):
    export_strategy:  {monolithic, split_head}
    multi_stream:     {single, branch_parallel}
    memory_pool:      {isolated, shared_static}

内循环搜索空间（非结构性配置，贝叶斯优化）:
  B2:
    quant_bits[layer_group_i]:  {FP16, INT8}        # 按敏感度分组（~5-8 组）
    calibration_method:         {entropy, minmax, percentile}
  D (非拓扑性):
    layer_precision_override[sensitive_layer_i]: {follow, fp16}
    temporal_cache_precision:    {fp16, int8}
    temporal_cache_frames:      {1, 2}
    num_agents_opt:             {1, 2, 4}

维度间约束:
    ffn_mid_dim % 8 == 0                  # INT8 对齐
    ffn_mid_dim % num_heads == 0          # 注意力整除
    shared_static 排斥 branch_parallel    # 内存冲突
    INT8 层的上下游不能全是 FP16          # 避免过多 reformatter
```

### 4.2 搜索方法

采用两级评估漏斗 + 贝叶斯优化（与 pipeline 文档一致，增加修正）：

#### Phase 0: 敏感度分析与搜索空间缩减（一次性，~8 小时）

```
Step 0a: 剪枝准则预选 (~1.5h)
  对 FFN 1024->768 固定剪枝率:
    L1 -> AMOTA_l1
    Taylor -> AMOTA_taylor
    FPGM -> AMOTA_fpgm
  锁定最优准则，移出搜索空间

Step 0b: 量化逐层敏感度 (~7h, 可过夜自动化)
  对 BEV encoder + 检测头 ~30 层:
    逐层单独 INT8 -> delta_AMOTA[i]
  分三档: 锁定INT8 / 搜索 / 锁定FP16

Step 0c: 关键交互项测试 (~1h, 解决 pipeline 问题三)
  对 3 个高耦合层对:
    (位置编码, BEV变换): 联合INT8 -> delta_joint
    (LiDAR分支融合点, Camera分支融合点): 联合INT8 -> delta_joint
    (TSA, SCA): 联合INT8 -> delta_joint
  生成交互修正项 delta_ij

Step 0d: 硬件 latency LUT (~0.5h)
  对关键层配置 benchmark:
    FFN dim={512,768,1024} x {INT8,FP16}
    Attention dim=256 x {INT8,FP16}
    各 plugin 实现对比
  -> latency_LUT[layer_type, dim, bits]

输出: sensitivity_map + interaction_terms + latency_LUT
有效搜索维度: ~30维 -> ~12-15维
```

#### Phase 1: 嵌套搜索（自动化，~15-20 小时）

```
外循环: 遍历 B1(剪枝) + D(拓扑) 配置
  |
  +-- 约束预检查 (通道对齐、显存粗估、策略兼容)
  |
  +-- 执行剪枝 + 知识蒸馏微调 (5 epochs, ~2h/配置)
  |
  +-- 内循环: 贝叶斯优化 B2(量化) + D(非拓扑) 配置
       |
       +-- Level 1 廉价评估 (<1秒/候选):
       |     pred_acc_drop = sum(w_i * delta_i) + sum(w_ij * delta_ij)
       |     pred_latency = latency_LUT 查表
       |     pred_memory = 解析估算
       |     过滤 ~70-80%
       |
       +-- Level 2 真实评估 (15-30min/候选):
             量化校准 -> ONNX 导出 -> TRT 构建 -> 真实 profiling
             验证集子集 (1/10) 精度
             -> 反馈至 GP 代理模型
             重复 15-20 轮

  GP warm-start: 外循环相邻配置间迁移先验
```

#### Phase 2: 外层固定优化（半自动，~8-12 小时）

```
对 Top-3 配置:
  A. 知识蒸馏 (如果 Phase 1 内已做则跳过)
  B. 重新量化校准
  C. 条件重搜索: if latency 偏移 > 10%, 触发轻量 BO (5-10轮)
  D. TRT 最终构建 (最高优化级别)
  E. 完整验证集评估 + 多场景测试
```

### 4.3 性能评估方法

| 评估层级 | 指标 | 评估方式 | 耗时 |
|---------|------|---------|------|
| Level 0 (约束检查) | 通道对齐、策略兼容 | 规则检查 | <1ms |
| Level 1 (代理评估) | 精度预估、延迟预估、显存预估 | sensitivity_map + LUT + 解析公式 | <1s |
| Level 2 (真实评估) | AMOTA(1/10验证集)、实测延迟、GPU显存峰值 | TRT engine 构建 + 推理 | 15-30min |
| Level 3 (完整评估) | AMOTA(全验证集)、多 agent 场景、显存水位 | 完整评估流程 | ~30min |

### 4.4 贝叶斯优化框架选择

| 框架 | 特点 | 推荐度 |
|------|------|:------:|
| **BoTorch** (Meta) | PyTorch 原生，支持多目标 BO、约束 BO、warm-start | 首选 |
| Optuna | 易用，支持 TPE/GP，但多目标支持较弱 | 备选 |
| Ax (Meta) | BoTorch 上层封装，更易用但灵活性低 | 备选 |

**推荐 BoTorch**：
- 原生支持约束优化（latency < target, memory < target）
- 支持 GP warm-start（外循环迁移先验）
- 支持自定义 acquisition function

```python
# BoTorch 核心用法示例
from botorch.models import SingleTaskGP
from botorch.optim import optimize_acqf
from botorch.acquisition import ExpectedImprovement

# 定义搜索空间
bounds = torch.tensor([
    [0.0, 0.0, 0.0, ...],  # 下界
    [1.0, 1.0, 1.0, ...],  # 上界
])

# 初始随机采样
train_X = generate_initial_samples(n=5)
train_Y = evaluate_configs(train_X)  # Level 2 真实评估

# BO 循环
for i in range(20):
    gp = SingleTaskGP(train_X, train_Y)
    ei = ExpectedImprovement(gp, best_f=train_Y.max())
    candidate, _ = optimize_acqf(ei, bounds=bounds, q=1)
    
    # Level 1 过滤
    if not passes_level1(candidate):
        continue
    
    # Level 2 真实评估
    y = evaluate_config(candidate)
    train_X = torch.cat([train_X, candidate])
    train_Y = torch.cat([train_Y, y])
```
