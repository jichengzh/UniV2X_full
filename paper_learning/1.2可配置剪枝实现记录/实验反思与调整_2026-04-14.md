## 二、错误归因：为什么零重训迁移失败

### 2.1 根因

DCN 的核心权重是在"有 offset+mask 辅助采样"的环境下训练的：
- 前向传播时，权重不是对规则 3×3 邻域卷积，而是对 offset 预测的**不规则位置**卷积
- 这些权重学到的表达是"**假设存在灵活采样**"的
- 直接搬到普通 Conv2d（固定 3×3 采样），等于"让一个习惯可变参考点的工具在刚性轨道上工作"
- TP 减半 = 目标特征表达已经崩溃

### 2.2 反例佐证：李星峰周报（2026-04-11）的做法

**他的步骤：**

1. 配置文件：`dcn=None` + `stage_with_dcn=(False, False, False, False)`
2. **权重初始化**：加载**原始 ImageNet ResNet-101 权重**（不是 stg2 的带 DCN checkpoint）
   - 即：该部分网络结构和权重初始化保持与 resnet101 原始一致
3. **微调 3-6 个 epoch**

**他的结果：**
> "经过 3-6 个轮次的微调后，模型的检测精度基本都能恢复到原始 resnet101 中带有 DCN 层 的模型几乎一致的水平（相差最大不超过 0.1），甚至有的微调后的模型比原始模型的检测精度还稍高。"

**对比：两种方法的关键差异**

| 维度 | 我的做法（失败） | 李星峰做法（成功） |
|---|---|---|
| 权重来源 | stg2 checkpoint 的 DCN weight | ImageNet ResNet-101 原始权重 |
| 权重含义 | "在 DCN 辅助下训练过的 weight" | "规则 conv 下从头训练的 weight" |
| 是否微调 | ❌ 零重训 | ✅ 3-6 epoch |
| 预期行为 | 直接可用 | 需要让下游重新适应 |
| 实测 AMOTA | 0.104 | 几乎恢复到 DCN 水平 |

### 2.3 本质认识

**DCN 替换不是"删一个算子"，而是"换一种特征范式"**：
- 删 DCN = 从"可变采样范式"切换到"规则卷积范式"
- 下游 BEV encoder + 各 head 都是基于"DCN 提取的特征"训练的
- 范式切换后，整个下游管线需要**重新适应新的特征分布**
- 3-6 epoch 是让下游适应新 backbone 的代价

## 六、关键技术教训（写给未来的自己）

### 6.1 权重迁移的陷阱

**错误假设**："权重形状一样 → 可以直接迁移"

**真实情况**：权重形状一样，但 **"训练时所处的算子环境"** 不同。DCN 环境下的 weight 和 Conv2d 环境下的 weight 即使 shape 相同，语义完全不同。

**教训**：
- 算子替换（DCN → Conv）必须伴随权重重新适应（至少 epoch 级别微调）
- "零重训"迁移只在**完全等价**的算子之间有效（如 Conv2d 换 bias=False）

### 6.2 "不可达参数"的幻觉

**我之前的判断**：backbone 44M 因为 DCN 不可剪，永久退出搜索空间。

**正确认识**：DCN 本身不是不可逾越的障碍，只要加上微调预算就能绕开。

**教训**：在排除"不可剪"之前，先算清楚"重训的成本 vs. 不剪的损失"。李星峰 3-6 epoch 的代价换来 44M 参数的解锁 + 21.6% 端到端加速，这个交易非常划算。

### 6.3 文献检索的重要性

**我花了 1 小时做 DCN 替换实验。** 如果我先读了李星峰的周报，就能直接知道：
- 方案 B（替换 DCN）**是可行的**，他已证明
- 需要的额外工作是**微调**，不是"零重训"
- non-DCN baseline checkpoint **可能已经存在**

**教训**：
- **新实验前先做文献检索**（项目内已有资料也算文献）
- `paper_learning/` 目录下的周报、笔记应该是起点，而不是事后补课

### 6.4 搜索空间设计的迭代

**原则**：**搜索空间不是一次性设计的**，随着实验进行会大幅调整。

- 第一轮（实施计划 1.2）：P1-P9 九维
- 第二轮（解耦分析）：锁定 P4-P7，搜索 5 维
- 第三轮（参数分布实测）：发现 P3/P8 ROI 太低
- 第四轮（DCN 实验 + 周报解读）：解锁 P10 backbone，新增 P12 seg_mask

**教训**：定期用"实际数据"去挑战假设。每个维度都要能回答：
1. 能触及多少参数？
2. 与其他维度有没有强相关？
3. 工程实现代价？
4. 对精度的实测影响（不是估计）？

## 九、P1 FFN 30% 实验的工程反思（2026-04-14 晚）

在完成 P1 FFN 30% 真实 AMOTA 实验过程中，遇到 **4 个评估管线的坑**，每一个都曾让实验卡住甚至失败。这些坑不在代码逻辑错误里，全是**环境/接口/生命周期**层面的工程问题。记录如下供后续参考。

### 9.1 坑 1：`test.py --launcher none` 被 `assert False` 禁用

**现象**：直接用 `python tools/test.py ... --launcher none` 启动，立刻 `AssertionError` 退出。

**根因**：`test.py:248` 有硬编码 `assert False`：
```python
if not distributed:
    assert False
    # model = MMDataParallel(model_multi_agents, device_ids=[0])
    ...
else:
    model_multi_agents = MMDistributedDataParallel(...)
```
作者当初为了强制分布式模式写死。

**解决**：新建 `tools/test_with_pruning.py` 绕开这个 assert，直接用 `MMDataParallel + single_gpu_test`（非分布式路径）。

**教训**：**不要假设现有工具脚本在所有路径上都工作**。看似无害的 `assert False` 可能锁死一条完整的执行路径。

---

### 9.2 坑 2：`workers_per_gpu > 0` 时 DataLoader worker spawn 失败

**现象**：分布式模式下，剪枝后模型进入评估循环，第一次迭代 `for data in data_loader` 时抛异常：
```
TypeError: cannot pickle 'dict_keys' object
    in torch.multiprocessing.reduction.dump
```

**根因**：PyTorch DataLoader 以 spawn 模式创建 worker 进程时，需要 pickle 整个主进程状态。剪枝代码在某处留下了 `dict_keys` 对象的引用（具体来源未定位），spawn 时无法序列化。

**解决**：改用 `workers_per_gpu=0`（主进程加载数据，无 spawn）。代价：评估变慢 3-4 倍（168 samples 从 ~3 min 增加到 ~17 min）。

**教训**：
- **剪枝对运行时状态的副作用很难预测**。除了权重维度变化，可能还有其他隐式状态污染
- **spawn pickle 边界**是常见的黑盒。一旦踩坑，最快的路是回退到 `num_workers=0` 而不是深挖根因
- 真实部署时这不是问题（TRT engine 不需要 PyTorch dataloader）

---

### 9.3 坑 3：`format_results` 的 `tmp_dir` GC 竞态

**现象**：评估时检测指标打印正常，但跟踪评估立刻抛 `AssertionError: Error: The result file does not exist!`。

**根因**：`spd_vehicle_e2e_dataset.py` 的 `evaluate()` 函数：
```python
result_files, tmp_dir = self.format_results(results, jsonfile_prefix)
result_files_det, tmp_dir = self.format_results_det(results, jsonfile_prefix)  # ← tmp_dir 被覆盖！
```

两次调用都返回一个 `TemporaryDirectory` 对象赋给 `tmp_dir`。第二次赋值后，第一次的 `tmp_dir` 失去引用，Python GC 触发其 `__del__`，整个临时目录（包括里面的 `results_nusc.json`）被删除。

随后 `TrackingEval_custom` 尝试读 `result_files`（指向已删除的文件）→ 触发 `assert os.path.exists(result_path)` 失败。

**为什么 baseline 没遇到？** 时序非确定性。baseline 的 GC 可能延迟到 TrackingEval 之后才回收。我们的运行恰好 GC 早于评估步骤。

**解决**：评估时显式传 `jsonfile_prefix` 参数，用持久化目录代替 `TemporaryDirectory`：
```python
metrics = dataset.evaluate(outputs, jsonfile_prefix="output/p1_ffn_30pct_eval_tmp")
```

**教训**：
- **依赖未显式文档化的 GC 时序 = 炸弹**。tmp_dir 必须有稳定引用或持久化
- 遇到"某次能跑某次不能跑"的现象，优先怀疑生命周期/GC/异步问题
- 这其实是上游 mmdet3d plugin 的 bug，应该报给作者

---

### 9.4 坑 4：`single_gpu_test` vs `custom_multi_gpu_test` 的 pkl 格式差异

**现象**：保存的 pkl 能被 `mmcv.load()` 读回，但传给 `dataset.evaluate()` 时走入错误分支。

**根因**：两种测试函数返回结构不同：

| 来源 | 返回格式 |
|---|---|
| `custom_multi_gpu_test`（baseline 使用） | `{'bbox_results': [dict×168]}` |
| `single_gpu_test`（本实验使用） | `[dict×168]` |

`evaluate()` 有条件分支：`if isinstance(results, dict): results = results['bbox_results']`。只有前者走这条分支，后者直接进入 `format_results(results, ...)`，格式不完全兼容。

**解决**：加载 pkl 后手动 wrap 成 dict：
```python
outputs = mmcv.load(args.pkl)
if isinstance(outputs, list):
    outputs = {"bbox_results": outputs}
```

**教训**：
- **输出格式是隐式接口**。不同测试函数产出的 pkl 互不通用
- 这类 API 不一致通常没有文档。最快的检测方法是对比一个已知工作的产物

## 十、搜索空间的"局部 vs 级联"分类盲区（2026-04-15）

### 10.1 触发事件

在准备执行 P2 (attn_proj_ratio) 实验时，重新审视了 `prune_direct` 的实现：

```python
# prune_direct 中的 P2 路径
for name, module in model.named_modules():
    if pruner_cls in pruners:
        idxs = _pick_channels_by_importance(module.value_proj, num_prune)
        pruners[pruner_cls].prune_out_channels(module, idxs)  # 仅剪 module 内部
        pruners[pruner_cls].prune_in_channels(module, idxs)   # 仅剪 module 内部
```

**意识到**：每个 attention 模块的 `embed_dims` 各自缩减，**但没有跨模块级联**。
- 上游 BEV embedding 仍然 256 维
- 下游 attention 模块的 input 仍然 256 维
- **如果跑 P2 实验，会在 forward 时 shape mismatch crash**

### 10.2 根本认知错误

我之前把 P1/P2/P3 都列为"通道剪枝维度"，认为它们是**同质问题**，差别只是剪枝目标的位置不同。**实际上它们是两类完全不同的工程问题**：

| 类型 | 维度示例 | 改变什么 | 是否需要级联 |
|---|---|---|:---:|
| **局部剪枝** | P1 FFN, P3 head, seg_MLP | 模块**内部中间层** | ❌ 不需要 |
| **级联剪枝** | P2 attn_proj, P14 embed_dims | 模块的 **I/O 维度** | ✅ 必须 |
| **拓扑剪枝** | P9 decoder 层数 | 模块**数量** | 独立机制 |

### 10.3 局部剪枝（已可用）的特征

```
input (D dim) → Linear(D, D_mid) → activation → Linear(D_mid, D) → output (D dim)
                       ↑ 剪 D_mid                ↑ 同步剪 D_mid
```

- 改的是中间维度 `D_mid`
- 输入输出都是 `D`，**保持不变**
- 一个 module 内自包含，不影响外部
- 实现简单：`_collect_ffn_pairs` 找成对的 (first_linear, second_linear) 即可
- **适用场景**：FFN 中间层、检测头中间层、MLP 中间层

### 10.4 级联剪枝（待实现）的特征

```
BEV embedding (D=256)              ← 必须同步缩
       ↓
LayerNorm (D=256)                  ← 必须同步缩
       ↓
attention.value_proj (D->D)        ← 这里想剪
       ↓
attention.output_proj (D->D)       ← 必须同步缩
       ↓
LayerNorm (D=256)                  ← 必须同步缩
       ↓
FFN.in (D=256)                     ← 必须同步缩
       ↓
FFN.out (D=256)                    ← 必须同步缩
       ↓
next attention.value_proj (D->D)   ← 必须同步缩
       ...
```

- 改的是 `embed_dims` 本身
- **整个 BEV 信息流都受影响**
- 需要完整依赖追踪（DepGraph 的本职工作）
- 我们用 `prune_direct` 绕过了 DepGraph，**也就放弃了级联能力**
- **适用场景**：embed_dims 全局缩减、attention I/O 通道剪枝

### 10.5 拓扑剪枝（独立实现）的特征

```
模块数量从 N 改为 M (M<N)
- 删除某些 transformer 层
- 删除对应的 cls/reg branches 索引
- 不改变剩余模块的内部维度
```

- P9 `decoder_num_layers` 是这一类
- 我们已通过 `prune_decoder_layers()` 单独实现
- 与通道剪枝完全正交

### 10.6 教训

**搜索空间设计应该按"实现机制"分类，而不是按"剪枝位置"分类。**

我之前的搜索空间表（实施计划 1.2 中）：

| 维度 | 名称 |
|---|---|
| P1 | ffn_mid_ratio |
| P2 | attn_proj_ratio |
| P3 | head_mid_ratio |
| ... | ... |

这个表暗示 P1/P2/P3 是同质问题。**但实际工程难度差 10 倍**：
- P1/P3：30 行代码 + 几小时调试就能跑通
- P2：需要要么搞 DepGraph tracing（我们已放弃），要么手写级联依赖
- P14：影响整个模型，需要重新 train 

**正确的分类应该是**：

```
搜索空间 = {
    局部剪枝维度（已可用）: [P1 FFN, P3 head_mid, seg_MLP],
    级联剪枝维度（待工程化）: [P2 attn_proj, P14 embed_dims],
    拓扑剪枝维度（独立机制）: [P9 decoder_num_layers],
    禁区: [DCN backbone（需要重训）, sampling_offsets/attention_weights（坐标敏感）]
}
```

### 10.7 对实施计划文档的修正

**`实施计划_1.2_可配置剪枝.md` 需要在搜索空间章节注明**：

> P1/P3/seg_MLP 是局部剪枝，已实现；P2/P14 是级联剪枝，需要专门实现 DepGraph 适配或手写依赖追踪。
> 不要用统一比例配置去跑 P2，会直接 crash。

### 10.8 给后续设计的建议

1. **每个新维度先回答**：是局部还是级联？
2. **如果是级联**：先评估"实现该维度的工程量" vs "该维度的最大参数节省 ROI"
3. **优先做局部**：因为它们边界清晰、代价可控、易于做敏感度扫描
4. **级联放后期**：当局部剪枝的 ROI 饱和后再投入

## 十三、Phase B 完成后的反思（2026-04-16 凌晨新增）

> Phase B 全部完成：3×3 ablation + 70/80% 扩展 + P9 层数扫描共 11 个微调实验。
> 从这批实验中提炼出**三个决定性认知升级**，推翻或细化了前面第 1-12 节的假设。

### 13.1 认知升级 1：零微调是伪上限

**旧认知**（第 9-11 节建立）：FFN 安全线 40%，崩溃线 60%。

**新数据**（完成报告 § 3.10）：
- 60% + q=2 微调 → AMOTA **0.3354 > baseline 0.3298**
- 70% + q=2 微调 → AMOTA 0.3102（仍可用）
- 80% + q=2 微调 → AMOTA 0.3102（同 70%，饱和）

**结论**：**FFN 真实上限至少到 70%，零微调测出的崩溃线整体需要 +20~30 个百分点修正**。

**教训**：做剪枝评估时**把微调当 baseline 的一部分**。脱离微调讨论剪枝上限没有工程意义（除非你是在学术上研究 zero-shot 剪枝理论）。

### 13.2 认知升级 2：queue_length × compression 的交互是 Pareto 决定因素

**意外发现**：

| 压缩率 | q=1 微调 AMOTA | q=2 微调 AMOTA | 最优 queue |
|---|---|---|---|
| 30% | **0.3356** 🏆 | 0.3306 | **q=1** |
| 50% | 0.2904 | **0.3129** 🏆 | q=2 |
| 60% | 0.2826 | **0.3354** 🏆 | q=2 |

**机制解释**（假设，需要后续实验验证）：
- 低压缩：FFN 容量充足，单帧重建即可；q=2 训练的"2 帧固定窗口"与 eval 的"可变长度 video_test_mode"分布不一致，反而过拟合训练分布
- 高压缩：FFN 容量被严重压缩，**必须利用 prev_bev 的时序冗余**；q=2 教会模型这一点

**教训**：
- 微调超参不应被独立于剪枝比例优化
- v1/v2 把 `queue_length` 归为"内存管理手段"是错的，它是**独立搜索维度**
- 这是一个可发论文的新发现：**"在 BEV E2E 任务上，微调的时序上下文长度应该与结构化剪枝比例正相关"**

### 13.3 认知升级 3：层剪枝 vs 通道剪枝的本质区别

**Phase B.3 数据**：

| 变化 | 参数省 | AMOTA 损失 |
|---|---|---|
| Decoder 6 → 5 层（去 1 层）| -1.08% | -0.02 |
| Decoder 5 → 4 层（去 1 层）| -1.08% | **-0.09**（断崖）|
| FFN 保留 70% → 40%（去 30%）| -3.4% | +0.006（反升）|
| FFN 保留 40% → 20%（去 20%）| -2.3% | -0.025 |

**观察**：同样 -1~3% 的参数缩减：
- FFN 通道剪枝：AMOTA 几乎无损
- Decoder 层剪枝：AMOTA 断崖

**机制**：
- FFN 是一个层内的**多通道冗余冗余**，减通道 = 降低表达能力，微调可补
- Decoder 层是 query → BEV 的一次**refinement iteration**，减层 = 跳过一次整体的特征对齐
- Refinement 的非线性累积使得 "5→4 层" 比 "4→3 层" 还要关键

**教训**：**在联合压缩时，优先剪 FFN 通道，次剪 decoder 层。**
- P1 FFN 的 Pareto 前沿是平滑的（可以一直剪到 70% 再回落）
- P9 decoder 的 Pareto 前沿是断崖的（只能剪 1 层，多剪必崩）

### 13.4 认知升级 4：`ego_agent.forward_train` 的 skip_aux_heads 是工程和论文的共同贡献

**发现**（完成报告 § 3.9.5）：UniV2X 原始训练配置（queue=3, 全 head loss）需要 **>24 GB GPU**。在消费级 4090 上必须：

- `--train-modules pts_bbox_head`：-600 MB 参数 state
- `--skip-aux-heads`：**-8 GB activation**（最大头）
- `--queue-length 1/2`：**-6 GB activation**

**把原本不能跑的训练压到 9-17 GB 之内**。

**论文贡献**（值得独立讨论）：
1. 系统记录 UniV2X 每个优化 flag 的显存/精度影响
2. 给出消费级 GPU 上的部署方法论
3. 对未来做 BEV E2E 剪枝/量化的团队提供可复用的实践

### 13.5 原始搜索空间定义的三处错误（对比第 5 节）

| § 5 的定义 | 新认识 | 修正 |
|---|---|---|
| "P1 FFN 可触及 6.3M 参数" | 实际 -6.96% = 7M（扩展到 seg_head decoder FFN 后增大）| 数字略错，结论对 |

### 13.6 对下一阶段（Phase C）的三条指导原则

1. **停止扩展剪枝维度**：Phase B 已证明 P1 + P9 组合已经是可用剪枝搜索空间的 90%。P2/P3/P8/seg 不是下一个实验的入口。
2. **立刻做剪枝 × 量化联合**：我们有现成的 INT8 管线（commit 99d5379），要尽快做 FFN 60% + INT8 的联合评估（Phase C 核心实验）。

### 13.8 认知升级 5（2026-04-16 再补充）：多任务影响的不对称性

**触发**：用户提醒"seg_head 被我们剪了，对 detection 有影响吗"。回看所有 eval log 的分割 IoU 数据后发现：

**P1 (FFN pruning) 和 P9 (decoder 层数) 在多任务上呈现完全相反的 trade-off**：

| 维度 | tracking (AMOTA) | detection (mAP) | BEV 分割 (drivable/lanes/crossing/contour) |
|---|---|---|---|
| **P1 FFN 30%** | 保（微调后 +0.006）| 微损 | -0.1% / -8% / -11% / -7% |
| **P1 FFN 60%** | 保（+0.006）| -0.003 | -6% / -55% / -57% / -54% |
| **P1 FFN 80%** | 微损（-0.02）| -0.007 | **-17% / -91% / -96% / -87%** 💀 |
| **P9 decoder 5 层** | -0.02 | -0.01 | **+2% / +1% / +3% / 0%**（保留）|
| **P9 decoder 4 层** | 断崖 (-0.11)| -0.025 | **+2% / 0% / +1% / -2%**（保留）|
| **P9 decoder 3 层** | 崩 (-0.26)| 崩 | **+2% / +1% / +2% / -1%**（保留）|

#### 机制解释

UniV2X 的架构是 **"共享 encoder, 独立 head"**：

```
encoder (共享 BEV 编码) → bev_embed ─┬─► track_head （decoder + branches）
                                      ├─► seg_head
                                      ├─► motion_head
                                      └─► occ_head / planning_head
```

- **P1 encoder FFN 被剪** → bev_embed 变化 → 所有 head 都受影响
  - track_head 内部可微调补偿 → AMOTA 保住
  - seg_head 在 `--skip-aux-heads` 下没训 → 无法适应 → 严重掉
- **P1 decoder FFN 被剪** → 只影响 track 内部迭代 → 其他 head 不受影响
- **P9 decoder 层数减少** → 同样只影响 track 内部 → 其他 head 完全不受影响

#### 实际影响

**我们的 P1 config 把 encoder 和 decoder 的 `ffn_mid_ratio` 绑定了**（用同一个值），所以 P1 实验实际上混合了：
- encoder FFN 剪（影响全任务）+ decoder FFN 剪（只影响 track）

**解耦后**应能得到更好的多任务平衡：
- `encoder.ffn_mid_ratio = 0.8`（保守保 BEV 共享特征）
- `decoder.ffn_mid_ratio = 0.3`（激进剪 track 路径，微调补）

这是一个 **Phase C 立即可做的消融**，工程量零（只需改一行 config）。

#### 对"seg_MLP 免费午餐"的校正

之前反复说的"seg_head MLP 剪枝对 AMOTA 零影响 = 免费午餐"需要精确化：

| 口径 | 结论 |
|---|---|
| 对 tracking | ✅ 免费午餐（AMOTA 零影响）|
| 对 detection | ✅ 免费午餐（mAP 零影响）|
| 对 drivable 分割 | ✅ 几乎免费（-0.06%）|
| 对 lanes 分割 | ⚠ 有成本（-7.3%）|
| 对 crossing 分割 | ⚠ 有成本（-10.5%）|
| 对 contour 分割 | ⚠ 有成本（-4.5%）|

**纠正前的表述**：绝对的"免费午餐"。
**纠正后的表述**：**"如果项目只看 track/detection 或只关心 drivable 这一类 BEV 分割，是免费午餐；如果关心 lane/crossing 这类精细结构，需付 5-10% 精度成本。"**

#### 评估方法论的修正

以前只报告 AMOTA + mAP。**新标准**：每个实验必须同时报告 `(AMOTA, mAP, drivable_iou, lanes_iou, crossing_iou, contour_iou)` **6 项核心指标**，用于 Pareto 分析。

### 13.9 最终修订的三句话反思（取代 § 12 和 § 13.6）

1. **不要用单一指标衡量多任务模型**。UniV2X 有 tracking + detection + segmentation + motion 等 4 条以上输出，剪枝对它们的影响是不对称的。
2. **架构决定了剪枝成本的去向**。共享 encoder → 剪 encoder 风险高；独立 head → 剪 head 内部只伤它自己。Pareto 最优配置来自"把剪枝放在只影响**你不关心**的输出路径上"。
3. **config 的耦合/解耦本身就是搜索空间的一部分**。P1 encoder/decoder 绑定是错失的机会；蒸馏 vs 不蒸馏是 trade-off；量化是否只量化 backbone vs 全量化，都是值得做的消融。
