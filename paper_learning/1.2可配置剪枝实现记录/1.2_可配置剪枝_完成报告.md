
- 推理完全相同的 168 个 val samples

### 3.5.4 跟踪子指标（TP / FN / ML / MT / IDS）

| 配置 | TP | FN | ML | MT | IDS |
|---|---:|---:|---:|---:|---:|
| baseline | 383 | 473 | 48 | 24 | 28 |
| FFN 20% | 364 | 494 | 47 | 17 | 26 |
| FFN 30% | 343 | 519 | 46 | 16 | 22 |

| FFN 50% | 298 | 571 | 50 | 16 | 15 |
| FFN 60% | **224** | **650** | **61** | **14** | 10 |

### 3.5.5 关键观察

**观察 1：AMOTA-参数关系非单调**

- FFN 30% (0.3189) > FFN 20% (0.3055)
- 这不符合直觉（更多剪枝应该有更大精度损失）
- 可能原因：
  1. **L1 范数选通道的随机性**：不同剪枝比例下，L1 选出的通道分布不同，可能恰好 30% 保留了更多关键通道
  2. **剪枝的正则效应**：FFN 冗余度较高时，适度剪枝可能移除噪声
  3. **评估集规模小（168 samples）**：~0.014 的 AMOTA 差异在统计误差范围内

**观察 2：精度下降不是线性的**

从 20% → 30% → 40% → 50% → 60%：
- Δ AMOTA: -0.024, +0.013, -0.022, -0.031, -0.030
- 30%-40% 是"平缓区"，40% 之后进入"崩溃区"
- **40% 是建议的"安全上界"**

**观察 3：AMOTP（位置精度）单调变差**

| 剪枝率 | AMOTP |
|:---:|:---:|
| 0% | 1.474 |
| 60% | 1.622 |

- 跟踪位置误差随剪枝加剧 ~10%
- 说明 FFN 剪枝主要影响特征表达质量，不影响跟踪拓扑

**观察 4：IDS 反常下降**

| 剪枝率 | IDS |
|:---:|:---:|
| 0% | 28 |
| 60% | 10 |

- ID 切换次数反而减少
- 解释：剪枝后模型变得更保守，跟踪目标数减少（TP 383→224），自然切换减少
- 这不是"改善"，是"因为跟得少，所以切换也少"

**观察 5：60% 剪枝是明显拐点**

| 指标 | 60% vs 50% | 信号 |
|---|---|---|
| AMOTA | -0.030 | 大幅下降 |
| TP | 298→224（-25%） | 目标大量丢失 |
| ML | 50→61（+22%） | 更多目标被完全跟丢 |
| MT | 16→14 | 稳定跟踪目标减少 |

**60% 是零微调 FFN 剪枝的有效天花板**。超过此比例会触发跟踪网络的级联失效。

### 3.5.6 参数缩减 vs 精度损失效率

| 配置 | 每 1% 参数缩减换取的 AMOTA 损失 |
|---|:---:|
| FFN 20% | 0.024 / 1.27 = **0.019** |
| FFN 30% | 0.011 / 1.96 = **0.006** (最优 ROI) |
| FFN 40% | 0.032 / 2.54 = **0.013** |
| FFN 50% | 0.063 / 3.13 = **0.020** |
| FFN 60% | 0.093 / 3.82 = **0.024** |

**FFN 30% 的 ROI 最优**（每 1% 参数节省只损失 0.006 AMOTA）。20% 反常的 ROI 差说明该比例下 L1 选出的通道质量低。

### 3.5.7 选通道策略的局限

本次扫描全部使用 `l1_norm`（最快、无需梯度）。从 20% 异常结果可见 L1 的局限：
- **L1 不知道任务相关性**：大权重 ≠ 对跟踪重要
- **FFN 30% 的优越表现可能是 L1 在该比例下"碰巧选对了"**
- Taylor 重要性（需梯度）理论上更可靠，但单次实验耗时多 30 倍

**建议后续对比 L1 vs Taylor**（在 40% 这一 "敏感比例" 上做对比），预估 Taylor 可稳定提升 0.01-0.02 AMOTA。

## 三点六、seg_head mask_head MLP 剪枝实验（2026-04-15）

### 3.6.1 动机

P1 FFN 扫描实验中，`_is_ffn_layer` 只匹配 `ffns.*.layers.0.0` 模式，未触及 seg_head 内的：
- `stuff_mask_head.blocks.N.mlp.fc1/fc2`（6 个 blocks）
- `things_mask_head.blocks.N.mlp.fc1/fc2`（4 个 blocks）

合计 10 对 MLP (fc1: 256→1024, fc2: 1024→256)，共 **5.26M 参数未剪**。

### 3.6.2 代码扩展

`_collect_ffn_pairs` 新增第二种模式：
```python
# 模式 2: mask_head MLP (block.mlp.fc1/fc2)
for name, module in model.named_modules():
    if hasattr(module, "mlp"):
        mlp = module.mlp
        if hasattr(mlp, "fc1") and hasattr(mlp, "fc2"):
            if mlp.fc1.out_features == mlp.fc2.in_features:
                pairs.append((f"{name}.mlp", mlp.fc1, mlp.fc2))
```

|---|---:|---:|---:|
| **AMOTA** | **0.3189** | **0.3189** | **0**（完全相同） |
| AMOTP | 1.5257 | 1.5257 | 0 |
| mAP | 0.0648 | 0.0648 | 0 |
| TP | 343 | 343 | 0 |
| FP | 102 | 102 | 0 |
| FN | 519 | 519 | 0 |
| MOTA | 0.2934 | 0.2934 | 0 |
| 参数缩减 | 1.96% | **3.55%** | **+1.59%** |

| 跟踪每帧耗时（见 § 3.7） | 166 ms | 148 ms | -18 ms (-11%) |

### 3.6.4 核心发现

1. **跟踪 AMOTA 完全不变**：seg_head.mask_head 是独立的**分割输出分支**，其 MLP 不参与任何 BEV encoder、decoder 或跟踪关联的计算路径。
2. **只有分割指标受影响**：lanes_iou 下降 7.3%，crossing_iou 下降 10.5%。分割对 mask_head MLP 容量敏感。
3. **ROI 评估**：
   - 每 1% 参数省获得 0 AMOTA 损失（无代价）
   - 但代价是分割精度 ~5-10% 下降
   - **如果任务重跟踪、轻分割 → 这是纯收益**
   - **如果任务关注分割 → 需要慎重**

### 3.6.5 对设计的启示

这验证了一个关键假设：**UniV2X 的任务头是"并行而非级联"**。pts_bbox_head（跟踪）和 seg_head（分割）共享 backbone/neck，但下游计算完全解耦。这为差异化剪枝提供了清晰空间：
- 跟踪敏感 → 保守剪 pts_bbox_head
- 分割敏感 → 保守剪 seg_head
- 两者不强关联 → 可以独立优化

## 三点八、P2 attn_proj 剪枝可行性分析（2026-04-15，未执行实验）

### 3.8.1 P2 是什么

P2 (`attn_proj_ratio`) 剪的是 **60 个注意力模块内的 value_proj 和 output_proj 投影层**，共 **3.95M 参数**。

| 位置 | 层数 | 参数 | 具体层 |
|---|:---:|:---:|---|
| pts_bbox_head.encoder | 24 | 1.58M | TSA.value_proj + TSA.output_proj + SCA.output_proj + MSDA3D.value_proj |
| pts_bbox_head.decoder | 12 | 0.79M | CustomMSDA.value_proj + CustomMSDA.output_proj |
| seg_head.encoder | 12 | 0.79M | encoder attentions value_proj + output_proj |
| seg_head.decoder | 12 | 0.79M | decoder attentions value_proj + output_proj |
| **合计** | **60** | **3.95M** | 全部为 256→256 的 Linear |

### 3.8.2 P1 vs P2 的本质区别

| 维度 | P1 (FFN) | P2 (attn_proj) |
|---|---|---|
| 改变什么 | FFN **中间维度** `feedforward_channels` (512→352) | **embed_dims** 本身 (256→204) |
| I/O 维度 | 输入输出**保持** 256 维不变 | **输入输出都变** |
| 影响范围 | 局部（单 FFN 内） | **全局**（整个 BEVFormer 的特征通道） |
| 级联依赖 | 无 | 必须同步：BEV embedding + LayerNorm + 下游 FFN + cross-attention 输入 + sampling_offsets/attention_weights 输入 |
| DepGraph 自动处理 | 不需要 | **强需要** |

### 3.8.3 当前实现的局限

我们的 `prune_direct` 路径用 `register_univ2x_pruners()` + 手动调用 `prune_out_channels` / `prune_in_channels`：

```python
# prune_direct 中的 P2 路径
for name, module in model.named_modules():
    if pruner_cls in pruners:
        idxs = _pick_channels_by_importance(module.value_proj, num_prune)
        pruners[pruner_cls].prune_out_channels(module, idxs)  # 剪 module 内部
        pruners[pruner_cls].prune_in_channels(module, idxs)   # 剪 module 内部
```

**问题**：每个模块的 `embed_dims` 各自独立缩减，但**没有跨模块的级联**：
- 上游 BEV embedding 仍是 256 维
- 下游 attention 模块期望的输入也是 256 维
- 同一个 attention 内的 `sampling_offsets` 输入维度被改成 204，但 BEV query 依然是 256

**直接执行 P2 会在 forward 时遇到 shape mismatch 而 crash。**

### 3.8.4 三种解决方案

| 方案 | 描述 | 复杂度 | 风险 |
|---|---|:---:|:---:|
| **A** | 跳过 P2，只剪 P1 + seg_MLP | 低 | 低 |
| **B** | 局部 P2：只剪 value_proj 内部，保持 I/O 不变 | 中 | 中（需修改 attention 内部 reshape 逻辑） |
| **C** | 完整 embed_dims 全局剪枝 | **高** | **高**（影响整个模型，等同于"P14 embed_dims"维度，原本就标记为高风险） |

### 3.8.5 方案 B 的具体说明

如果要让 P2 工作但不引入全局级联，可以：

1. 修改 attention 模块 forward，使 value_proj 的输入仍是 256 维（保持 I/O 接口）
2. 但内部计算用更小的维度：`value_proj: Linear(256, 204)` → `head_dim` 改为 `204/num_heads`
3. 计算完后用 `output_proj: Linear(204, 256)` 还原回 256 维

这种做法本质是"瓶颈 attention"（bottleneck attention），保持外部接口不变。

但 MSDA 的 CUDA kernel 可能不支持非标准 `head_dim`（比如 256/8=32 改为 204/8=25.5 不是整数）。需要保证：
- `embed_dims_new % num_heads == 0` （round_to=8 不一定够）
- MSDA kernel 接受新 head_dim

### 3.8.6 当前结论

**P2 暂不执行实验**。原因：
1. 实现需 ~半天工程（方案 B 的 attention 改造）或 ~2 天（方案 C 的级联剪枝）
2. P2 总可触及参数只有 3.95M（占模型 4%），ROI 远低于 P1 + seg_MLP
3. 当前 P1+seg_MLP 已经达到 3.55% 缩减 + AMOTA 仅 -3.3%，搜索空间仍未饱和
4. 优先级排序：先搞定 Taylor / 微调 / 量化联合，再回头看 P2

### 3.8.7 这个分析的价值

虽然 P2 没跑实验，**但发现了搜索空间设计中的一个根本认知盲区**：

> **"剪同一个模块的不同部分"和"剪相互依赖的多个模块"是两类完全不同的工程问题。**

我们之前把 P1 / P2 / P3 都列为"通道剪枝维度"，认为它们是一类问题。实际上：

- **P1 局部剪枝**（FFN 中间层）：DepGraph 不需要，写一个简单 `_collect_ffn_pairs` 就够
- **P2 级联剪枝**（embed_dims）：必须有 DepGraph 或等价的依赖追踪机制
- **P3 局部剪枝**（head 中间层）：和 P1 同类
- **新发现 seg_MLP 局部剪枝**：和 P1 同类

**这意味着我们的搜索空间应该按"是否需要级联"重新分类**：
- 局部剪枝维度（已可用）：P1, P3, seg_MLP
- 级联剪枝维度（待实现）：P2, P14 embed_dims, 以及未来可能的 backbone 通道剪枝
- 拓扑剪枝维度（独立机制）：P9 decoder 层数

### 3.10.4 关键规律（论文级发现）

#### 规律 1：压缩率越高，queue=2 的相对优势越大

| ratio | q=2-q=1 AMOTA Δ |
|---|---:|
| 30% | **-0.005**（q=2 反差）|
| 50% | **+0.023** |
| 60% | **+0.053 ⚡** |

**机制推测**：
- **低压缩**（30%）：FFN 容量仍充足，单帧 BEV 重建能力强，q=1 的 stateless 训练让模型鲁棒对付 eval 的可变长度 prev_bev；q=2 的固定 2 帧训练反而过拟合训练分布
- **高压缩**（60%）：FFN 被激进压缩，单帧信息不足以重建准确 BEV，**必须依赖 prev_bev 里的时序冗余信息**；q=2 在训练时就教会模型如何利用这个信息

#### 规律 2：微调收益与压缩率强正相关（q=2 比 q=1 更陡）

| ratio | q=1 Δ vs zero-FT | **q=2 Δ vs zero-FT** |

**60% + q=2 从 0.2366 飙到 0.3354（+0.0988），是所有 cell 中最大的恢复量**。

#### 规律 3：IDS 与压缩率负相关（与直觉相反）

| ratio | zero-FT IDS | q=1 IDS | q=2 IDS |
|---|---:|---:|---:|
| baseline | 28 | — | — |
| 30% | 22 | 21 | 24 |
| 50% | 15 | 18 | 22 |
| 60% | **10** | **11** | 15 |

**观察**：压缩率越高，ID 切换越少（10-15 vs baseline 28）。
**机制**：剪枝后的模型关注更 salient 的通道 → 丢多检少 → 能跟的目标反而跟得更稳 → IDS 低。

## 六、重要实现细节与陷阱

### 6.1 multi-agent 模型加载（A2 新增能力）

UniV2X config 使用 `model_ego_agent` + `model_other_agent_*` 而非单 `model`。需要：
1. 构建 ego_agent 模型 + 加载权重
2. 构建 other_agent(s) + 加载权重
3. 包装为 `MultiAgent(ego, others)`
4. 用该 wrapper 加载 full checkpoint

剪枝仅作用于 `model.model_ego_agent`（infra agent 可能后续单独处理）。

### 6.2 direct_prune（绕过 DepGraph tracing）

原因：UniV2X 的 `forward(return_loss=True, ...)` 签名复杂，DepGraph 无法追踪。  
方案：手动实现 FFN/attention_proj/head 的局部剪枝，跳过 DepGraph 追踪。  
结果：剪枝耗时从预估 10-30 秒降到 **0.1 秒**。

### 6.3 评估管线的 3 个坑

#### 坑 1：`test.py --launcher none` 被 `assert False` 禁用
test.py 第 248 行直接 `assert False` 在非分布式模式。需要绕开（我们改用自己的 `MMDataParallel` + `single_gpu_test` 路径）。

#### 坑 2：`workers_per_gpu > 0` 时 spawn worker pickle 失败
剪枝后的模型在 multiprocessing spawn 时遇到 `TypeError: cannot pickle 'dict_keys' object`。解决：`workers_per_gpu=0`（代价是推理慢 3-4 倍）。

#### 坑 3：`format_results` 的 `tmp_dir` GC 竞态
`dataset.evaluate()` 中：
```python
result_files, tmp_dir = self.format_results(results, jsonfile_prefix)
result_files_det, tmp_dir = self.format_results_det(results, jsonfile_prefix)  # 重新赋值 tmp_dir！
```

第一次的 `TemporaryDirectory` 对象失去引用被 GC 清理，后续 `TrackingEval_custom` 找不到 JSON 文件。  
**解决**：显式传入 `jsonfile_prefix` 参数，使用持久目录。

#### 坑 4：pkl 格式不兼容
- `custom_multi_gpu_test`（baseline 用）：保存为 `{'bbox_results': [...]}` dict
- `single_gpu_test`（本实验用）：直接保存 `[...]` list

修复：在加载 pkl 后 wrap 成 dict 格式再评估。

### 14.5 核心总结（每次更新时校对）

**当前"实际剪过刀"的部分**：

| 剪枝类型 | 触及模块数 | 理论可剪参数 | 实际剪掉参数（按 30%）|
|---|:---:|---:|---:|
| **P1 FFN**（pts_bbox_head + seg_head.transformer） | 36 个 FFN 层 | 6.31M + ~2M = ~8.3M | ~2.5M |
| **seg_MLP**（stuff + things mask_head） | 10 个 MLP | 5.26M | ~1.6M |
| **合计** | **46 个 Linear 对** | **~13.6M** | **~3.6M (3.55% 总模型)** |

**主要特征**：
- 全部是 **"D → D_mid → D" 瓶颈结构的局部剪枝**
- **不涉及任何跨模块依赖**（所以不需要 DepGraph 级联追踪）
- 所有"特征通道数" `embed_dims` 保持不变（256）
- 所有坐标敏感层完全保护

**为什么 AMOTA 只掉 0.011（-3.3%）**：
- 我们只剪了模型里"中间冗余" 部分
- 没有动"特征通道数" `embed_dims`
- 没有动 backbone 网络
- 没有动注意力机制的核心计算（Q/K/V）

**想要扩大剪枝覆盖，未来方向**：

| 扩展方向 | 额外可触及参数 | 工程量 | 风险 |
|---|---:|---:|:---:|
