## 0. 背景：为什么要出 v2

### 0.1 v1 执行后的实际状态（2026-04-15 盘点）

| 维度 | v1 定位 | 计划做法 | 实际做了什么 |
|---|---|---|---|
| P1 ffn_mid_ratio | 搜索 | 扫描 {0.5~1.0} | ✅ 跑了 20/30/40/50/60% 真实 AMOTA |
| P2 attn_proj_ratio | 搜索 | 扫描 {0.0~0.5} | ❌ 发现是级联问题，暂缓 |
| P3 head_mid_ratio | 搜索 | 扫描 {0.5~1.0} | ❌ 工程就绪但未跑 |
| **P4 importance_criterion** | 锁定（Phase 0 对比后选最优，推荐 taylor） | 4 种准则对比一次选定 | ❌ **只用了 L1**，Taylor/FPGM/Hessian 从未在真实 checkpoint 验证 |
| P5 pruning_granularity | 锁定（local） | 无需实验 | ✅ 使用 local |
| P6 iterative_steps | 锁定（5） | 无需实验 | ✅ 使用 5 |
| P7 round_to | 锁定（8） | 无需实验 | ✅ 使用 8 |
| P8 head_pruning_ratio | 搜索 | 扫描 {0, 0.125, 0.25} | ❌ 与 P2 一样归为受限 |
| **P9 decoder_num_layers** | 搜索 | 扫描 {5, 6} | ❌ 代码已实现但未跑 |
| (额外) seg_head mask MLP | v1 未列 | — | ✅ 发现是"免费午餐"（AMOTA 0 变化） |

**v1 最大的两个欠账**：
1. **P4 准则对比** —— 原本就是 Phase 0 要做的锁定决策，被跳过了。直接后果是
   FFN sweep 出现 "20% AMOTA 0.3055 < 30% AMOTA 0.3189" 的反常，疑似 L1 选通
   道的随机性。
2. **P9 解码器层数** —— 原本就是搜索维度，没跑就判断"优先级最低"。但每层
   decoder ≈ 1.5M，6→4 = 3M，规模甚至大于整个 FFN sweep 的收益。

### 0.2 来自 survey 的方向校正

`agent_2_quant.md`（10 篇量化/剪枝协同）+ `agent_5_driving.md`（10 篇自动驾
驶部署）的核心启示：

1. **"纯剪枝不单走"** —— 主流 2024-2025 工作几乎都是剪枝 × 量化 × 蒸馏三
   者协同。UPAQ（2025）在 Jetson 上用半结构化剪枝 + 混合精度量化，5.62×
   压缩 + 1.97× 加速；MicroScopiQ（ISCA'25）把剪枝规则直接设计成量化
   outlier 识别规则。
2. **重要性准则已走向"激活感知"** —— AWQ（MLSys'24 Best）用激活分布保护
   1% 显著权重通道，其余激进量化；这与 Taylor（梯度 × 权重）一脉相承，都
   是在否定"只看权重幅度"。
3. **硬件友好稀疏格式** —— TASDER / UPAQ / Panacea 都走 N:M（2:4）路线，
   RTX 4090（Ada Lovelace）原生支持 2:4 Sparse Tensor Core，TRT 直接可用。
4. **精度恢复几乎都用蒸馏** —— QD-BEV（视角引导蒸馏）、PlanKD（E2E 规划
   蒸馏）、DiMA（多模态蒸馏），都验证了蒸馏是剪枝/量化后恢复精度的标准套路。

### 0.3 v2 的核心修正

| 修正方向 | 与 v1 的差异 |
|---|---|
| **补齐 P4/P9 欠账** | v1 规划中已有，v2 把它们前置到第一周 |
| **引入激活感知重要性** | v1 只用了 Torch-Pruning 内置的 L1/Taylor 等，v2 增加 AWQ 风格的 `act_aware` 准则 |
| **P1 与 INT8 合流** | v1 各自独立，v2 设计 `prune → INT8 calibrate → TRT export` 的串联管线 |
| **加入 2:4 N:M 稀疏** | v1 完全没有，v2 列为 Phase C 可选探索 |
| **加入 BEV 蒸馏** | v1 `finetune.epochs: 0` 默认不做，v2 设计最小蒸馏管线 |
| **收缩新剪枝维度** | v1 规划了 P2/P3/P8/P9 全部展开，v2 只收尾 P3/P9，暂缓 P2 |

### 2.1 Phase B 为何扩到 2 周（v2.1 关键调整）

原 v2 把微调留到 Phase C.4a，Phase B 只做零微调对比。这是**错误的框架**，因为：

1. 李星峰 DCN 实验（`实验反思与调整_2026-04-14.md` §1.2）证明：微调可以
   把去 DCN 后 AMOTA 从 **0.104 恢复到 ~0.33**（-68% → ≈0%）。
2. 这意味着我们当前的结论（"P1 60% 零微调崩溃"）**很可能是零微调伪上限**。
   微调后 FFN 60-70% 极可能完全可用。
3. 零微调下的 P4 准则差异（L1 vs Taylor）也可能在微调后被抹平 ——
   如果真如此，P4 的研究价值会下降，需要提前知道。

**结论**：微调必须前置到 Phase B，否则 Phase B 得出的所有"上限"和"准则优劣"
都是不可信的中间结果，浪费实验预算。

#### B.0.X 可训模块白名单策略（`--train-modules`）

**动机**：之前"冻结 backbone + other_agent, 其他全训"的做法导致 OOM（激活内
存在 seg_head / occ_head 等不需要训的模块上浪费）。v2.1 改为**基于剪枝目标
推断可训模块的白名单**。

**通用规则**：
- 被剪模块：必训
- 下游模块（接收它输出的）：必训（适应新输入分布）
- 上游模块：可冻结（未变，下游自适应即可）
- `other_agent_*`（V2X 对端）：**永久冻结**（不在本工作剪枝范围）

**剪枝方案 → 训练范围映射表**（实验时 CLI 传 `--train-modules <xxx>`）：

| 剪枝方案 | 被剪模块 | 下游 | `--train-modules` 值 | 可训参数 (相对 197M) |
|---|---|---|---|---|
| **P1 FFN (encoder+decoder)** | transformer 的 FFN | branches | `pts_bbox_head` | ~20M (10.5%) |
| **P1 FFN (仅 decoder)** | decoder FFN | cls/reg branches | `pts_bbox_head` | 同上 |
| **seg_head MLP** | `seg_head.{stuff,things}_mask_head` | 无（末端） | `seg_head` | ~29M (14.7%) |
| **P3 head_mid** | `pts_bbox_head.{cls,reg}_branches` | 无 | `pts_bbox_head` | ~20M |
| **P9 decoder_num_layers** | decoder 层数 | branches | `pts_bbox_head` | ~20M |
| **联合 P1+seg MLP** | 二者 | 二者 | `pts_bbox_head,seg_head` | ~49M (24.8%) |
| **联合 P1+P9** | 二者 | branches | `pts_bbox_head` | ~20M |
| **全训（对照组）** | — | — | `all` | ~97M（剪枝后的 ego_agent） |

**内存效应**：解冻 N 个参数 ≈ 多占 N × 2~3（activation for backward + optimizer state）的显存。每个实验 log 里会明确写出 "可训参数" 一行，论文可以直接拿来做 ablation（"只训 pts_bbox_head vs 全训"）。

**实现**：`apply_train_modules(model_multi_agents, "pts_bbox_head,seg_head")`
→ 按 `model_ego_agent` 的直接子模块名匹配解冻。参见
`tools/finetune_pruned.py` 中 `apply_train_modules`。

#### B.0.Y 激活内存优化三件套（2026-04-15 实战总结）

B.0.1 冒烟暴露的核心事实：**UniV2X 原始训练配置超过 4090 24GB 显存上限**（观
察到 OOM at 20.92 GiB, forward-only 就 22.66 GiB）。这不是剪枝引入的，是模型
架构本身的激活内存。要在 24GB 显卡上跑 3 epoch 微调，必须组合使用：

| CLI flag | 省显存原理 | 省显存规模 | 对精度的影响 |
|---|---|---|---|
| `--train-modules <subset>` | 只对目标子模块保留 optimizer state + param grad | ~600 MB | 无（只改可训范围）|
| `--skip-aux-heads` | 跳过 seg/motion/occ/planning head 的 forward, 它们的激活不保存 | **~8-10 GB**（最大头）| 这些 head 的权重在本次微调中不更新；对**只追求 track AMOTA** 的实验无影响；**涉及 seg_head 剪枝时禁用** |
| `--queue-length 1` | 从 3 帧时序降到 1 帧, BEV encoder 的 temporal context 消失 | **~6-8 GB** | 无时序监督；适合轻度剪枝短期恢复；若剪枝激进可能需要恢复 queue=3 |

**组合效果**（实测数字，以 P1 FFN 30% 为例）：
- 旧方案（仅冻结 backbone+other_agent）：**22.66 GiB**（OOM）
- 加 `--train-modules pts_bbox_head`：**20.92 GiB**（仍 OOM）
- 再加 `--skip-aux-heads`：仍 20.92 GiB（aux heads 在 forward 时就吃了不少）
- 再加 `--queue-length 1`：预期 **6-10 GiB** ✅

**论文价值**：这张表本身就是个有价值的 ablation —— **说明 UniV2X E2E 微调的
可行性强依赖于显存管理策略**。在自动驾驶 / V2X 领域这是首次系统化记录。

**风险**：V2X-Seq-SPD 的 MultiAgent wrapper dataloader 配置复杂 —— 用 10%
训练集 + 1 epoch 先冒烟，确认 loss 曲线健康再扩到完整 3 epoch。

### B.1 FFN/MLP 剪枝上限重新评估（2026-04-17 ~ 04-23，核心实验）

**这是 v2.1 相对 v2 的最大修正**：从"找零微调安全点"改为"找微调后真实上限"。

**动机**：
- 现有零微调结论："FFN 40% 安全，60% 崩溃"
- 李星峰 DCN 先例：-68% 掉点 → 3 epoch 微调恢复
- **合理推断**：FFN 60-70% 零微调看似崩溃的配置，微调后极可能完全可用
- 如果成立，**真实可交付的剪枝比例比我们之前以为的高一倍**

**实验矩阵**（核心表）：

| 实验 ID | FFN ratio | criterion | 零微调 AMOTA | + 微调 3ep AMOTA | 耗时 |
|---|:---:|---|:---:|:---:|---|
| B.1.1 | 30% | l1_norm | 0.3189（已有） | ? | 4h |
| B.1.2 | 40% | l1_norm | 0.3091（已有） | ? | 4h |
| B.1.3 | 50% | l1_norm | 0.2857（已有） | ? | 4h |
| **B.1.4** | **60%** | l1_norm | 崩（已有） | **? 预期 ≥ 0.30（关键验证点）** | 4h |
| **B.1.5** | **70%** | l1_norm | 未测，预期崩 | **? 预期 ≥ 0.28（探上限）** | 4h |
| B.1.6 | 80% | l1_norm | 未测 | ? 可能彻底崩 | 4h |

**决策流程**：
- 先做 B.1.4（60%）—— 如果微调后 AMOTA ≥ 0.30，证明我们过去的"崩溃线"是假的
- 再做 B.1.5（70%）—— 探真实上限
- B.1.6 只有在 B.1.5 仍成功时才做

**成本**：5-6 个实验 × 4h = 20-24h GPU 时间 ≈ **5-6 天**（连续跑）。这是 Phase B
扩到 2 周的主要成本来源。

**交付**：
- `prune_configs/p1_ffn_{60,70,80}pct.json`
- 完成报告 § 3.12 "微调前后对比表"（v2 最有论文卖点的表之一）
- **锁定决策**：基于 B.1 结果选 1-2 个"微调后可用的激进比例"作为 Phase C 输入

**顺带收益**：B.1 的实验本身就是 Phase C.4a 普通微调的 5 个实验。做完 Phase B.1
等于 Phase C.4a 的 P1 部分已经完成，Phase C.4a 后面只需跑 P9 和联合 INT8 的微调。

### B.2 P4 准则对比（2026-04-24 ~ 04-25，保留但缩减）

**关键调整**：如果微调把准则差异抹平，P4 的研究价值下降。需**先快筛零微调，
再只对极端值做微调验证**，避免在"微调后可能都一样"的维度上花大钱。

#### B.2a 零微调快筛（快速低成本）

固定 FFN mid_ratio = 0.7（30%），只变 criterion：

| 实验 ID | criterion | 需 grad | 耗时 |
|---|---|:---:|---|
| B.2a.1 | l1_norm（已有） | ❌ | — |
| B.2a.2 | l2_norm | ❌ | 17 min |
| B.2a.3 | **taylor（重点）** | ✅ | 20 min |
| B.2a.4 | fpgm | ❌ | 17 min |
| B.2a.5 | random（下界） | ❌ | 17 min |

总耗时 ~1.5h。识别**零微调下最好和最差**的两个准则。

#### B.2b 微调后确认（只跑 2-3 个）

| 实验 ID | criterion | 理由 | 耗时 |
|---|---|---|---|
| B.2b.1 | l1_norm + 微调 3ep | baseline（来自 B.1.1） | 0（复用） |
| B.2b.2 | B.2a 最优 + 微调 3ep | 看是否仍胜 L1 | 4h |
| B.2b.3 | random + 微调 3ep | 看微调能否抹平 random 的坏 | 4h |

**决策规则**：
- 若 B.2b.2 > B.2b.1 显著（+0.01 以上）→ 锁定新准则，Phase C 沿用
- 若 B.2b.2 ≈ B.2b.1（差距 < 0.005）→ **准则不重要**，继续用 L1，把精力放到
  比例搜索和联合量化
- 若 B.2b.3 ≈ B.2b.1 → **准则彻底不重要**（连 random 微调后都一样），直接写
  入论文作为一个反直觉发现

**工程前置**：Taylor 需要 `grad_collector.py` 工作。先在 B.0 冒烟期写一个
`tools/validate_grad_collector.py` 确保 `GroupTaylorImportance` 拿到非零梯度。

### B.3 P9 解码器层数 + 微调（2026-04-26 ~ 04-28）

**调整**：v2 原本是"零微调扫描 + 条件触发蒸馏"，现在改为**直接带微调扫描**。

| 实验 ID | num_layers | 零微调 AMOTA | + 微调 3ep AMOTA | 耗时 |
|---|:---:|:---:|:---:|---|
| B.3.1 | 6 (baseline) | 0.3298 | — | — |
| B.3.2 | 5 | 快速测（17min） | 4h | ~4h |
| B.3.3 | 4 | 快速测 | 4h | ~4h |
| B.3.4 | 3 | 快速测（预期崩） | 4h（关键验证） | ~4h |

**决策**：
- 零微调 + 微调两档都跑，用来验证**层数剪枝是否也服从"微调恢复大部分"的
  规律**（直觉上不一定，因为层数变化是拓扑变化，不是通道变化）
- 若 B.3.4 微调后 AMOTA ≥ 0.30 → decoder 6 层深度确实冗余，可激进
- 若 B.3.4 微调后仍崩 → 锁定 4 层为实用下限

**实现要点**（`prune_decoder_layers()` 已在 `prune_univ2x.py:577`）：
- 保留最后 N 层（当前实现）

## 5. Phase D：N:M 稀疏 + TRT 实测（Week 4）

**目标**：在 Phase C 合流管线基础上，叠加 2:4 半结构化稀疏；验证 4090 Ada
Tensor Core 的真实加速。对标 UPAQ / TASDER。

### D.1 2:4 稀疏实现（2026-05-06 ~ 05-08）

**范围**：只应用到 **剪枝后剩余** 的 FFN Linear 权重（因为结构化剪枝已经删
掉整通道，剩下的权重再做 2:4 稀疏）。

**实现**：
```python
# nm_sparsity.py
import torch
from torch.nn.utils.prune import remove
from torch.sparse import to_sparse_semi_structured

def apply_24_sparsity(model: nn.Module, target_modules: List[str]):
    for name, module in model.named_modules():
        if name in target_modules and isinstance(module, nn.Linear):
            # 每 4 个权重保留 2 个最大的
            W = module.weight.data
            mask = compute_24_mask(W)  # [out, in]
            module.weight.data = W * mask
            # TRT 侧不需要 sparse tensor，只要权重符合 2:4 模式即可
```

**配置**：
- 目标模块：`*.ffns.*.layers.*.0`（FFN 第一层 Linear）
- 不对 attention Linear 做（级联复杂）

### D.2 TRT 导出与实测（2026-05-09 ~ 05-12）

**步骤**：
1. 在 Phase C.3.4 模型基础上叠加 D.1 的 2:4 mask
2. 导出 ONNX（mask 自动体现为零权重）
3. TRT build 时加 `--sparsity=enable` flag
4. 实测 latency（对比 dense 版本）

**实验矩阵**：

| 实验 ID | 基础模型 | 2:4 | INT8 | 测量点 |
|---|---|:---:|:---:|---|
| D.2.1 | P1 30% pruned | ❌ | ❌ | FP32 baseline |
| D.2.2 | P1 30% pruned | ✅ | ❌ | FP32 + 2:4 |
| D.2.3 | P1 30% pruned | ❌ | ✅ | Phase C.3.4 结果复用 |
| **D.2.4** | **P1 30% pruned** | **✅** | **✅** | **v2 最高压缩点** |

**预期**：
- 2:4 on FFN Linear 可给 matmul 带来 **1.3-1.8× 加速**（取决于 batch size）
- 联合 INT8 后 e2e 可能从 -5% 跃升到 -30%+

**风险**：
- 4090 的 Sparse Tensor Core 对 shape 对齐敏感（M/N 必须 8 的倍数，v1 的
  `round_to=8` 正好对齐）
- TRT 版本需要 ≥ 8.5 支持 2:4；现有版本需确认

### D.3 Phase D 决策

如果 2:4 在 BEV 模型上精度掉点不可接受（预计 -0.02 ~ -0.05 AMOTA），可以：
- 降级为"只对 seg_head MLP 做 2:4"（已验证是免费午餐区）
- 或放弃 D，作为未来工作
