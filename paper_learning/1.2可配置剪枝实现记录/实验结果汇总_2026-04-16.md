## 七、多任务权衡分析（2026-04-16 重大补充）

**本节回答用户提出的关键问题**："seg_head 剪枝对检测指标有影响吗？"以及更广泛的"不同剪枝维度对各任务的相对影响"。

### 7.1 UniV2X 的多任务输出

UniV2X ego_agent 产出 **4 条并行任务流**：

| 任务 | 出处模块 | 评估指标 |
|---|---|---|
| **3D 目标跟踪** | `pts_bbox_head` + tracker | AMOTA, AMOTP, MT, ML, IDS |
| **3D 目标检测** | `pts_bbox_head.cls_branches/reg_branches` | mAP, NDS |
| **BEV 语义分割** | `seg_head.stuff_mask_head + things_mask_head` | drivable/lanes/crossing/contour IoU |
| 运动预测/占用/规划 | motion/occ/planning head | 本次 Phase B 实验全部 `--skip-aux-heads`，不评估 |

**关键架构观察**：四个 head **都读取同一个 BEV embedding**（由 `pts_bbox_head.transformer.encoder` 产出）。所以：
- 剪 encoder FFN → BEV 分布变 → **所有下游 head 都可能受影响**
- 剪 decoder 层或 decoder FFN → 只改 track_head 内部细化 → **其他 head 不受影响**

### 7.3 核心发现：P1 和 P9 的本质差异

**✅ 发现 1：P1 FFN 剪枝 → 分割严重下降，跟踪反而改善**

| ratio | AMOTA Δ | drivable Δ | lanes Δ | crossing Δ | contour Δ |
|---|---:|---:|---:|---:|---:|
| 30% q=2 | +0.001 ⬆ | -0.1% | -8.3% | -10.9% | -6.7% |
| 50% q=2 | -0.017 | -2.5% | -33.6% | -38.1% | -30.5% |
| 60% q=2 | **+0.006 ⬆** | -6.0% | -54.9% | -56.7% | -54.1% |
| 70% q=2 | -0.020 | -11.4% | -78.9% | -82.1% | -76.7% |
| 80% q=2 | -0.020 | -16.6% | -91.1% | -96.4% | -87.1% |

**规律**：FFN 剪得越多，分割精度单调下降；跟踪 AMOTA 在 60% 前反而可能超过 baseline。

**✅ 发现 2：P9 decoder 层数剪枝 → 分割不受影响，跟踪断崖**

| 5 | -0.02 | **+2.1%** | +1.3% | +3.0% | +0.3% |
| 4 | -0.11 ⚠ | **+1.9%** | -0.4% | +1.2% | -1.7% |
| 3 | -0.26 💀 | **+2.4%** | +0.6% | +2.1% | -0.8% |

**规律**：decoder 层数减少**不影响 seg IoU**（甚至略有上升，属于 evaluation variance），但跟踪断崖。

### 7.4 为什么会这样？架构分析

```
img ─► backbone ─► neck ─► pts_bbox_head.transformer.encoder (**P1 encoder FFN 在这里**) ─┐
                                                                                          ├─► bev_embed
pts_bbox_head.transformer.decoder (P1 decoder FFN + **P9 decoder 层数** 在这里) ──────────┘                  │
             │                                                                                             │
             ├─► cls/reg branches ────► 3D 检测 → AMOTA 跟踪                                               │
             │                                                                                             │
             └─► (decoder 内部 query refinement)                                                           │
                                                                                                           ↓
                                     seg_head (读 bev_embed) ────────────────────────► BEV 分割 (drivable/lanes/...)
                                     motion_head (读 bev_embed) ─────────────────────► 运动预测
                                     occ_head, planning_head ───────────────────────► 占用/规划
```

- **P1 encoder FFN 剪**：直接影响 bev_embed 质量 → 所有 4 个 head 都受影响
  - 其中 cls/reg branches 因为是 track_head 内部，**可以微调补偿**（所以 AMOTA 不掉）
  - seg_head 在 `--skip-aux-heads` 模式下没训 → 无法补偿 → 严重下降
- **P1 decoder FFN 剪**：只影响 track decoder 的 query 迭代，**不影响 bev_embed**
  - 但我们的 config 把 encoder 和 decoder 的 ffn_mid_ratio 绑定为同一个值，所以 P1 实验实际上两处一起剪
- **P9 decoder 层剪**：只影响 track decoder 的 refinement 迭代次数，**不影响 bev_embed**
  - seg_head 输入完全不变 → seg IoU 持平
  - 但 track 需要多层 refinement → 层数降一档就断崖

### 7.5 立刻有用的两个推论

#### 推论 1：我们的 P1 config 把 encoder/decoder FFN 绑定是个局限

现在 `prune_configs/p1_ffn_*.json` 用同一个 `ffn_mid_ratio`：
```json
"encoder": { "ffn_mid_ratio": 0.4 },  // 剪 BEV encoder FFN (连带影响 seg)
"decoder": { "ffn_mid_ratio": 0.4 },  // 剪 track decoder FFN (只影响 track)
```

**应该解耦**。理论上最优方案：
- `encoder.ffn_mid_ratio = 0.8`（保守剪 BEV encoder，保 seg 质量）
- `decoder.ffn_mid_ratio = 0.3`（激进剪 decoder，省更多参数，track 可微调补）

这是 Phase C 的一个立即可做的消融（约 50 min，单实验）。

#### 推论 2："seg_MLP 免费午餐"需要部分修正

我们之前说"剪 seg_head.stuff/things_mask_head 的 MLP 对 AMOTA 零影响 = 免费午餐"。这**对跟踪/检测任务是真的**，但对**分割任务**：
- drivable IoU：几乎不变（-0.06%），对道路识别影响小
- lanes IoU：-7.3%
- crossing IoU：-10.5%
- contour IoU：-4.5%

**更精确的说法**：如果任务只要 track/detection，seg_MLP 剪枝是免费午餐；如果任务关心道路结构分割，这个剪枝要付成本。

### 7.6 这对 Phase C 联合优化的影响

| 原假设（错）| 修正假设 |
|---|---|
| "剪枝只影响 tracking" | "剪枝 maybe 不影响 tracking, 但影响其他任务"|
| "seg_head 免费剪" | "seg_head 剪枝有分割任务 cost"|
| "P1 和 P9 都是剪枝维度" | "P1 影响共享特征, P9 只影响 track" |
| "用 AMOTA 作为唯一指标" | "需要多任务 Pareto 分析"|

**Phase C 建议新增实验**：
- **C.2b**：FFN 60% + q=2 FT（**不用 --skip-aux-heads**）— 看 seg IoU 能否通过微调恢复（代价是显存需求 + 训练时间增加）
- **C.3b**：FFN encoder_ratio=0.7 + decoder_ratio=0.3 + q=2 FT — 解耦后能否同时保 seg 和 track

### 8.3 三个 counter-intuitive 发现

#### 发现 1：**FFN 剪枝在 TRT 端没有加速**

S1 相对 S0 参数 -6.96%，但 TRT FP32 延迟 **几乎零变化**（46.138 → 46.109 ms, -0.06%）。

**原因**：BEV encoder 的延迟瓶颈是 MSDA 的 bilinear sampling（40000 query × 32 采样点 = 128 万次采样），不是 FFN 的矩阵乘。剪 FFN mid 维度（512 → 200）并不减少 MSDA 的 sampling count。

**对部署的启示**：**结构剪枝在这类模型上主要是"压体积"不是"加速"**。真实加速要靠：
- INT8 量化（Tensor Core 2-4×）
- 或剪枝 MSDA 的采样密度（李星峰周报方向，非 DepGraph 可达）
- 或 2:4 Sparse Tensor Core（Ampere+）

#### 发现 2：**Decoder 层剪枝确实有效（PyTorch 端可见）**

S2 相对 S0 的 `track_decoder` 延迟 **-20.8%**（10.95 → 8.67 ms）。这是线性的（6→5 层 = 17% 减少，实测 -20% 吻合）。

但 **TRT engine 看不到这个效果**，因为 `--backbone-only` 导出不含 decoder。如要让 decoder 加速体现在部署，需要单独导出 track decoder（`export_onnx_univ2x.py --heads-only`），这是 Phase C.2 之后的工作。

#### 发现 3：**S2 (FFN60 + dec5) AMOTA 远差于预期**

- Phase B 独立测：FFN 60% q=2 AMOTA 0.3354；dec=5 q=2 AMOTA 0.3095
- 线性叠加估计：0.3354 + (0.3095 - 0.3298) = **0.3151**
- S2 实测：**0.2749**
- 组合交互代价：**-0.04 AMOTA**

**机制推测**：
- FFN 60% 已经让 BEV 特征"变紧"
- 再减一层 decoder 让 query refinement 迭代少一次
- 两者对 track query → BEV 采样精度的损害**叠加放大**
- Phase B 的"P1 和 P9 不干扰"假设在**同时激进**时不成立

**对 Phase C.2 的警示**：联合量化前要谨慎 — 可能 `S2 + INT8` 比 `S1 + INT8` 更不稳定。

### 8.4 C.1 结论 → 给 Phase C.2 的输入

## 九、Phase C.2 — 剪枝 × INT8 联合：集成阻塞记录（2026-04-16 晚）

### 9.1 已完成的 engine 构建 + latency

| Engine | 精度 | 大小 (MB) | Latency (ms) | 构建时间 |
|---|:---:|---:|---:|---:|
| S0 baseline FP32 | FP32 | 75.64 | 46.138 | 24.7s |
| S0 baseline FP16 | FP16 | 40.48 | 24.126 | 90.5s |
| **S0 baseline INT8** | **INT8** | **39.50** | **24.584** | ~3 min |
| S1 FFN 60% FP32 | FP32 | 72.56 | 46.109 | 24.6s |
| S1 FFN 60% FP16 | FP16 | 39.23 | 24.147 | 91.3s |
| **S1 FFN 60% INT8** | **INT8** | **37.70** | **24.208** | ~10 min |

**观察**：INT8 engine latency（~24.5 ms）和 FP16（~24.1 ms）几乎相同。说明 implicit mode 下大量层 fallback 到 FP16（TRT 10 的 deprecation warning 也支持这个推测）。

### 9.2 ⚠ 集成阻塞：test_trt.py 端到端评估 AMOTA=0

| Engine | test_trt.py AMOTA | mAP | TP / FP / FN |
|---|---:|---:|---|
| S0 baseline INT8 | **💀 0.0** | 0.0045 | 0 / 0 / 884 |
| S1 FFN 60% INT8 | **💀 0.0** | 0.0011 | 0 / 0 / 884 |
| S0 baseline FP16（诊断） | **💀 0.0** | 0.0056 | 0 / 0 / 884 |

**诊断结论**：**不是 INT8 也不是剪枝的问题，FP16 baseline 也崩溃**。

**根因初步分析**：

问题在 `test_trt.py` 的 `_trt_get_bev_features` hook 和我们新导出的 200×200 6-cam ONNX 的集成上。1.1 原管线用的是 **1-cam** ONNX（`univ2x_ego_bev_encoder_200_1cam.onnx`），而我们用 `export_onnx_univ2x.py` 默认导出的是 **6-cam**。6-cam 模式下 input tensor shapes 和 hook 里的 reshape 逻辑可能不一致。

### 9.3 诊断过程与根因定位（2026-04-17）

**诊断步骤**：

| 步骤 | 实验 | AMOTA | 结论 |
|:---:|---|---:|---|
| 1 | baseline 1-cam FP16 engine + baseline ckpt | **0.378** | test_trt.py 本身工作正常 |
| 2 | baseline 1-cam FP16 engine + **剪枝微调 ckpt** | **0.351** | ckpt 没问题，decoder/heads 能用剪枝权重 |
| 3 | **剪枝 1-cam INT8 engine** + 剪枝微调 ckpt (NO --prune-config) | **0.0** | ❌ 剪枝 engine 输出的 BEV embed 有问题 |
| 4 | **剪枝 1-cam INT8 engine** + 剪枝微调 ckpt (WITH --prune-config) | **0.0** | ❌ 同上 |
| 5 | 6-cam FP16/INT8 engine（baseline 或剪枝）| **0.0** | ❌ 6-cam engine 本身和 test_trt hook 不兼容 |

**根因**：`export_onnx_univ2x.py --prune-config` 导出的剪枝 ONNX graph 在 TRT engine 推理时产出错误的 BEV embed。具体表现：
- engine 数值上非零非 NaN（benchmark_trt_engine 确认）
- 但 BEV 特征空间分布严重偏移，导致下游 decoder 无法正确解码目标

**可能机制**（待验证）：
- ONNX trace 时 MSDAPlugin 的 sampling_offsets / attention_weights 使用了剪枝前的 dummy 值
- 剪枝改变了 FFN layer shape 但 MSDA CUDA kernel 内部有 hardcode 的 channel 假设
- 校准数据（`calibration/bev_encoder_calib_inputs.pkl`）是 baseline 模型采集的，对剪枝模型激活分布不适用

**不影响的部分**：
- ✅ PyTorch 端全部精度数据有效（Phase B 完整）
- ✅ TRT engine 构建 + latency 测量有效
- ✅ baseline (未剪枝) TRT 管线完整可用

**对 C.2 的影响**：
- **剪枝 × INT8 的 TRT AMOTA 被阻塞**，需修复 ONNX 导出
- **备选方案**：用 baseline engine + 剪枝 ckpt（decoder/heads 走 PyTorch）已验证可得 AMOTA 0.351，可作为"部分 TRT 加速"的数据点
- 但 PyTorch 侧的精度数据已在 Phase B 全部完成

### 9.4 可用的联合数据点（不依赖剪枝 TRT engine）

虽然剪枝 TRT engine 的 AMOTA 被阻塞，但"baseline engine + 剪枝 ckpt"路径有效：

| 配置 | BEV encoder | Decoder/Heads | AMOTA | mAP | 备注 |
|---|---|---|---:|---:|---|
| baseline 全 PyTorch | PyTorch FP32 | PyTorch FP32 | 0.3298 | 0.0724 | Phase B baseline |
| baseline TRT FP16 + baseline ckpt | TRT FP16 engine | PyTorch FP32 | **0.378** | 0.0768 | 1.1 成果复用 |
| **baseline TRT FP16 + 剪枝微调 ckpt** | TRT FP16 engine | **PyTorch 剪枝** | **0.351** | 0.0738 | **混合加速可用** ✅ |
| 剪枝全 PyTorch (q=2 微调) | PyTorch FP32 | PyTorch FP32 | **0.3354** | 0.0689 | Phase B 最优点 |

**"baseline TRT FP16 + 剪枝 ckpt" 的意义**：
- BEV encoder 获得 TRT FP16 加速（24 ms vs PyTorch 58 ms = **2.4× 加速**）
- Decoder/heads 用剪枝微调后的 PyTorch 权重（AMOTA 保持 0.351）
- 这是一个**不需要修复 ONNX 导出就能部署的方案**
