## 一、Phase A（2026-04-13）— 基础设施搭建

### A1: DepGraph 自定义剪枝器

**目标**：为 UniV2X 中 MSDA/SCA/TSA 等含 CUDA 自定义算子的模块编写剪枝器。

**实现**：在 `custom_pruners.py` 中实现 4 个自定义剪枝器，绕过 DepGraph 自动追踪（因 CUDA 算子不可 trace），改为手动描述依赖关系。

**效果**：剪枝器可正确处理 `sampling_offsets`、`attention_weights` 等坐标敏感层的保护。

### A2: 剪枝执行引擎

**目标**：实现剪枝主入口 `prune_univ2x.py`。

**关键设计决策**：采用 `direct_prune` 路径绕过 DepGraph tracing，原因是 UniV2X 的 `forward(return_loss=True, ...)` 签名复杂。剪枝耗时从预估 10-30 秒降到 **0.1 秒**。

**实现**：
- `prune_model` / `build_pruner` / `_build_ratio_dict` 等核心函数
- `prune_decoder_layers`：decoder 层数剪枝（mock 测试 6→4 通过）
- `post_prune.py`：剪枝后状态更新（embed_dims / num_heads / head_dim / LayerNorm）+ 一致性验证

### A3: 配置与评估管线

**目标**：配置驱动的剪枝管线。

**产出**：
- `prune_and_eval.py`：统一入口
- 3 个预设配置（conservative / moderate / aggressive）
- `apply_prune_config`：将 locked/search 维度分离融入代码

### A4: Phase 0 实验代码

**目标**：敏感度分析脚本。

**状态**：代码完成，但后来被更高效的"全局比例扫描"替代（5 个实验 vs 50+ 个），不再单独执行。

## 二、DCN 验证实验（2026-04-14）— 失败的对照

### 实验动机
探索能否完全移除 backbone 的 DCNv2，解锁 44M 参数的剪枝空间。

### 方法
从 `univ2x_coop_e2e_stg2.pth` 直接将 `ModulatedDeformConv2dPack.weight` 迁移到 `nn.Conv2d`，丢弃 offset/mask 分支，**零微调**。

### 结果
| 指标 | DCN baseline | non-DCN 零重训 |
|---|---:|---:|
| AMOTA | 0.3298 | **0.1039** (-68.5%) |
| TP | 383 | 217 (-43%) |

### 发现
零重训迁移不可行。DCN 环境下训练的权重在规则 Conv2d 下完全失效。但李星峰周报已证明**加上 3-6 epoch 微调**可恢复精度。

## 三、P1 FFN 零微调扫描（2026-04-14）— 首次真实 AMOTA 数据

### 3.1 P1 FFN 30% 首次实验

**实验配置**：encoder+decoder FFN `ffn_mid_ratio=0.7`，L1 norm，local 粒度，5 步迭代，round_to=8。

**结果**：AMOTA 0.3189（-3.3%），参数减少 1.96%。

**重要发现**：
- 部分指标反而改善（RECALL +2%, FP -6%, IDS -21%）
- 印证"FFN 中间维度冗余度高"的假设

### 3.2 FFN 比例扫描（20%-60%）

在同一天完成 20/40/50/60% 四个额外比例的扫描。

**关键观察**：
1. **非单调性**：FFN 30% (0.3189) > FFN 20% (0.3055)，可能是 L1 选通道随机性
2. **效率最优**：FFN 30% 的 ROI 最优（每 1% 参数省仅损 0.006 AMOTA）
3. **崩溃线**：60% 是零微调的有效天花板（TP 从 298 骤降到 224）

### 3.3 评估管线工程坑（4 个）

| 坑 | 表现 | 解决 |
|---|---|---|
| `test.py assert False` | 非分布式模式被禁 | 新建 `test_with_pruning.py` |
| `dict_keys` pickle 失败 | workers>0 崩溃 | `workers_per_gpu=0` |
| `tmp_dir` GC 竞态 | tracking eval 找不到 json | 传 `jsonfile_prefix` |
| pkl 格式不一致 | single vs multi_gpu_test 不兼容 | `eval_pkl_amota.py` 格式包装 |

## 四、seg_head MLP + Latency（2026-04-15）— 扩展剪枝范围

### 4.1 seg_head mask_head MLP 剪枝

**扩展**：`_collect_ffn_pairs` 新增 `block.mlp.fc1/fc2` 模式，覆盖 stuff_mask_head（6 blocks）+ things_mask_head（4 blocks），共 5.26M 参数。

**结果**：
- tracking AMOTA：**完全不变**（0.3189 = FFN 30% 相同）
- 参数缩减：1.96% → **3.55%**（+1.59%）
- 分割：lanes IoU -7.3%, crossing IoU -10.5%

**发现**：seg_head 是独立的分割输出分支，不参与跟踪关联计算。这验证了**UniV2X 的任务头是"并行而非级联"**。

### 4.2 Latency Benchmark

**设置**：RTX 4090, CUDA Event 精确计时, 3 次 warmup + 20 次正式。

**核心数据**：
- bev_encoder -14.9%, seg_things_mask_head -19.7%, e2e -5.0%
- e2e 加速远低于模块加速，原因是 data loading/tracking update 等 CPU-bound 开销占大部分

## 五、P2 可行性分析（2026-04-15）— 未执行实验

### 分析结论
P2 attn_proj 与 P1 FFN 是**完全不同性质的问题**：
- P1（局部剪枝）：改 FFN 内部中间维度，I/O 不变，无级联
- P2（级联剪枝）：改 embed_dims 本身，整个 BEV 信息流都受影响

当前 `prune_direct` 不支持级联，直接执行 P2 会 shape mismatch crash。

**决策**：P2 暂不执行，先完成 ROI 更高的方向。

## 六、Phase B（2026-04-15 ~ 2026-04-16）— 微调突破

### 6.1 B.0 微调 Pipeline 实现

**产出**：`tools/finetune_pruned.py`

**关键 flag**：
- `--train-modules pts_bbox_head`：仅训练 20M 参数
- `--skip-aux-heads`：跳过 seg/motion/occ/planning head forward
- `--queue-length`：控制时序上下文长度

**显存优化发现**：三个 flag 必须组合使用才能在 24GB 4090 上训练。单独使用任何一个都不够。

### 6.2 B.0.1 首次微调突破

**实验**：P1 FFN 30% + q=1 微调 3 epoch。

**结果**：AMOTA 0.3356，**超过未剪枝 baseline 0.3298**（+1.8%）。

**意义**：首次实证"零微调崩溃线是假的"。20 分钟训练即可让剪枝模型超越 baseline。

### 6.3 B.1 3x2 Ablation + 扩展

**完整实验矩阵**：FFN {30%, 50%, 60%} x queue {1, 2} = 6 个微调实验。

**里程碑发现**：
1. **FFN 60% + q=2 = AMOTA 0.3354**，再次超过 baseline
2. **queue_length 最优值随压缩率变化**：30% 用 q=1，>=50% 用 q=2
3. **压缩率越高，q=2 的相对优势越大**：60% 下 q=2 比 q=1 高 +0.053

**70%/80% 扩展**：
- 70%/80% q=2 微调后 AMOTA 均为 0.3102（饱和），说明 FFN 到 30% 剩余后 salient 通道已经饱和

### 6.4 B.3 P9 Decoder 层数扫描

**结果**：5 层可用（-0.02），4 层断崖（-0.11），3 层崩溃（-0.26）。

**对比 FFN 剪枝**：同样 -1~3% 参数缩减，FFN 通道剪枝近乎无损，decoder 层数剪枝断崖。每层 decoder 是一次 query refinement iteration，减层比减通道更具"结构性损伤"。

### 6.5 Phase B 实验时间统计

| 批次 | 实验数 | 挂钟时间 |
|---|:---:|---:|
| 零微调扫描 | 5 | ~1.5h |
| 3x2 ablation | 12 | ~3h |
| 70/80% + P9 | 10 | ~4h |
| **合计** | **~27** | **~10h** |

原计划 Phase B 需 5-6 天，实际 **1.5 天完成**（40 倍加速）。

## 七、Phase C.1（2026-04-16）— TRT 集成验证

### 7.1 目标
验证剪枝模型的 ONNX → TRT 全链路，测出真实部署延迟。

### 7.2 关键发现

**发现 1：FFN 剪枝在 TRT 端无加速**
- S1（FFN 60%）TRT FP32 延迟与 baseline 几乎相同（46.138 vs 46.109 ms）
- 原因：BEV encoder 延迟瓶颈是 MSDA bilinear sampling（128 万次采样），不是 FFN matmul

**发现 2：Decoder 层剪枝确实有效**
- S2（FFN 60% + dec 5）的 track_decoder PyTorch 延迟 -20.8%

**发现 3：FFN60% + dec5 组合 AMOTA 远差于预期**
- 线性叠加估计 0.3151，实测 0.2749，组合交互代价 -0.04
- 两者对 track query → BEV 采样精度的损害叠加放大

### 7.3 TRT 剪枝 Engine AMOTA=0 问题

所有剪枝 TRT engine（FP16/FP32/INT8）的 AMOTA 均为 0。

**诊断排除**：
- 1-cam baseline engine + 剪枝 ckpt → AMOTA 0.351（ckpt 没问题）
- 剪枝 engine 输出 BEV embed 数值非零非 NaN，但特征分布严重偏移

**根因**：`BEVEncoderWrapper` ONNX trace 在 FFN shape 变化后产出错误 BEV。

**Workaround**："baseline TRT engine + 剪枝 ckpt" 混合方案 AMOTA 0.351。

## 八、Phase D.1（2026-04-18）— Encoder/Decoder FFN 解耦

### 8.1 动机
Phase B 的 P1 config 把 encoder 和 decoder FFN 绑定为同一个 ratio，导致剪 encoder FFN 影响 BEV 共享特征 → 分割严重下降。解耦后可以：
- 保守剪 encoder（保 BEV 共享特征 → 保分割）
- 激进剪 decoder（只影响 tracking 内部，微调可补）

### 8.2 工程改造
修改 `_apply_ffn_pruning`，根据 pair_name 区分 encoder/decoder，使用各自的 ratio。

### 8.3 四组实验

| ID | enc | dec | AMOTA | lanes IoU |
|:---:|:---:|:---:|:---:|:---:|
| D.1.1 | 1.0 | 0.3 | 0.329 | 0.211 |
| D.1.2 | 0.8 | 0.3 | 0.337 | 0.214 |
| D.1.3 | 0.7 | 0.4 | 0.306 | 0.209 |
| **D.1.4** | **1.0** | **0.7** | **0.367** | **0.214** |

### 8.4 核心发现

1. **D.1.4 是全局最优**：AMOTA +0.038 vs baseline，ego_forward -42%，seg IoU 完全保留
2. **解耦远优于绑定**：D.1.2 解耦 lanes IoU 0.214 vs S1 绑定 0.096（2.2 倍），AMOTA 还更高
3. **Encoder 不剪是关键**：encoder FFN 影响共享 BEV 特征 → 所有下游任务

## 九、Phase D.2（2026-04-18 ~ 2026-04-19）— 联合量化

### 9.1 工程改造
修改 `quick_eval_quant.py`，支持 `--prune-config` + `--finetuned-ckpt`，实现"先剪枝再量化"的联合管线。

### 9.2 实验结果

| 配置 | AMOTA | 与 FP32 对比 |
|---|:---:|:---:|
| D.2.2 S1 绑定 + INT8 | 0.287 | -0.048 |
| **D.2.7 解耦 + INT8** | **0.360** | **-0.007** |

### 9.3 核心发现

1. **D.2.7 联合后 AMOTA 仅掉 0.007**（0.367 → 0.360），INT8 在解耦配置上近乎无损
2. **解耦在联合量化下优势更大**：D.2.7 (0.360) vs D.2.2 (0.287) = +0.073 AMOTA
3. 解耦配置因为保留了完整 encoder，BEV 特征分布更稳定，量化校准更有效
